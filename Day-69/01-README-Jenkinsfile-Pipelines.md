## Why Jenkins pipelines are code, not just clicks

Early Jenkins (pre-2016) was configured almost entirely through the web UI — jobs, build steps, and triggers all lived as XML config generated by clicking around, which meant zero version control, zero code review, and configuration drift between "what the UI shows" and "what actually runs." **Pipeline as Code**, via a `Jenkinsfile` checked into the repository itself, fixed this: the build definition travels with the code, gets reviewed in PRs, and is reproducible from a clean Jenkins instance. This is the same underlying motivation as `.github/workflows/*.yml` or `.gitlab-ci.yml` — Jenkins just got there first, in its own idiosyncratic Groovy-based syntax.

## Declarative vs. Scripted — two different DSLs, same engine

Both compile down to Groovy running on the Jenkins controller, but they look and behave very differently:

**Declarative** (the modern, recommended default) — a structured, opinionated syntax with a fixed top-level shape (`pipeline { agent {} stages {} }`). Easier to read, easier to lint, and has built-in support for things like `post` blocks and `options` without extra Groovy.

```groovy
pipeline {
    agent any

    environment {
        APP_ENV = 'staging'
    }

    options {
        timeout(time: 30, unit: 'MINUTES')
        disableConcurrentBuilds()
    }

    stages {
        stage('Checkout') {
            steps {
                checkout scm
            }
        }
        stage('Build') {
            steps {
                sh 'docker build -t myapp:${BUILD_NUMBER} .'
            }
        }
        stage('Test') {
            steps {
                sh 'pytest --junitxml=results.xml'
            }
        }
        stage('Deploy') {
            when {
                branch 'main'
            }
            steps {
                sh './deploy.sh staging'
            }
        }
    }

    post {
        always {
            junit 'results.xml'
        }
        failure {
            slackSend(channel: '#ci-alerts', message: "Build failed: ${env.BUILD_URL}")
        }
    }
}
```

**Scripted** (the older, more flexible syntax) — raw Groovy with Jenkins pipeline steps sprinkled in, wrapped in a single `node { }` block. No fixed structure — you write imperative Groovy control flow directly.

```groovy
node {
    stage('Checkout') {
        checkout scm
    }
    stage('Build') {
        sh 'docker build -t myapp:${env.BUILD_NUMBER} .'
    }
    stage('Test') {
        try {
            sh 'pytest --junitxml=results.xml'
        } finally {
            junit 'results.xml'
        }
    }
    if (env.BRANCH_NAME == 'main') {
        stage('Deploy') {
            sh './deploy.sh staging'
        }
    }
}
```

### When you actually need scripted (or `script { }` blocks inside declarative)

Declarative deliberately restricts what you can do directly inside a `steps { }` block — no arbitrary loops, no complex conditional branching beyond `when { }`. When you need real programming logic (a loop building a matrix of parallel jobs dynamically, complex string manipulation, calling out to external Groovy classes), you drop into a `script { }` block inside declarative, which switches you into full scripted-Groovy mode for just that section:

```groovy
stage('Dynamic Matrix') {
    steps {
        script {
            def environments = ['dev', 'staging', 'prod']
            for (env in environments) {
                echo "Would deploy to ${env}"
            }
        }
    }
}
```

This hybrid approach — declarative for structure, `script { }` islands for actual logic — is what most real-world Jenkinsfiles converge on. Pure scripted pipelines are increasingly rare for new work but common in legacy repos you'll inherit.

## Key Jenkinsfile building blocks

- **`agent`** — where the pipeline (or a stage) runs: `agent any` (any available executor), `agent none` (must specify per-stage), `agent { label 'linux' }` (specific node label), or `agent { kubernetes { ... } }` (dynamic pod agent — covered in file 2).
- **`stages` / `stage`** — the visual/logical pipeline steps shown in Jenkins' UI and Blue Ocean.
- **`when`** — conditional stage execution (`branch`, `environment`, `expression`) — this is how you get "only deploy on `main`" without wrapping the whole stage in scripted `if`.
- **`post`** — lifecycle hooks (`always`, `success`, `failure`, `unstable`, `changed`) that run regardless of pipeline outcome — the natural place for notifications and test-result publishing.
- **`environment`** — sets environment variables for the whole pipeline or a single stage; also where you pull in **credentials** safely (`environment { AWS_KEY = credentials('aws-key-id') }`, which Jenkins masks in logs automatically).

## Points to Remember

- Both declarative and scripted pipelines compile to Groovy on the controller — the difference is syntax/structure and how much raw programming logic you're allowed to write directly, not a different execution engine underneath.
- Declarative is the modern default: structured, lintable, has built-in `post`/`when`/`options` support. Use `script { }` blocks inside it for the rare cases needing real loops/conditionals.
- `Jenkinsfile` in the repo root (or wherever configured) is what makes pipelines version-controlled, reviewable, and reproducible — the same motivation as `.github/workflows/*.yml`, just Jenkins-flavored and older.
- `credentials('id')` inside `environment { }` is the correct way to inject secrets — Jenkins automatically masks the resulting value in console logs, unlike a raw `sh "echo $SECRET"` which would leak it in plaintext build logs.
- `agent none` at the top level forces every stage to declare its own `agent` — useful for pipelines that need different executor types (e.g., a build stage on a Docker agent, a deploy stage on a specific labeled node) per stage.

## Common Mistakes

- Writing `sh "some command with ${env.SECRET_VAR}"` (string interpolation) instead of using single-quoted `sh 'some command'` with the variable referenced inside — Groovy's double-quoted string interpolation happens *before* the shell sees the string, meaning secrets end up expanded directly into the Jenkins console log in plaintext, defeating credential masking.
- Mixing scripted-style `if`/`for` directly at the top level of a declarative pipeline without wrapping it in `script { }` — this doesn't parse; declarative's DSL doesn't allow raw Groovy control flow outside a `script` block, and the resulting syntax error can be confusing if you don't know this rule exists.
- Forgetting `post { always { junit '...' } }` — test results are generated but never published to Jenkins' UI, so a failing test can silently pass the pipeline stage (exit code 0 for the test runner) while the actual failure detail is buried in a log nobody reads.
- Using `agent any` for every single stage in a long pipeline, causing each stage to potentially land on a different executor/node with none of the previous stage's workspace files — leading to confusing "file not found" errors between stages that assumed a shared filesystem. Use a single top-level `agent` (or explicitly `stash`/`unstash` files between stages) when stages depend on each other's output.
- Treating the Jenkinsfile as static — not realizing `when { branch 'main' }` and similar conditionals need the multibranch pipeline job type (not a plain pipeline job) configured correctly to even have `BRANCH_NAME` populated, leading to `when` conditions silently never matching.
