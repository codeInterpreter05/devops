# Day 65 — Quiz: Deployment Strategies

Try to answer without looking at your notes. Answers are at the bottom.

1. During a Rolling Update, do old and new pod versions ever run at the same time? Why does this matter for database migrations?
2. What do `maxSurge` and `maxUnavailable` control, and what's a gotcha when applying percentage values to a low-replica-count Deployment?
3. Why would a team deliberately choose Recreate over Rolling Update, given that it introduces downtime?
4. What must be true for a Rolling Update's safety guarantee (readiness-probe-gated pod replacement) to actually hold?
5. In Argo Rollouts' blue/green strategy, what's the difference between `activeService` and `previewService`, and what does `scaleDownDelaySeconds` protect?
6. What's the resource-cost tradeoff of blue/green versus canary during the transition window?
7. What component does Argo Rollouts' canary strategy delegate actual traffic-percentage splitting to, and why can't vanilla Kubernetes do this on its own?
8. What does an `AnalysisTemplate` add to a canary rollout that a plain `pause`-only canary doesn't have?
9. What's the difference between "deploying" code and "releasing" a feature, and which tool category (name two examples) is built specifically to decouple them?
10. What are the five steps of the expand/contract migration pattern, and why is the final "contract" step deliberately separated in time from the rest?
11. Why is limiting rollback triggers to "process liveness" (is the pod Running/Ready) insufficient, and what should they check instead?
12. **Interview question:** Walk me through a zero-downtime deployment strategy for a stateful service with a database migration.

---

## Answers

1. Yes — a Deployment controller creates a new ReplicaSet and scales it up while scaling the old one down, so both versions run simultaneously for the duration of the rollout. This matters for database migrations because if the new version depends on a schema change the old version's code can't handle (or vice versa), the coexistence window means old code will actually run against the new schema (or new code against old schema) and can error out or corrupt data — Rolling Update's safety guarantee is about pod availability, not application/schema compatibility between versions.
2. `maxSurge` caps how many extra pods beyond the desired `replicas` count can exist during rollout; `maxUnavailable` caps how many pods below the desired count can be unavailable. Both are usually percentages, and percentages get rounded to whole pod counts — on a low-replica Deployment (e.g., 2 replicas), a 25% value can round down to 0, meaning effectively no surge/unavailability headroom is actually granted, which is easy to miss if you only read the percentage and don't compute the actual pod count.
3. When old and new versions genuinely cannot coexist safely — e.g., a breaking/non-backward-compatible database migration, or a singleton workload that can't have two instances holding the same exclusive resource/lock at once — Recreate guarantees zero overlap by fully stopping old pods before starting new ones, at the cost of a downtime window. It may also be chosen in resource-constrained environments that can't afford Rolling Update's surge headroom.
4. The `readinessProbe` must accurately reflect whether the application is actually ready to correctly serve traffic (e.g., DB connections established, caches warmed) — not just whether the process/port is up. If the probe reports ready too early, Kubernetes will route real traffic to a not-actually-ready pod, or remove old (working) pods before new ones are genuinely serving correctly, breaking the entire safety model the strategy depends on.
5. `activeService` is what real users/traffic hit — pointing at the currently "live" version. `previewService` gives access to the new version for testing/validation before it receives any real traffic. `scaleDownDelaySeconds` keeps the old version's pods running (not torn down) for a window after cutover, so that "instant rollback" (switching `activeService` back) is actually possible without needing to redeploy the old version from scratch.
6. Blue/green runs both versions at **full capacity simultaneously** during the transition — roughly double the resource cost. Canary only runs a smaller subset of new-version replicas proportional to the current traffic weight (e.g., 10% weight needs far fewer new-version pods than 100%), so its resource overhead during the transition is much smaller than blue/green's.
7. It delegates to a traffic-splitting layer — a service mesh (Istio, Linkerd), an ingress controller with weighted routing support (NGINX, ALB), or a Gateway API implementation. Vanilla Kubernetes has no native concept of "route X% of traffic to ReplicaSet A and the rest to ReplicaSet B" — Services route to all matching pods roughly evenly by pod count, not by an explicit configurable percentage, so a real traffic-splitting layer is required to make `setWeight` percentages meaningful and precise.
8. `AnalysisTemplate` adds an **automated, metrics-based gate** — it queries a real metrics source (e.g., Prometheus) for a defined success condition (like error rate) during the pause, and automatically aborts/rolls back the rollout if the condition fails, without requiring a human to be watching. A plain `pause`-only canary just waits for a fixed time or manual resume — it has no automated judgment about whether the new version is actually behaving well in production.
9. Deploying means the code exists and is running in production (possibly dormant/inactive); releasing means real users can actually see/use the feature. Feature flag tools decouple these — two examples: LaunchDarkly (commercial/hosted) and Unleash (open-source/self-hosted).
10. (1) Expand — add new schema elements (nullable column/new table) that are backward compatible with old code; (2) deploy code that dual-writes to both old and new schema; (3) backfill existing data into the new schema in the background; (4) deploy final code that only uses the new schema, once backfill is verified complete; (5) contract — drop the old column/table. The contract step is separated and deliberately last because it's irreversible, and doing it only after every prior step is fully verified avoids permanently destroying data based on an assumption that turns out to be wrong.
11. A pod can report `Running`/`Ready` at the Kubernetes level while still serving errors (5xx responses) or badly degraded latency to real requests, if the readiness probe doesn't actually reflect true application health. Rollback triggers should instead check real symptom metrics users would notice — error rate, latency percentiles, saturation — via something like an `AnalysisTemplate` querying Prometheus, not just whether the process/port is up.
12. Strong answer: "I'd combine an infrastructure-level strategy with an application-level migration pattern. For the deploy itself, I'd use a canary rollout via Argo Rollouts with automated analysis gates on real error-rate/latency metrics, so a bad release only ever reaches a small percentage of traffic and gets automatically rolled back if metrics degrade. For the database, I wouldn't ship the schema change and the dependent code in the same step — I'd use an expand/contract migration: first an additive, backward-compatible migration, then a dual-write/dual-read capable code version that works whether or not the migration has finished propagating, then a backfill, then a final code version that only uses the new schema, and only at the very end, once everything is verified, a separate contract migration dropping the old schema elements. This is necessary because any zero-downtime strategy — rolling, canary, even blue/green's cutover window — has old and new code running simultaneously for some period, so the database has to tolerate both versions at once until that overlap is fully resolved."
