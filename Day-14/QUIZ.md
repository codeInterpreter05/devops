7. What is connection draining, and why does skipping it cause user-facing errors during rolling deploys?

9. What does the `ip_hash` directive in nginx actually do, and what real-world condition breaks its assumption?

## Answers

1. L4 load balancers make decisions using only the transport-layer connection tuple — source/destination IP and port — without looking at the payload at all. L7 load balancers terminate the connection, parse the actual application-layer request (HTTP method, path, headers, cookies, host), and route based on that content.
2. Because L7 has to fully terminate the client's connection and open a *separate* connection to the chosen backend (proxying two connections instead of forwarding one), parse the HTTP request to make the routing decision, and — if TLS is involved — perform a full handshake and decrypt/re-encrypt cycle. L4 can often just forward packets for an already-established connection tuple without any of that overhead.
3. **Least connections** — better than round robin when request processing time varies a lot, since round robin can pile slow requests onto one backend while others sit idle; least connections actively balances by current load. **IP hash / consistent hashing** — needed when you require sticky sessions (the same client must keep hitting the same backend for in-memory session state to work), which plain round robin cannot guarantee.
4. Plain hashing (e.g., `hash(key) % N`) remaps almost every key when N (the backend count) changes — adding or removing one backend reshuffles most of the mapping. Consistent hashing (e.g., ketama) only remaps the keys that specifically belonged to the added/removed backend, leaving everyone else's mapping untouched — critical for minimizing cache invalidation storms or session disruption when scaling.
5. Active health checks are the load balancer proactively probing each backend on a timer (TCP connect or HTTP request to a health endpoint), independent of real traffic — this catches failures before users hit them and works even at low traffic volume. Passive health checks infer backend health from real request failures (timeouts, connection errors, 5xx) within a rolling window — cheaper, no dedicated endpoint needed, but at least one real user request has to fail first, and detection is slow when traffic is light. Using both gives faster, more complete detection: active covers what its health endpoint checks, passive is a backstop for anything the active check's endpoint doesn't verify (e.g., a downstream dependency failure that a naive `/health` returning `200` unconditionally wouldn't catch).
6. Open-source nginx has no background timer that independently probes backends regardless of traffic — that proactive probing loop is gated behind nginx Plus's `health_check` directive (or a third-party module like `nginx_upstream_check_module`). What OSS nginx does have is `max_fails`/`fail_timeout` on each `server` line inside `upstream`, which is reactive: it only marks a backend down after real proxied requests to it fail a set number of times within a window.
7. Connection draining is the practice of taking a backend out of rotation for *new* traffic while letting requests already in flight to it finish naturally (up to a timeout) before it's actually removed/shut down. Skipping it and just killing/removing a backend outright drops every in-flight request to it instantly — the client sees a connection reset or a 502 — which is exactly what happens on every deploy if draining isn't wired in.
8. A forward proxy sits in front of clients, on their behalf (configured into the client, the destination server is often unaware a proxy is involved) — used to control/anonymize outbound traffic. A reverse proxy sits in front of servers, on their behalf (the client connects to it believing it's the real server, unaware of what's behind it). nginx doing load balancing is acting as a reverse proxy — clients talk to nginx, nginx decides which backend actually serves the request.
9. `ip_hash` hashes the client's IP address to deterministically and consistently pick the same backend for every request from that IP, which is how it provides sticky sessions without a shared session store. It breaks down behind NAT/CGNAT, where many distinct real users share one public IP as seen by nginx — they all get pinned onto the same backend, which both defeats the point of balancing that traffic and can overload that one backend. It also breaks for a single client whose IP changes mid-session (e.g., switching from WiFi to cellular).
10. Termination means the load balancer holds the certificate/key, decrypts the client's TLS connection, and forwards plaintext to the backend — cheaper for backends, centralizes cert management, enables L7 content-based routing. Passthrough means the load balancer forwards the encrypted bytes untouched and the backend itself terminates TLS; the LB can still route based on the SNI hostname visible during the handshake, but can't inspect anything else. A concrete scenario requiring passthrough: mutual TLS (mTLS), where the backend must itself validate the client's certificate — if the LB terminates the connection first, the backend never sees that certificate and the validation can't happen.
11. L4 (transport layer) load balancing routes purely on the connection tuple — source/destination IP and port — without looking at the payload; it's fast and protocol-agnostic but can't make decisions based on content (AWS NLB, plain TCP/UDP load balancers). L7 (application layer) load balancing terminates the client's connection, parses the actual HTTP request (path, host, headers, cookies), and opens a new connection to the chosen backend — more CPU-intensive because of the extra connection, parsing, and (if TLS) handshake/decrypt overhead, but it enables routing decisions L4 can't make at all. You need L7 whenever the routing decision depends on something *inside* the request rather than just where it's addressed to: path-based routing (`/api` vs `/static`), host-based routing (multiple domains behind one IP), cookie/header-based routing (canary releases, A/B tests), or when you want to centralize SSL termination and certificate management at the load balancer instead of on every backend.
