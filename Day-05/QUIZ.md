# Day 5 — Quiz: Bash Scripting I

Try to answer without looking at your notes. Answers are at the bottom.

1. What is the shebang line, and how does the Linux kernel actually use it when you run `./script.sh`?
2. Why is `#!/usr/bin/env bash` generally preferred over `#!/bin/bash`? Why is `#!/bin/sh` not a safe substitute for either on Debian/Ubuntu?
3. What's the difference between how bash expands a variable inside single quotes vs. double quotes vs. unquoted?
4. Why does `for f in $(ls *.txt)` behave differently — and worse — than `for f in *.txt`?
5. What's the practical difference between `[ ]`, `[[ ]]`, and `(( ))`? When would you reach for each?
6. Why does `if` in bash not need a boolean expression the way `if` does in Python or JavaScript?
7. What does `set -o pipefail` change about how a pipeline's exit status is determined, and why does that matter for a script that pipes through `grep`/`sort`?
8. Name two specific situations where `set -e` will *not* cause a script to exit on a failing command, even though `set -e` is active.
9. What's wrong with `for line in $(cat file.txt)` as a way to process a file line by line, and what should you write instead?
10. What do exit codes `127`, `126`, and `130` each specifically indicate?
11. What does `trap cleanup EXIT` guarantee that a cleanup step placed at the bottom of a script does not?
12. **Interview question:** What does `set -euo pipefail` do and why should every script start with it?

---

## Answers

1. The shebang (`#!`) is a kernel convention read via the `execve()` syscall: when a file is executed directly and its first two bytes are `#!`, the kernel treats the rest of that line as the interpreter to invoke, then re-executes the file as `<interpreter> <script-path> <args>`. It must be the very first line/bytes of the file, and it's only honored when the script is run directly (`./script.sh`) with its execute bit set — it's ignored if you invoke it as `bash script.sh`.
2. `#!/usr/bin/env bash` resolves `bash` from `$PATH` at runtime, so it works regardless of where bash happens to be installed (`/bin/bash` on most Linux, but e.g. `/opt/homebrew/bin/bash` on macOS with Homebrew, or a different path on NixOS/minimal containers). `#!/bin/bash` hardcodes one specific location and breaks if bash isn't there. `#!/bin/sh` is not interchangeable with either — on Debian/Ubuntu, `/bin/sh` is symlinked to `dash`, a strict POSIX shell without bash features like arrays, `[[ ]]`, or `local`; a script using those features but shebanged `#!/bin/sh` will fail (or misbehave) if invoked as `sh script.sh` or on a system where `sh` truly is dash.
3. Single quotes (`'...'`) suppress *all* expansion — `$VAR` stays literally `$VAR`. Double quotes (`"..."`) still allow variable expansion, command substitution (`$(...)`), and arithmetic expansion (`$((...))`), but the result is treated as one word — no word-splitting or glob expansion happens afterward. Unquoted expansion does both variable/command substitution *and* subjects the result to word-splitting (on `$IFS`) and pathname/glob expansion — which is almost never desired for a variable holding a path or arbitrary value.
4. `$(ls *.txt)` runs `ls`, and its unquoted output is word-split on whitespace before the `for` loop sees it — a filename containing a space gets split into multiple bogus "files," and it's a redundant subprocess since bash can already glob-expand `*.txt` directly. `for f in *.txt` lets bash expand the glob itself into a proper list of whole filenames, with no word-splitting risk and no extra process.
5. `[ ]` is the `test` command (POSIX, portable, but requires careful quoting/spacing since it's an ordinary command subject to word-splitting). `[[ ]]` is a bash keyword — safer by default (no word-splitting/globbing on unquoted variables inside it), and adds glob matching (`==`) and regex matching (`=~`). `(( ))` is for arithmetic expressions/comparisons (`(( a > b ))`) and treats a non-zero result as true. Use `[[ ]]` for string/file tests in bash scripts, `(( ))` for numeric comparisons, and `[ ]` only when you need strict POSIX `sh` portability.
6. Bash's `if` evaluates the **exit status** of a command, not a boolean expression — `0` (success) is treated as true, any non-zero value as false. This is why `if grep -q pattern file; then` works directly: `grep`'s own exit code (found/not-found/error) is exactly the condition being tested, no explicit comparison needed. `[ ]`/`[[ ]]`/`(( ))` are just commands/keywords that happen to return 0 or 1, so `if` treats them like any other command.
7. Without `pipefail`, a pipeline's exit status is only the exit status of its *last* command — if an earlier stage (e.g. `grep`) fails but a later stage (e.g. `sort`, `uniq`) still exits 0 on empty input, the whole pipeline reports success, hiding the real failure. `set -o pipefail` changes the pipeline's exit status to the last *non-zero* status among all stages (or 0 if everything succeeded), so a failure anywhere in a `grep | sort | uniq -c` chain is no longer silently swallowed.
8. Two examples: (1) a command used as the condition of `if`/`while`/`until` — its failure is the expected, tested-for outcome, so `set -e` is suspended there; (2) a non-final command in a pipeline without `pipefail` active — only the last stage's exit status is checked, so an earlier failing stage doesn't trigger `set -e`. (A third valid answer: a command whose result feeds `&&`/`||`, since its "failure" is being explicitly handled by that operator.)
9. `$(cat file.txt)` is unquoted, so its output gets word-split on *every* whitespace character (spaces, tabs, and newlines) — this destroys line boundaries entirely (a line with two words becomes two loop iterations) rather than iterating one whole line at a time. The correct idiom is `while IFS= read -r line; do ...; done < file.txt`, which reads exactly one line per iteration, preserves leading/trailing whitespace (`IFS=`), and doesn't interpret backslashes in the content (`-r`).
10. `127` = command not found (typo or not on `$PATH`). `126` = command was found but could not be executed (e.g., missing execute permission, or trying to "run" a file that isn't actually a valid executable/script). `130` = process terminated by `SIGINT` (Ctrl-C), computed as `128 + 2` where `2` is `SIGINT`'s signal number.
11. A cleanup step at the bottom of a script only runs if execution reaches that line — it's skipped entirely if the script exits early via `set -e`, an explicit `exit` in an error branch, or a signal (Ctrl-C, `kill`). `trap cleanup EXIT` registers the cleanup function against the `EXIT` pseudo-signal, which bash fires on *any* script termination path, guaranteeing it runs regardless of how or where the script ends.
12. **Full answer:** `set -e` (errexit) makes the script exit immediately the first time any command returns a non-zero exit status, instead of bash's default behavior of continuing on to the next line with a possibly broken assumption. `set -u` (nounset) turns any reference to an unset variable into an immediate error rather than silently expanding it to an empty string — this specifically guards against typo'd variable names that would otherwise combine with something like `rm -rf "$VAR"/*` into a silent, catastrophic no-op-turned-wildcard-delete. `set -o pipefail` fixes the fact that, by default, a pipeline's exit status is only its *last* command's status — with `pipefail`, the pipeline reports failure if *any* stage failed, not just the final one. Together, `set -euo pipefail` converts three classes of silent, "keep going anyway" failure modes into loud, immediate ones — which is exactly the behavior you want in an unattended script (cron, CI, a deploy hook) where nobody is watching the output live to notice something quietly went wrong. It's not a complete substitute for explicit error handling (it has known gotchas around conditionals and certain command-substitution contexts), but it's the right conservative default for virtually every script that isn't a one-off interactive experiment.
