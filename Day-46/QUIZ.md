# Day 46 — Quiz: K8s Operators & CRDs

Try to answer without looking at your notes. Answers are at the bottom.

1. What does a CRD actually give you, and what does applying a CRD-defined custom resource *not* do by itself?
2. What is the Operator pattern, in one sentence?
3. Why is a controller's reconciliation loop described as "level-triggered" rather than "edge-triggered," and why does that matter for resilience to missed events?
4. Why must a reconcile function be idempotent, and what practice does that idempotency enable (hint: think about resync timers)?
5. What's the difference between `spec` and `status` on a custom resource, and how is that split enforced?
6. What problem do owner references solve, and what problem do finalizers solve? How are they different?
7. Name the three Operator SDK authoring approaches and the key tradeoff between them.
8. How does CloudNativePG handle a primary Postgres pod failing, and what's different about how the read-write Service reacts?
9. Why does Strimzi split Kafka into multiple CRDs (`Kafka`, `KafkaTopic`, `KafkaUser`) instead of one big object?
10. Why does Strimzi's rolling restart process wait for in-sync replicas before restarting the next broker?
11. What's a legitimate reason to write a custom Operator, versus a reason that's usually a mistake?
12. **Interview question:** What problem do Kubernetes Operators solve? When should you write your own?

---

## Answers

1. A CRD extends the Kubernetes API's vocabulary — it defines a new resource type with a schema, storage, versioning, and RBAC support, just like a built-in resource. Applying a custom resource of that type only stores structured data in etcd and validates it against the schema — it does not, by itself, cause any actual infrastructure or software to be provisioned or run. That requires a controller (the Operator) watching and acting on it.
2. An Operator is a CRD (or set of CRDs) plus a controller that continuously watches them and takes whatever action is needed to make real-world state match the declared desired state, encoding domain-specific operational knowledge (not just deployment, but day-2 operations like backup, failover, and upgrades) as running software.
3. Level-triggered means every reconcile call re-derives the *full* current desired and actual state and acts on the diff, regardless of what specifically triggered it — rather than reacting to a single specific event and assuming it captured everything that changed. This matters because if some watch events are missed (e.g., during a controller restart), a level-triggered reconcile still converges correctly on the next run since it's comparing full state, not replaying a potentially incomplete event history.
4. Idempotency means calling reconcile repeatedly with the same state produces the same result with no harmful side effects. This is essential because reconciliation is called far more often than "once per real change" — including on periodic resync timers used as a safety net against missed or dropped watch events — and none of those extra calls should cause incorrect behavior.
5. `spec` represents the desired state, written by the user (or another controller acting on their behalf). `status` represents observed/actual state, written only by the controller. This is enforced via the Kubernetes `/status` subresource, which can have separate RBAC rules from the main resource, physically preventing regular users from writing to status and controllers from needing spec-write permission.
6. Owner references let Kubernetes' built-in garbage collector automatically cascade-delete resources a controller created (e.g., deleting a custom resource also deletes the StatefulSet/Pods it owns) without custom cleanup code. Finalizers guarantee a controller gets a chance to run cleanup logic for *external*, non-Kubernetes resources (e.g., deprovisioning a cloud resource) before Kubernetes actually finalizes deletion of the object — they block deletion until removed, unlike owner references which trigger automatic cascading deletion.
7. Go-based (full flexibility, write your own reconcile logic in Go using controller-runtime), Ansible-based (reconcile logic is an Ansible playbook/role), and Helm-based (reconcile logic is "apply this Helm chart with values from the CR's spec"). The tradeoff is flexibility vs. ease of authoring: Helm-based is simplest but can only express what Helm templating allows (no genuinely custom imperative day-2 logic); Go-based is the most flexible but requires the most engineering investment.
8. CNPG continuously monitors primary health; on failure, it promotes the most up-to-date standby to primary and reconfigures the remaining standbys to replicate from it — fully automatically, no human intervention. The read-write Service's selector is automatically updated to point at the new primary, so applications connecting to the stable `-rw` Service name transparently follow the new primary without any connection-string or DNS change.
9. Splitting into multiple CRDs allows RBAC to be scoped differently per concern — e.g., an application team can be granted permission to create `KafkaTopic` and `KafkaUser` objects in their own namespace without ever having access to the underlying `Kafka` broker cluster resource, mirroring a real separation between platform-team and application-team responsibilities.
10. Because restarting a broker before its partition replicas have caught up (fully in the in-sync replica set) risks under-replicated partitions or even data loss if another broker also has issues during the window — waiting for full replication health before proceeding to the next broker preserves Kafka's replication guarantees throughout the rolling restart.
11. Legitimate: genuinely custom, ongoing operational logic with no existing tool covering it — e.g., automating a company's own bespoke internal platform lifecycle. Usually a mistake: building a custom Operator for well-known software (a database, a message queue) that already has a mature, widely-used open-source Operator, reinventing operational logic that took the existing project years to get right.
12. Strong answer: "Operators encode operational knowledge — not just how to deploy something, but how to run it reliably over time: backups, failover, upgrades, scaling — as software that continuously reconciles a declared desired state against reality, instead of that knowledge living in a person's head or a runbook. You should write your own only when you have genuinely custom, ongoing operational logic that no existing tool covers — for example, automating your own internal platform's tenant lifecycle. For well-known software like Postgres or Kafka, use a mature existing Operator (CloudNativePG, Strimzi) rather than reinventing failover and backup logic that took those projects years to get right, and don't reach for the Operator pattern at all if there's no real ongoing day-2 operational complexity — a plain Helm chart is often sufficient."
