# Day 76 — Quiz: CI/CD & DevSecOps Review

Try to answer without looking at your notes. Answers are at the bottom.

1. Why is "reread all my notes" a weak review strategy compared to a closed-book topic dump?
2. What does the "explain out loud" method surface that silent review doesn't?
3. What's the difference between being able to "describe" a tool (like ArgoCD or Prowler) and being able to "build with" it — why does the distinction matter?
4. Why is pinning a third-party GitHub Action to a full commit SHA safer than pinning to a version tag like `@v4`?
5. Describe the specific vulnerability pattern involving `pull_request_target`. What makes it dangerous, and what's the actual triggering condition (not just using the event alone)?
6. A pipeline has a Trivy scan step configured with `exit-code: '0'`. What's wrong with this, even if the step "runs successfully" on every build?
7. Name two of the most common, highest-value, lowest-effort CI pipeline optimizations you'd look for first when auditing an unfamiliar pipeline.
8. Why should a security-improvement PR to an open-source project be small and narrowly scoped rather than a sweeping rewrite?
9. What professional trade-off should you explicitly call out when recommending SHA-pinning of Actions in a PR description?
10. Why is updating your own documentation part of "the review," not a separate afterthought task?
11. **Interview question (practice format):** Review and critique this pipeline... (practice explaining your audit process using a real example pipeline you've looked at, covering supply chain, secrets, gating, and efficiency in under 3 minutes.)

---

## Answers

1. Rereading produces recognition (the material feels familiar when you see it again), which is a weaker and different cognitive skill than recall (producing the answer unprompted from memory). Interviews test recall — you have to generate the answer yourself with no notes in front of you — so a review method that only builds recognition gives a false sense of readiness.
2. It surfaces hedging and uncertainty ("I think it's something like...") that silent, in-your-head review never reveals — when you only think through an answer without saying it aloud, vague or incomplete understanding can feel more solid than it actually is. Speaking it out forces you to commit to specific words and exposes exactly where you're unsure.
3. "Describing" means you can explain what a tool does and why, from having read about it or used it briefly in a guided lab. "Building with" means you could set it up, configure it, and troubleshoot it from scratch, unguided, right now. The second is a much stronger, more interview- and job-resilient signal — many people can describe ArgoCD's sync policies but couldn't actually stand up an Application manifest and debug a sync failure without documentation open.
4. A version tag like `@v4` is a mutable pointer — the action's maintainer (or an attacker who compromises their account/repo) can repoint it to different, potentially malicious code at any time, and your workflow file wouldn't change at all, so you'd have no visible signal that the code you're running changed. A commit SHA is an immutable, content-addressed reference — it always points to the exact same code, regardless of what tags get moved later.
5. `pull_request_target` runs with the **base repository's** secrets and permissions (unlike plain `pull_request`, which runs with the fork's limited permissions). The dangerous pattern specifically is when a workflow using `pull_request_target` also checks out and executes the **fork's** (i.e., the PR submitter's, potentially untrusted) code — e.g., checking out `github.event.pull_request.head.sha` and running its scripts/tests. That combination lets an attacker who opens a PR run arbitrary code with access to the base repo's secrets. Using `pull_request_target` alone, without checking out/executing fork code, is not itself the vulnerability.
6. `exit-code: '0'` (or `continue-on-error: true` left unaddressed) means the scan step never actually fails the build regardless of what it finds — it produces a report, but nothing structurally blocks a vulnerable image from proceeding to build/sign/push/deploy. This is "compliance theater": it looks like a security gate exists, but it has no actual enforcement power, and a team can end up trusting a scan that's never once blocked anything.
7. Any two of: adding dependency/build caching (`actions/cache`, Docker layer caching via `cache-from`/`cache-to`) where it's missing; removing unnecessary `needs:` serialization between jobs that don't actually depend on each other so they can run in parallel; scoping `on: push`/full test suite runs away from every branch/every push where no PR exists; trimming matrix builds down to only currently-supported version combinations.
8. Maintainers are far more likely to review, understand, and merge a small, well-justified, low-risk change quickly. A sweeping rewrite is harder to review, riskier to merge (more surface area for something to break), and asks the maintainer to trust a large diff from someone they don't know — small, individually justified PRs respect the maintainer's time and review capacity.
9. That pinning to SHA means the pin no longer auto-updates with new releases of the action — someone (a person, or ideally a bot like Dependabot/Renovate configured for this) now needs to manually bump the pinned SHA when a new version is needed, which is an ongoing maintenance cost the maintainer should be aware they're taking on in exchange for the security benefit.
10. Because a gap identified but not written down (with a concrete example, cross-links to related days, or an explicit "still shaky" flag) is easily forgotten again — the entire value of a review is that future-you (in an interview prep session weeks later, or mid-incident) has a fast, trustworthy reference. Skipping the documentation-update step means you redo the same discovery work repeatedly instead of compounding it.
11. Strong answer structure: "I start with supply chain — are third-party Actions pinned to a SHA or a movable tag, and does anything using `pull_request_target` also check out and run fork code? Then secrets — are any values echoed into logs, and is the pipeline using long-lived static cloud credentials where OIDC federation could remove the stored secret entirely? Then gating — do build/deploy jobs actually `needs:` the test/SAST/scan jobs, and does the scan step's exit code actually fail the build on real findings, or is it just informational? Finally efficiency — is there dependency/Docker layer caching, and are independent jobs serialized unnecessarily via `needs:` edges that don't reflect a real dependency? For this specific pipeline, [cite 2-3 concrete findings you actually saw], and here's the smallest fix I'd propose for each, plus any trade-off it introduces." Practicing this exact structure out loud against 2-3 real pipelines is the best rehearsal for this question.
