# Day 74 — Quiz: Compliance & Audit

Try to answer without looking at your notes. Answers are at the bottom.

1. What makes a CIS Benchmark recommendation "Scored" vs. "Not Scored," and why does that distinction matter for automation?
2. Why do most `kube-bench` master-node (section 1.x) checks fail meaninglessly on an EKS/GKE/AKS cluster, and what should you use instead?
3. What's the difference in what `kube-bench` checks vs. what `Checkov` checks, in terms of *when* in the lifecycle each one runs?
4. Name the five SOC2 Trust Service Criteria. Which one is mandatory?
5. What's the practical difference between a SOC2 Type I and Type II audit, and why does that difference make compliance-as-code more valuable for Type II?
6. Give two concrete infrastructure/engineering implications of GDPR's "right to erasure."
7. Why does the GDPR 72-hour breach notification requirement make detection speed a compliance issue, not just an operational one?
8. What does `--enable-log-file-validation` do for a CloudTrail trail, and why does it matter for audit purposes?
9. What is Prowler's key advantage over manually checking compliance items one by one, in terms of report reusability across audits?
10. What's the difference between what AWS Security Hub provides versus what Prowler provides, and why might a team run both?
11. **Interview question:** How would you ensure your EKS cluster is CIS benchmark compliant? What tools automate this?

---

## Answers

1. "Scored" recommendations have a fully automatable audit procedure — a script/tool can definitively determine pass/fail (e.g., checking an API server flag's value). "Not Scored" recommendations require manual/organizational judgment that can't be fully automated (e.g., "ensure your incident response plan covers X"). This matters because tools like kube-bench can only meaningfully automate the Scored items — Not Scored items still need a human process/attestation.
2. On managed Kubernetes, the cloud provider owns and operates the control plane (API server, etcd, scheduler) — you have no access to their flags or configuration, so the checks structurally cannot pass or even be meaningfully evaluated; they fail because the thing being checked is inaccessible, not because it's misconfigured. Use the cloud-specific CIS Benchmark variant (e.g., CIS Amazon EKS Benchmark, or `kube-bench run --benchmark eks-1.2.0`), which scopes checks to only the node/policy sections you actually control.
3. `kube-bench` audits a **running** cluster's live node/control-plane configuration at scan time (runtime/post-deployment compliance). `Checkov` scans **infrastructure-as-code source** (Terraform, CloudFormation, Kubernetes YAML) before it's ever applied — a shift-left check that can block a PR from introducing a non-compliant configuration in the first place.
4. Security (mandatory), Availability, Processing Integrity, Confidentiality, Privacy. Security is the only one required in every SOC2 audit; the other four are included based on which services/claims the organization is being audited against.
5. Type I audits whether controls are designed correctly at a single point in time ("do these exist and make sense?"). Type II audits whether controls actually operated effectively over an extended period (typically 6–12 months), sampling real evidence across that window. Compliance-as-code matters more for Type II because it produces continuous, timestamped, re-runnable evidence automatically — manually reconstructing a year's worth of access-review/scan evidence after the fact is painful and error-prone.
6. (a) You need a way to actually and completely delete a user's personal data across every system that holds it — primary DB, caches, data warehouse, log files — not just flag it as deleted, which typically means designing a single PII source of truth with cascading deletes from the start. (b) You need visibility into everywhere PII has been copied (e.g., prod-to-staging restores, analytics exports) so an erasure request can actually be fulfilled everywhere the data landed, not just in the primary datastore.
7. The 72-hour clock starts at the moment the organization becomes *aware* of a breach, not at the moment the breach occurred. If your logging/monitoring can't detect a breach until weeks after it happened, you cannot possibly notify within 72 hours of that (much later) awareness — so detection speed (centralized logging + real-time alerting) directly determines whether the notification deadline is even achievable.
8. It generates hourly cryptographic digest files that let you later cryptographically verify a CloudTrail log file hasn't been altered since it was written, making the logs tamper-evident. This matters because an auditor (or incident responder) needs to trust that a log wasn't modified by an attacker (or anyone else) after the fact — without validation, a log's integrity is just an assumption.
9. Prowler maps each individual check to multiple compliance frameworks simultaneously (CIS, SOC2, GDPR, PCI-DSS, ISO 27001, NIST, etc.), so a single scan run produces evidence reusable across several different audits at once, instead of running separate manual reviews (or separate tools) per framework.
10. AWS Security Hub provides continuous, native, AWS-integrated compliance monitoring (aggregating its own checks plus GuardDuty/Inspector/Macie findings) with near-real-time updates and potential auto-remediation via EventBridge, but it's AWS-only and has per-check pricing. Prowler is free, open-source, broader in scope (increasingly multi-cloud/Kubernetes), and CI-friendly for point-in-time or scheduled gate checks, but isn't natively continuous unless you schedule it yourself. Many teams run both — Security Hub for always-on native monitoring, Prowler for CI gates and multi-framework reporting.
11. Strong answer: "First, recognize that EKS has a managed control plane, so the standard CIS Kubernetes Benchmark's master-node section (1.x) doesn't apply — AWS owns that layer. I'd run kube-bench with the EKS-specific benchmark profile (`kube-bench run --benchmark eks-1.2.0`) to check only what's actually within my control: kubelet configuration, RBAC, and pod security policies/standards on the node and cluster-policy layers. I'd also shift left by running Checkov against the Terraform/Kubernetes manifests that provision the cluster and workloads, so CIS-relevant misconfigurations (like an overly permissive security group feeding node access, or a pod spec running as root) are caught at PR time before they're ever applied. For continuous drift detection after deployment, I'd enable AWS Security Hub with the CIS AWS Foundations standard for account-level checks, and optionally run Prowler on a schedule for broader multi-framework reporting. The key nuance I'd flag proactively: don't report inapplicable master-node findings as real gaps — that's a common mistake that undermines credibility with whoever's reading the compliance report."
