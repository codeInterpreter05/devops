## Answers

1. Which identity/access-control system the org is already standardized on — AWS IAM (favors ECR), GitHub repo permissions (favors GHCR) — since matching your existing auth boundary avoids standing up and maintaining a second, parallel identity system, more than any specific feature difference between the registries.
2. Anonymous/unauthenticated pull-rate limiting — many CI runners sharing an egress IP can exhaust Docker Hub's free anonymous pull quota, causing mysterious "pull rate limit exceeded" build failures. Standard fix: authenticate pulls (raises the limit) or proxy/mirror frequently-used base images through an internal registry (Nexus/Artifactory/ECR) so the whole org shares one cached pull.
3. An OCI artifact is any content (a Helm chart, an SBOM, a signature, a generic file bundle) pushed to a registry using the OCI distribution spec's manifest+blob API — the same push/pull mechanism used for container images. This works because the underlying registry API is a generic, content-addressable store; nothing about it is inherently specific to container image layers.
4. You reference the chart directly via an `oci://` URL (`helm push .../mychart.tgz oci://registry/helm-charts`, `helm install ... oci://registry/helm-charts/mychart --version X`). This eliminates the separate `helm repo add` step and the need for a static `index.yaml` chart-repository file — the registry itself is the source of truth for available versions.
5. Hosted: stores your own internally-published packages. Proxy: a caching mirror of a public registry (PyPI, npm, Docker Hub, Maven Central) — first request fetches and caches, subsequent requests are served locally. Group: a single virtual endpoint merging hosted + proxy repos so client tools only need one URL configured.
6. MAJOR promises the release contains breaking/incompatible changes (consumers must expect to update their code). MINOR promises new functionality added in a backward-compatible way (safe to adopt without changes). PATCH promises backward-compatible bug fixes only (always safe to adopt). If a publisher breaks this contract (e.g., ships a breaking change as a "patch"), every consumer pinned to a permissive version range (like `^2.4.0`) can be silently broken by an auto-update they trusted to be safe.
7. It inspects every commit message since the last release tag, looking for Conventional Commits prefixes (`feat:`, `fix:`, `feat!:`/`BREAKING CHANGE:`), determines the highest-severity bump type present across all of them, and computes the new version number automatically — removing manual, inconsistent human judgment from the release process.
8. A signature answers "who produced this artifact, and is it unmodified since." An SBOM answers "what's inside this artifact" (a full package inventory). Provenance answers "how was this artifact built" (which source commit, which build system/steps, what inputs) — a structured, verifiable build history rather than just an identity check or a contents list.
9. Level 2 requires provenance to be generated and signed by a hosted build platform (trusting the platform to self-report honestly, but the untrusted build script itself is not the one generating the record). Level 3 additionally requires strong tamper resistance — isolated per-build environments (no shared mutable state one build could use to influence another) and non-forgeable provenance, meaning even a compromised build step or someone with partial access to the build system can't fake or alter the provenance record.
10. SolarWinds' attackers compromised the build process itself, injecting malicious code during compilation — the resulting artifact would still pass a vulnerability scan (no known CVEs in the injected code) and could still be validly signed (the compromised build process is the one doing the signing, using legitimate credentials). SLSA Level 3's isolation and non-forgeable-provenance requirements specifically target this: they make it structurally impossible for a compromised or malicious build step to influence/lie about the trusted provenance record, since the untrusted build logic is isolated from the platform's own trusted provenance-generation mechanism.
11. `permissions: id-token: write` and `attestations: write`, combined with the `actions/attest-build-provenance` action — it generates a signed SLSA-compatible provenance attestation tied to the specific workflow run, commit, and build inputs automatically.
12. Strong answer: "SLSA (Supply-chain Levels for Software Artifacts) is a framework that measures how much you can trust the *process* that produced a software artifact, not the artifact's code quality directly. It's a maturity ladder: Level 0 is no guarantees at all; Level 1 means an automated build produces some provenance record, but that record is self-reported by the build script itself, so a compromised script could fake it; Level 2 requires that provenance be generated and signed by a trusted, hosted build platform instead of the build script; Level 3 is the meaningful jump — it requires the build environment to be isolated and tamper-resistant per build, so that even a compromised dependency or malicious build step can't forge or influence the provenance record, because the untrusted build logic has no access to the trusted provenance-generation mechanism. This matters concretely because of attacks like SolarWinds, where the build process itself — not the shipped source code — was compromised; scanning and even signing the final artifact wouldn't have caught that, because the compromised build process would produce a validly-signed, apparently-clean artifact. SLSA Level 3 is designed specifically to make that class of attack structurally much harder, not just detect it after the fact."
