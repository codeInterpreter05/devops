# Day 75 — Quiz: Database Migrations in CI/CD

Try to answer without looking at your notes. Answers are at the bottom.

1. Why does a rolling deployment with 3 replicas make a single-step, breaking schema change dangerous, even if the migration itself runs instantly?
2. Walk through the three phases of expand-contract for adding a `NOT NULL` column with no default.
3. Why should a large backfill run in small batches instead of one giant `UPDATE` statement?
4. What is the core mechanism (the thing they all have) shared by Flyway, Liquibase, and Alembic that lets them know which migrations still need to run?
5. What happens if you manually run a migration's SQL directly against production, bypassing the migration tool entirely?
6. Why is Alembic's `--autogenerate` unable to detect a column rename, and what does it generate instead?
7. Why is a Kubernetes Job generally preferred over an init container for running a schema migration before a deploy?
8. Why is schema rollback fundamentally asymmetric with application code rollback?
9. In a Blue/Green deployment sharing one database, why must the contract-phase migration wait until after the rollback window (or Blue's decommission)?
10. What's the practical difference between Flyway's philosophy on rollback and Liquibase's?
11. **Interview question:** How do you deploy a database schema change with zero downtime when you have 3 running replicas?

---

## Answers

1. Because a rolling deployment is never instantaneous — for some window of time, old-version and new-version code run simultaneously against the same database. A one-step breaking change (e.g., renaming/dropping a column, or adding `NOT NULL` with no default) is guaranteed to break whichever code version's expectations don't match the new schema during that overlap window, however brief it is.
2. Phase 1 (Expand): add the column as nullable, typically with a default, so old code (which doesn't know about it) keeps working. Phase 2 (Migrate/backfill): deploy new code that actively populates the column on writes, and backfill historical rows in batches until zero remain NULL. Phase 3 (Contract): once 100% of traffic is on the new code and the backfill is verified complete (zero NULLs), add the `NOT NULL` constraint.
3. A single giant `UPDATE` on a large table holds a long-running transaction/lock, which can block other writes, bloat the write-ahead log/transaction log, and risk replication lag or lock/statement timeouts. Small batches with brief pauses between them keep individual transactions short and let other traffic interleave normally.
4. All three maintain a migration history table inside the target database itself (Flyway: `flyway_schema_history`; Liquibase: `DATABASECHANGELOG`; Alembic: `alembic_version`) that records which migrations have already been applied, letting the tool diff that against what's available in the codebase and apply only what's missing, in order.
5. It desyncs the tool's tracked history table from the database's actual state. The next legitimate migration run through the tool can then fail unexpectedly (checksum mismatch, unexpected schema state) or, in worse cases, silently skip or re-apply something, because the tool's record of "what's already been done" no longer matches reality.
6. Autogenerate works by diffing schema *shape* (does this column exist or not), not by tracking intent — a rename looks identical, structurally, to "one column was dropped and an unrelated new column was added." It generates a drop+add pair, which — if actually run as-is — would silently lose the data that was in the original column, so any rename must be hand-edited into an explicit rename operation (or handled as its own expand/backfill/contract sequence).
7. A Job runs to completion exactly once and gives you a clean, waitable success/failure signal (`kubectl wait --for=condition=complete`) that can gate the subsequent Deployment rollout, matching the "run once, block on result" semantics a migration actually needs. An init container runs once *per pod*, so with N replicas it runs N times concurrently as pods start, relying entirely on the migration tool's own idempotency/locking to avoid races on its history table — riskier and harder to reason about.
8. Rolling back application code just means redeploying a previous, already-tested container image — cheap and non-destructive. Rolling back a schema change, especially a contract-phase one (e.g., a dropped column), can be destructive: the data that used to live in that column is genuinely gone, and there's no "undo" that recreates data from nothing — so schema rollback capability has to be designed in ahead of time (or avoided by simply not contracting until you're sure), not assumed to mirror code rollback's safety.
9. Blue and Green share the same database, and during the rollback window Blue is kept "warm" specifically so traffic (or a rollback decision) can go back to it if Green has a problem. If the contract migration (which removes something Blue's code still depends on) runs before that window ends, rolling back to Blue is no longer safe — Blue would immediately break against the now-contracted schema, eliminating the rollback safety net that Blue/Green is supposed to provide.
10. Flyway (community edition) treats migrations as one-way, immutable, versioned SQL files with no built-in "down" migration — reversing a change means authoring a new forward migration that undoes it. Liquibase treats rollback as a first-class, explicitly authored block defined alongside the forward changeset at authoring time, letting you run `liquibase rollback-count N` to invoke those pre-written rollback steps directly.
11. Strong answer: "I'd use the expand-contract pattern rather than a single breaking migration, because with 3 replicas doing a rolling update, there's always a window where old and new code run simultaneously against the same database. First, I run an expand-only migration — say, adding a new nullable column with a default — as a Kubernetes Job that completes before the rollout starts, so it's applied exactly once and doesn't break either code version. Then I roll out the new application version, which writes to the new column going forward, and run a batched backfill for historical rows so nothing gets locked for long. Only after confirming the rollout is 100% complete and a verification query shows zero NULLs remaining do I run a separate contract migration — also as its own Job — to add any constraint like `NOT NULL` or drop anything the old code path used. The key principle is decoupling the schema-change timeline from the code-rollout timeline entirely, since they're never atomic together in a rolling deploy, and treating any destructive step as a one-way door that only happens after full cutover is verified, not assumed."
