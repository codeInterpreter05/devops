# Day 28 — Quiz: Helm & Kustomize

Try to answer without looking at your notes. Answers are at the bottom.

1. In `Chart.yaml`, what's the difference between the `version` field and the `appVersion` field?
2. What command should you run before `helm install`/`upgrade` in production to see exactly what YAML will be applied, without touching the cluster at all?
3. What happens if you `kubectl edit` a resource that's managed by a Helm release, and then later run `helm upgrade` on that same release with no other changes?
4. What is a Helm hook, and what's the canonical use case for a `pre-upgrade` hook?
5. What does `--atomic` do on `helm upgrade`, and why is it recommended for automated pipelines?
6. Since which Helm version can charts be distributed via OCI registries, and what's the practical benefit of doing so instead of a traditional chart repository?
7. What is the fundamental philosophical difference between how Helm and Kustomize handle per-environment configuration differences?
8. What's the difference between a Kustomize strategic merge patch and a JSON 6902 patch? When would you need the latter?
9. Does Kustomize require installing a separate binary to use with `kubectl`? What about tracking release history/rollback — does Kustomize provide that natively?
10. Give a concrete example of when you'd choose Helm over Kustomize, and a concrete example of when you'd choose Kustomize over Helm.
11. What is a common pattern for using Helm and Kustomize together, particularly in GitOps pipelines like ArgoCD/Flux?
12. **Interview question:** What are the limitations of Helm? When would you choose Kustomize instead?

---

## Answers

1. `version` is the chart's own version (bump it when the chart's templates/structure change). `appVersion` is the version of the application the chart deploys (typically matching the image tag). They're independent fields that can change on completely different schedules — a chart's `version` might bump for a templating fix with no application code change at all, and vice versa.
2. `helm template <release> ./chart -f values.yaml` (renders locally with no cluster interaction) or `helm upgrade/install ... --dry-run --debug` (renders and validates against the live cluster's API without actually applying). Either should be the default first step before any real install/upgrade.
3. Nothing changes immediately from the manual edit, but the **next** `helm upgrade` (even with no actual chart/values changes) will reconcile the live resource back to whatever Helm's tracked release state says it should be — silently reverting the manual edit. Helm reconciles against its own tracked state (the chart + values used at install/upgrade time), not the cluster's actual current state, so manual out-of-band edits are not "remembered" and get overwritten on the next Helm operation.
4. A Helm hook is a Kubernetes resource (commonly a `Job`) annotated with `helm.sh/hook` that Helm runs at a specific point in a release's lifecycle (e.g., `pre-install`, `post-upgrade`) rather than as a regular tracked chart resource. The canonical use case is a **database migration Job run via `pre-upgrade`**, which guarantees the schema migration completes successfully before the new application version's pods are rolled out — and blocks the upgrade entirely if the migration fails, rather than deploying new app code against an incompatible database schema.
5. `--atomic` makes `helm upgrade` automatically roll back to the previous successful release revision if the upgrade fails or times out, instead of leaving the release in a partially-applied, half-upgraded state. It's recommended for automated pipelines because a failed automated deploy shouldn't require a human to notice and manually run `helm rollback` — the safety net should be built into the deploy step itself.
6. Since Helm 3.8. The practical benefit is that charts can be pushed/pulled from **any OCI-compliant registry** (ECR, GHCR, Harbor, ACR, Docker Hub) — the same registry infrastructure you already use for container images — inheriting its access control, audit logging, and (increasingly) vulnerability scanning and artifact signing tooling, instead of requiring a separate, special-purpose chart-repository server.
7. Helm parameterizes YAML via a **templating language** (Go templates + Sprig functions, `{{ }}` placeholders) filled in from a values file per environment. Kustomize takes the opposite approach: base YAML is plain, valid, non-templated Kubernetes manifests, and per-environment differences are expressed as declarative **patches** (strategic merge or JSON 6902) layered on top via overlays — no templating syntax anywhere.
8. A strategic merge patch is YAML that looks like a partial version of the target resource, merged in using Kubernetes' own field-merge semantics (e.g., matching array elements like containers by their `name` field rather than blindly overwriting the whole array). A JSON 6902 patch is an explicit, path-based list of operations (`add`/`remove`/`replace`/etc.) — you'd reach for it when you need a precise edit a strategic merge can't express cleanly, such as removing one specific element from an array by index or renaming a field.
9. No — Kustomize is built directly into `kubectl` (`kubectl apply -k`, `kubectl kustomize`), so no separate binary is strictly required (though a standalone `kustomize` CLI also exists with a few extra features). Kustomize does **not** provide built-in release tracking or rollback the way Helm does (`helm history`/`helm rollback`) — any rollback/history story with Kustomize relies entirely on your own git history or GitOps tool's deployment history.
10. Choose Helm when distributing a reusable, versioned, packaged application to multiple teams/environments, or when consuming third-party software that ships as a chart with real dependency management needs (e.g., installing `bitnami/redis` or `ingress-nginx`'s official chart). Choose Kustomize when you own the YAML directly and just need small, declarative per-environment differences (replica count, resource limits, a couple of labels) without needing packaging, versioning, or a templating language at all — especially inside a GitOps pipeline that's already managing plain YAML in git.
11. Render the base configuration via `helm template` (turning a chart plus its default/base values into plain YAML), then apply Kustomize patches on top of that rendered output for final, environment-specific tweaks — combining Helm's packaging/dependency/versioning strengths with Kustomize's simpler, template-free per-environment patching. Tools like ArgoCD and Flux natively support this "Helm then Kustomize" rendering pipeline as a first-class deployment pattern.
12. Strong answer: "Helm's main limitations are: its templating language can get genuinely hard to read and debug at scale (nested conditionals and helper functions inside YAML-in-a-templating-language is not a pleasant debugging experience); it requires learning Helm-specific concepts (charts, releases, hooks, values precedence) on top of Kubernetes YAML itself; and because it reconciles against its own tracked release state rather than the cluster's live state, manual out-of-band changes get silently reverted on the next upgrade, which can surprise people. I'd choose Kustomize instead when I own the YAML directly and just need small, declarative differences between environments — no templating language needed, it's built into kubectl with no extra tooling, and the base manifests stay plain, valid, readable Kubernetes YAML the whole way through. In practice, though, it's often not strictly either/or — I'd use Helm to consume third-party charts with real packaging/dependency needs, and Kustomize (or Helm plus a Kustomize patch layer) for my own application's straightforward per-environment tweaks."
