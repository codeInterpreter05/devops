3. What's the difference between an `Application`'s Sync status and its Health status? Can an Application be Synced and Unhealthy at the same time?
4. What does `prune: true` do, and what happens to a resource removed from Git if `prune` is left off?
5. What does `selfHeal: true` do, and what operational habit does it break if you're not careful during an incident?
6. What is a sync wave, and what does ArgoCD wait for between two waves before proceeding to the next?
7. Name the four hook phases and give a concrete use case for `PreSync` and one for `SyncFail`.
8. Why do custom CRDs show `Unknown` health by default, and how do you fix that?
9. What are the three composable pieces of the ArgoCD Notifications engine, and what does each one define?
10. What problem does `ApplicationSet` solve that a single `Application` resource doesn't, and name two generator types.
11. What does `argocd app rollback` actually do — does it revert your Git repository's commit history?
12. **Interview question:** What is GitOps? How does ArgoCD implement the pull vs push deployment model?

---

## Answers

1. GitOps is a deployment model where the desired state of a system is declared in Git, and an automated agent continuously reconciles the live system to match it. It's "pull" because the reconciling agent runs *inside* the target environment (the cluster) and pulls changes from Git on its own schedule/trigger — no external system pushes changes into the cluster or needs cluster-admin credentials to do so.
2. `repo-server` — renders manifests from Git; not responsible for talking to the Kubernetes cluster at all. `application-controller` — diffs live vs. desired state and reconciles (applies changes); not responsible for rendering manifests or serving the API/UI. `api-server` — the external gateway for CLI/UI/SSO/RBAC; not responsible for actually reconciling cluster state itself (it delegates to the controller).
3. Sync status answers "does live cluster state match Git?" Health status answers "is the deployed thing actually working?" Yes — an Application can be fully Synced (matches Git exactly) while still Unhealthy (e.g., pods stuck in `CrashLoopBackOff` due to a bad image or bad config that was, itself, faithfully applied from Git).
4. `prune: true` deletes cluster resources that were previously managed by the Application but have since been removed from the Git source. Without it, deleting a manifest from Git only stops ArgoCD from tracking/managing that resource going forward — the resource itself remains running in the cluster indefinitely, orphaned.
5. `selfHeal: true` makes ArgoCD automatically revert any manual change made directly to a live resource (e.g., `kubectl scale`, `kubectl edit`) back to what's declared in Git, on the next reconciliation loop. It breaks the habit of "quickly patch prod directly to unblock an incident" — that patch will be silently reverted within the reconciliation interval unless automated sync is paused first.
6. A sync wave is an integer (via the `argocd.argoproj.io/sync-wave` annotation, default 0, negative allowed) that orders resource application within a single sync operation. Between waves, ArgoCD applies all resources in the current (lowest remaining) wave and waits for them to reach a `Healthy` status before proceeding to apply the next wave's resources.
7. `PreSync` (before any resources are applied — e.g., database migrations that must complete before new app code depending on the new schema starts), `Sync` (interleaved with regular resource application, ordered by wave), `PostSync` (after all resources are applied and healthy — e.g., smoke tests or cache warm-up), `SyncFail` (only runs if the sync fails — e.g., triggering an automated rollback or incident alert).
8. ArgoCD only has built-in, type-aware health logic for well-known Kubernetes resource types (Deployment, Service, Job, PVC, etc.) — it has no generic way to know what "healthy" means for an arbitrary CRD's `status` schema. You fix it by writing a custom health check as a Lua script under `resource.customizations.health.<group>_<kind>` in the `argocd-cm` ConfigMap, which inspects the CR's `obj.status` fields and returns a `Healthy`/`Progressing`/`Degraded`/etc. verdict.
9. `service` (how to deliver a notification — Slack token, SMTP config, webhook, etc.), `template` (what the message says, using Go templating over the `Application` object's fields), and `trigger` (the condition, evaluated against live Application state, that decides when to fire and which template+service to use).
10. `ApplicationSet` solves the problem of hand-maintaining many near-identical `Application` resources across environments, clusters, or microservices — instead, you define one template plus a generator that produces the varying parameters, and the controller creates/updates/deletes the corresponding `Application` resources automatically. Any two of: `list` (static, explicit set), `cluster` (one per registered cluster), `git` (directory/file generator scanning a repo for matches), matrix (cartesian product of two generators).
11. It re-applies a previously recorded revision's rendered manifests to the cluster (from ArgoCD's own sync history) — it does **not** touch or revert your Git repository's commit history at all. Since ArgoCD (especially with automated sync) will reconcile back toward whatever HEAD currently is, a rollback that isn't followed by a matching Git revert can get overwritten by the next automated sync unless you've also paused automated sync or actually fixed/reverted Git to match.
12. Strong answer: "GitOps is a model where Git is the single source of truth for desired system state, and an automated in-cluster agent continuously reconciles live state to match it, rather than an external CI pipeline pushing changes directly. ArgoCD implements the pull side via its `application-controller`, which runs inside the cluster, continuously compares live resources against manifests rendered by the `repo-server` from a Git source, and either reports drift or actively applies changes depending on the sync policy — automated or manual. Because the reconciling agent lives inside the cluster and initiates all the pulling, no external CI/CD system ever needs standing cluster-admin credentials, which meaningfully shrinks the blast radius compared to a traditional push-based pipeline that runs `kubectl apply` from outside using stored cluster credentials."
