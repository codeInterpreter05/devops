2. Name the six CKS curriculum domains. Which one covers Falco, seccomp, AppArmor, and audit logging?
3. What are the four audit log "Stages" and the four "Levels," and are they the same axis or two independent axes?
4. Why does adding `--audit-policy-file` and `--audit-log-path` to the kube-apiserver static pod manifest, by itself, often fail to work as expected?
5. What's the difference between what seccomp restricts and what AppArmor restricts? Why are they complementary rather than redundant?
6. Why is running kube-bench's default `master` benchmark against a managed EKS/GKE/AKS cluster the wrong move, and what should you do instead?
7. What's the difference between CI-time image scanning (`trivy image` in a pipeline) and the Trivy Operator running in-cluster? Why do you need both?
8. Why should container images be signed and verified by digest rather than by tag?
9. What's the difference between `validationFailureAction: Audit` and `Enforce` in a Kyverno policy, and why would you deliberately start with `Audit` on a new policy?
10. What is the single most common mistake teams make when writing a default-deny-egress NetworkPolicy, and why does it produce a confusing symptom rather than an obvious one?
11. Why is loading an AppArmor or `Localhost` seccomp profile onto only one node in a multi-node cluster a subtle problem rather than an immediately obvious one?
12. **CKS self-assessment question (no interview question was assigned for today — this row's prep is exam-simulation-focused, not interview-focused):** Given a killer.sh mock-exam score with several missed tasks, what's your process for turning that score into an actual study plan rather than just re-attempting the simulator?

## Answers

1. An active CKA certification. CKS assumes you can already administer a cluster fluently — fast `kubectl`, YAML editing, static pod manifest changes, troubleshooting — and spends its entire time budget testing security specifically on top of that baseline; without CKA-level speed already automatic, there wouldn't be enough time left in the exam to also demonstrate the security-specific skills.
2. Cluster Setup, Cluster Hardening, System Hardening, Minimize Microservice Vulnerabilities, Supply Chain Security, and Monitoring/Logging and Runtime Security. Falco, seccomp, AppArmor, and audit logging fall under **Monitoring, Logging and Runtime Security**.
3. Stages (`RequestReceived`, `ResponseStarted`, `ResponseComplete`, `Panic`) describe *when in the request lifecycle* an event is emitted. Levels (`None`, `Metadata`, `Request`, `RequestResponse`) describe *how much detail* is captured for a matched request. They're independent axes — a Policy rule's `omitStages` field controls the former, `level` controls the latter.
4. Because those flags reference a policy file and a log directory that must actually be mounted into the kube-apiserver static pod via matching `volumeMounts`/`volumes` entries in the same manifest. Adding the flags without the volume entries means the apiserver container can't find the referenced paths, and the static pod crash-loops on restart — silently, since the API server itself being down means there's no `kubectl` available to report the error.
5. Seccomp restricts *which syscalls* a process is permitted to call at all (the kernel API surface). AppArmor restricts *what those permitted syscalls can act on* — specific file paths, network operations, and capabilities — enforced as mandatory access control regardless of the process's own UID/permissions. They're complementary because a syscall can be seccomp-permitted in general but still AppArmor-denied against a specific sensitive target.
6. Managed control planes aren't administered by you, so most `master`-targeted CIS checks are inapplicable — you have no ability (or need) to remediate them, and running the vanilla benchmark produces a wall of FAILs for things outside your control. Instead, use the cloud-specific benchmark variant kube-bench supports (e.g., `--benchmark eks-1.0`), which reflects the checks actually relevant to what you control on that platform.
7. CI-time scanning blocks a vulnerable image from ever being pushed into the pipeline/registry in the first place. The Trivy Operator continuously scans workloads already running in the cluster, catching images that entered outside the pipeline, or images whose base image had a new CVE disclosed *after* they were already deployed and running. Both are needed because each only covers a different point in time — one at push, one continuously afterward.
8. Tags are mutable — the same tag (e.g., `myapp:latest` or even a specific version tag, depending on registry policy) can be repointed to different image content later, silently invalidating an earlier signature check performed against "the tag." A digest is an immutable hash of the actual content, so signing/verifying by digest guarantees you're checking the exact bytes that were signed, not whatever currently happens to sit behind a mutable name.
9. `Audit` reports policy violations without blocking anything; `Enforce` actively rejects non-compliant requests at admission time. Starting a new policy in `Audit` mode lets you catch policy bugs or unanticipated-but-legitimate workload patterns (false positives) before they start blocking real deployments cluster-wide, mirroring standard progressive-rollout discipline for any production-impacting change.
10. Forgetting to include an explicit egress allow rule for DNS (UDP/TCP port 53 to CoreDNS/kube-dns) alongside the default-deny-egress rule. It's confusing rather than obvious because the resulting failure — every pod in the namespace losing the ability to resolve service/domain names — often surfaces through client libraries as a generic "connection failed"-style error rather than a clearly labeled DNS resolution failure, making the real root cause much harder to spot quickly.
11. Because the pod's success or failure to start becomes *dependent on which node the scheduler happens to place it on* rather than failing consistently everywhere — it works fine on the node where the profile was loaded and fails with a confusing startup error on any other node, which looks like a flaky/intermittent problem rather than the deterministic "profile isn't loaded here" cause it actually is, unless you specifically pin scheduling or load the profile cluster-wide.
12. Strong process: score strictly against the provided official solutions (no partial credit for objects that wouldn't actually pass a real grader), then for every miss write one sentence identifying the *actual* misconception behind it rather than a surface-level excuse ("I forgot a flag" isn't useful; "I didn't know `PeerAuthentication` precedence meant a namespace-level policy overrides mesh-wide STRICT" is). Then map each miss to one of the six CKS curriculum domains — this converts a raw score into a concrete list of exactly which domain(s) need more study time before the next attempt or the real exam, rather than just re-running the simulator and hoping repetition alone closes the gaps.
