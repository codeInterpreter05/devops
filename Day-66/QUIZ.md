# Day 66 — Quiz: Shift-Left Security

Try to answer without looking at your notes. Answers are at the bottom.

1. What does SAST analyze, and name two categories of vulnerability it's well-suited to catch.
2. What does DAST analyze, and name one category of vulnerability it can catch that SAST structurally cannot.
3. Why is it wrong to treat SAST and DAST as redundant/interchangeable, and pick just one?
4. Why must SCA (dependency scanning) walk the full resolved dependency tree instead of just the direct manifest entries? Name the real-world vulnerability used as the canonical example.
5. What two detection techniques do secret scanners like Gitleaks/TruffleHog combine, and what does TruffleHog's `--only-verified` flag additionally do?
6. Why does deleting a secret from the current file and committing the fix NOT actually remove the exposure?
7. In the correct incident-response order for a leaked credential, what should happen first — revoking the credential, or rewriting Git history to remove it? Why?
8. What's the key difference between a pre-commit hook and a CI-level security check, and why can't a pre-commit hook alone be trusted as the sole enforcement mechanism?
9. Why does pre-commit hook speed matter operationally, and what's one concrete technique to keep hooks fast?
10. Why is suppressing a SAST finding with an inline ignore comment (`# nosec`, etc.) not equivalent to fixing it?
11. What makes developer security training "just-in-time" and contextual actually more effective than an annual disconnected training video, according to today's notes?
12. **Interview question:** A developer accidentally committed an AWS key to a public repo. What do you do in the next 5 minutes?

---

## Answers

1. SAST analyzes source code directly, without executing it (static analysis of the codebase/AST). Any two of: hardcoded credentials, classic injection patterns (SQL built via string concatenation, unsanitized shell command construction), use of dangerous/deprecated functions or crypto primitives, taint-traceable flows from untrusted input to a dangerous sink.
2. DAST analyzes a running application from the outside, sending real requests and observing real responses (dynamic testing). It can catch things SAST cannot, such as: actual exploitable authentication/session bypass flaws only observable via real request/response behavior, misconfigured security headers or exposed debug endpoints in the *deployed* environment (which may differ from what's in code, e.g., stripped by a proxy), or business-logic flaws requiring multi-step interaction to observe.
3. They inspect fundamentally different layers — SAST looks at what the code *says* (structurally, without running it), DAST looks at what the deployed, running system actually *does* when attacked. A codebase can pass SAST cleanly while still being exploitable due to runtime misconfiguration or logic flaws, and DAST can only find what it can reach/exercise by crawling and fuzzing, missing code paths it never triggers. Skipping either leaves an entire class of vulnerabilities structurally undetected.
4. Because a vulnerability in a transitive dependency (one pulled in indirectly, several levels deep) is just as exploitable as one in a directly declared package — scanning only the top-level manifest misses these entirely. The canonical real-world example is Log4Shell (CVE-2021-44228) in `log4j-core`, which many affected organizations didn't even know they depended on because it was buried several levels deep in their dependency trees.
5. They combine regex pattern matching against known credential formats (e.g., `AKIA` prefix for AWS keys, `ghp_` for GitHub tokens, PEM headers for private keys) with entropy analysis (flagging high-randomness strings that don't match a known format but still look secret-like, as opposed to low-entropy strings like "password123"). TruffleHog's `--only-verified` additionally attempts to actually use the discovered credential against the real provider's live API to confirm whether it's currently active/exploitable, turning a suspected finding into a confirmed one and sharply reducing false positives.
6. Because `git rm`/editing the current file only changes the latest commit/working tree — the secret remains fully recoverable from earlier commits in Git history (via `git log -p`, cloning the repo, etc.) unless that history is explicitly rewritten (e.g., with `git filter-repo` or BFG Repo-Cleaner) and force-pushed, and even then, anyone who already cloned the repo before the rewrite retains a copy regardless.
7. Revoking/rotating the credential at the provider must happen first, immediately. Git history cleanup is necessary hygiene but is not sufficient on its own and does not need to happen before revocation — the exposed credential remains live and exploitable for as long as it isn't revoked, regardless of what the repository's history looks like, and automated bots scan public repos for exposed secrets within minutes, so delaying revocation to first "clean up the repo" leaves the real vulnerability (a live, usable credential) open the entire time.
8. A pre-commit hook runs locally on a developer's machine before a commit completes, and can be trivially bypassed (`git commit --no-verify`, or simply never running `pre-commit install` on a fresh clone) — it depends on individual developer setup and discipline. A CI-level check runs centrally, on every pushed commit/PR, regardless of any individual developer's local configuration, and cannot be skipped by a local flag — making it the actual non-bypassable enforcement gate. Pre-commit hooks should supplement, not replace, CI-level scanning.
9. Slow local hooks (minutes per commit) create enough friction that developers are strongly incentivized to bypass or disable them, defeating their purpose entirely — this is a well-documented adoption failure mode. One concrete technique: scan only staged/changed files locally (incremental scanning) rather than the full repository on every commit, reserving full-repo scans for CI or periodic scheduled runs.
10. A suppression comment removes the finding from the tool's *report*, making the pipeline pass, but does nothing to change the actual code — the underlying vulnerability (e.g., the SQL injection pattern, the unsafe shell call) still ships to production exactly as before. It creates a false sense of resolution while leaving the real risk in place.
11. Just-in-time, contextual training ties the explanation to a real finding or design decision at the exact moment it's relevant (e.g., a SAST finding linking directly to why that specific vulnerability class matters, or a threat-modeling discussion during design review) — this is far more likely to actually change behavior and retain understanding than a disconnected annual compliance video with no immediate, concrete trigger tying it to the developer's actual current work.
12. Strong answer: "First, revoke/rotate the exposed AWS credential immediately at the IAM level — this is the only action that actually closes the exposure, since automated bots scan public repos for leaked secrets within minutes. Second, check CloudTrail (or equivalent access logs) for any usage of that key since it was exposed, to determine if there's already been unauthorized activity requiring further incident response. Third, remove the secret from the current code and rewrite Git history to purge it from past commits (git filter-repo/BFG), understanding this is hygiene, not the fix itself, and coordinate a force-push with any collaborators. Fourth, rotate any other credentials that might have been co-located or could be derived from the same source. Finally, make sure secret scanning (pre-commit and CI-level) is actually in place going forward so this doesn't recur silently. The critical point is ordering: revocation comes first, before any Git surgery, because the live credential is the actual risk, not the repo's history."
