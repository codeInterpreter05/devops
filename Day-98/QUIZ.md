# Day 98 — Quiz: Toil Elimination & Reliability Patterns

Try to answer without looking at your notes. Answers are at the bottom.

1. Give the full, precise definition of toil (all six properties), and explain the single sharpest test for whether something is toil.
2. Why does Google's SRE model cap toil at roughly 50% of an SRE's time rather than trying to drive it to zero or ignoring it as "just the job"?
3. Give an example of tedious work that is NOT toil in the strict SRE sense, and explain why.
4. Describe the three circuit breaker states and what triggers each transition.
5. Why is "fail fast" (an open circuit breaker) better for the calling service than letting every request time out normally?
6. What problem does the bulkhead pattern solve that a circuit breaker alone does not?
7. What is a retry storm, and what two techniques together prevent it?
8. Why is jitter necessary even if you already have exponential backoff?
9. Define idempotency, and explain concretely why a network timeout makes "just retry" unsafe without it.
10. How does an idempotency key work, mechanically, on both the client and server side?
11. Give one example of a naturally idempotent operation and one naturally non-idempotent operation, plus how to make the non-idempotent one safe.
12. **Interview question:** What is toil? Give me a real example and how you eliminated it.

## Answers

1. Toil is work that is manual (a human must do it), repetitive (done over and over), automatable (a machine could do it instead), tactical (reactive, not strategic), devoid of enduring value (the system is in the same state after as before), and scales linearly with service growth. The sharpest test: does doing this task leave the system durably better afterward, or does the same amount of work recur regardless of how many times it's been done? If nothing changes, it's toil.
2. Because toil scales linearly (or worse) with growth — as traffic/service count grows, toil grows proportionally, requiring proportionally more headcount just to keep up forever. Engineering project work is a one-time investment that durably reduces future toil, so time spent there compounds in the opposite direction. Toil is also a morale/retention problem independent of the math — unaddressed toil burns people out.
3. Triaging a genuinely novel, ambiguous incident is tedious but not toil, because it requires real judgment (not automatable) and, once resolved and documented (e.g., turned into a runbook or a fix), it can leave the system durably better — it's not automatically repetitive with no lasting value. (Any answer identifying work that fails the "automatable" or "no enduring value" property, with correct reasoning, is acceptable.)
4. Closed (normal state, calls pass through, breaker watches the failure rate) transitions to Open once failures exceed a configured threshold (e.g., a high proportion of recent calls failed) — in Open, calls fail immediately without attempting the dependency, for a cooldown period. After the cooldown, it moves to Half-Open, allowing a small number of trial requests; success transitions back to Closed, failure sends it back to Open for another cooldown.
5. Because letting every request time out normally means the calling service holds a thread/connection open for the full timeout duration on every attempt — multiplied across many concurrent requests, this exhausts the calling service's own resource pool, making it unable to serve even unrelated requests. Failing fast (open breaker) returns in milliseconds, preserving the caller's capacity to serve everything else, and also reduces load on the already-struggling dependency.
6. A circuit breaker limits how often/long you keep calling a known-unhealthy dependency, but doesn't prevent the *initial* resource exhaustion if all outbound calls (to different dependencies) share one pool — a bulkhead partitions resources (thread/connection pools) per dependency or criticality tier, so one dependency's failure can only exhaust its own partition, leaving unrelated calls unaffected. They solve different problems and are meant to be used together.
7. A retry storm is when many failing clients retry simultaneously (and often repeatedly, at the same fixed interval) against an already-struggling dependency, adding enough extra load to prevent or delay its recovery — effectively becoming the sustaining cause of the outage. Exponential backoff (progressively longer delays between attempts) combined with jitter (randomizing the delay) prevents it — backoff reduces the aggregate retry rate over time, and jitter prevents synchronized retry waves at each backoff interval.
8. Because exponential backoff alone still produces synchronized retry spikes: if many clients fail at the same moment and all compute the exact same backoff schedule, they all retry again at the same subsequent instants (just spaced further apart in discrete waves rather than continuously) — jitter randomizes each client's actual delay so the retries are smeared out over time instead of clustering at each backoff interval.
9. Idempotency means performing an operation multiple times has the same effect as performing it once. A network timeout is unsafe to blindly retry without idempotency because the caller cannot distinguish "the operation never happened" from "the operation succeeded server-side but the response was lost" — retrying a non-idempotent operation (like "charge card") in the latter case duplicates the side effect (double charge), even though from the caller's perspective it looked like a clean retry of a failed request.
10. The client generates a unique key (typically a UUID) once, before the first attempt, and sends the same key with every retry of that logical operation. The server checks, before processing, whether it has already seen that key; if so, it returns the stored result of the original processing without re-executing the side effect; if not, it processes normally and stores the key + result for a retention window long enough to cover realistic retry timeframes.
11. Naturally idempotent: `DELETE resource_id` (deleting an already-deleted resource is a no-op) or `SET x = 5`. Naturally non-idempotent: `INSERT` without a uniqueness constraint (creates duplicate rows) or `INCREMENT x BY 1` (changes result each time). Made safe via an upsert on a natural unique key (`INSERT ... ON CONFLICT DO NOTHING`) or an explicit idempotency-key mechanism at the API layer.
12. Strong answer: "Toil is operational work that's manual, repetitive, automatable, tactical, leaves no lasting improvement, and scales with the system's growth — the test is whether the task leaves things durably better or just recurs unchanged. A real example: [describe a specific recurring manual task from your own experience — e.g., manually restarting a service that leaked memory every few days, or manually re-running failed batch jobs because retries weren't implemented]. I measured it at roughly [X hours/month], then eliminated it by [specific automation — e.g., writing a scheduled health-check-and-restart script, or adding automatic retry-with-backoff to the batch job], which brought that recurring manual work down to zero and freed that time for actual engineering project work."
