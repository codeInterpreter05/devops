# Day 10 — Quiz: Docker Deep Dive I

Try to answer without looking at your notes. Answers are at the bottom.

1. Why does a naive single-stage Dockerfile for a Python app often end up over 1GB, even though the actual application code might be a few kilobytes?
2. What is a Docker multi-stage build, and what specifically determines which files end up in the final image?
3. What is a distroless image, and what's missing from it compared to a normal base image like `python:3.11-slim`?
4. When is `scratch` a viable base image, and why does it break for a normal dynamically linked binary?
5. Explain how Docker's build cache invalidation works: what determines a cache miss, and what happens to the layers after it?
6. Why does the ordering `COPY package.json . && RUN npm install` before `COPY . .` matter for cache efficiency?
7. What does `.dockerignore` do, and what are two distinct problems it prevents?
8. Why is passing a secret via `ARG` or `ENV` insecure, even if a later instruction unsets or overwrites it?
9. What does BuildKit's `RUN --mount=type=secret` do differently, and what two conditions must both be true for that syntax to work at all?
10. What is `dive` used for, and what specific metric does it surface that `docker images` alone cannot show you?
11. Name two `hadolint` rules and what each one checks for, and describe where you'd wire `hadolint` into a real pipeline.
12. **Interview question:** "How would you reduce a 1.2GB Python image to under 100MB? Walk me through the process."

---

## Answers

1. Every layer in a Docker image is additive and permanent — the full `python:3.11` base (900MB+ of Debian plus the CPython build toolchain), the entire build context copied via `COPY . .` (including `.git`, tests, docs, `__pycache__` if there's no `.dockerignore`), and `pip`'s build cache/any transient compiler used for source-built wheels all end up baked into the image. None of it is needed at runtime, but nothing removes bytes that a previous layer already committed.
2. A multi-stage build uses one or more disposable "builder" stages (started with their own `FROM`) to compile/install things, and a separate final stage that becomes the shipped image. Only files explicitly moved with `COPY --from=<stage>` cross into the final stage — everything else about a builder stage (its base OS, its toolchain, anything not copied out) is discarded once the build finishes.
3. Distroless (`gcr.io/distroless/*`) images contain only the application runtime and its dependencies — no shell (`/bin/sh` doesn't exist), no package manager, and no general-purpose coreutils. Compared to `python3-slim`, which still has `apt`, a shell, and a full Debian userland, distroless strips all of that, leaving a much smaller image and a much smaller attack surface, at the cost of not being able to `docker exec` a shell into it (without the `:debug` tag variant).
4. `scratch` is Docker's reserved keyword for a completely empty filesystem — no libc, no shell, nothing. It's only viable for a binary with zero dynamic dependencies: a statically linked Go binary built with `CGO_ENABLED=0`, or a Rust binary built against a musl target. A normal dynamically linked binary expects a dynamic linker and libc to be present at runtime; on `scratch` those don't exist, so the container fails to start with a confusing `exec format error` or "not found" even though the file is clearly present.
5. Each instruction is cached based on its own key: `RUN`'s key is the literal command string (not what it did on disk), while `COPY`/`ADD`'s key is a checksum of the actual files being copied. The layer cache is a chain — the moment one instruction misses, every instruction after it in that stage reruns, regardless of whether their own individual inputs changed.
6. `package.json`/`package-lock.json` change far less often than application source. Copying and installing from them first means a source-only change only invalidates the later `COPY . .` layer, leaving the `npm install` layer served from cache. Reverse the order (`COPY . .` before installing) and any source change invalidates the install step too, since it's now downstream of a layer that just changed.
7. `.dockerignore` excludes matching paths from the build context — the tarball sent to the Docker daemon before the build starts. It solves two separate problems: (1) a smaller, faster build context (excluding a large `.git` history or `node_modules` that would otherwise be transferred on every build regardless of caching), and (2) cache stability and secret safety — files like a local `.env` or IDE artifacts can't invalidate the cache unexpectedly or accidentally get baked into a layer via a broad `COPY . .`.
8. Image layers are permanent. `ARG`/`ENV` values are recorded in the layer's command history (visible via `docker history --no-trunc`) and, for `ENV`, in the image's config manifest (`docker inspect`, `.Config.Env`). A later instruction that unsets or overwrites the value only changes what a *new* layer or runtime container sees — it doesn't remove the value from the earlier layer, which still exists in the image and is still extractable by anyone who can pull it.
9. `RUN --mount=type=secret,id=<name>` mounts the secret as a file at a temporary path (`/run/secrets/<name>` by default) that exists only for the duration of that single `RUN` instruction's process — it's never written into the committed layer diff, so it never appears in `docker history` or `docker inspect`. It requires both BuildKit to be the active builder (`DOCKER_BUILDKIT=1` or a modern default-BuildKit Docker install) and `# syntax=docker/dockerfile:1` as the first line of the Dockerfile — the legacy builder's parser doesn't understand `--mount` and fails outright without both.
10. `dive` inspects a built image layer by layer, showing what each layer added, modified, or removed, plus a computed **wasted space** figure — bytes present in the image due to duplication or unnecessary files across layers, and an overall efficiency score. `docker images` only reports a single total size number; it can't tell you which layer is responsible or how much of that size is avoidable duplication.
11. Examples: DL3006 requires pinning the `FROM` image to a specific tag rather than `latest`, so rebuilds stay reproducible; DL3009 requires clearing `apt-get`'s package list cache (`rm -rf /var/lib/apt/lists/*`) in the same `RUN` as the install, so the cache doesn't survive as dead weight in its own layer. Wire `hadolint Dockerfile` in as a pre-build CI step — it exits non-zero on unignored violations, so a bad Dockerfile fails the pipeline before time is spent building and pushing it.
12. Strong answer: "First I'd establish a baseline — build the current Dockerfile and check `docker images` and `docker history` to see where the size actually comes from; it's almost always the base image plus the full build toolchain and build context landing in the final image. Then I'd refactor to a multi-stage build: a `builder` stage on something like `python:3.11-slim` that installs dependencies (`pip install --user --no-cache-dir -r requirements.txt`), and a final stage on `gcr.io/distroless/python3-debian12` that only copies in the installed packages (`COPY --from=builder /root/.local /root/.local`) and the application source — no compiler, no shell, no package manager in the shipped image. I'd add a `.dockerignore` so the build context and cache aren't polluted by `.git`, `__pycache__`, or local virtualenvs, order the Dockerfile so `requirements.txt` is copied and installed before application source for cache efficiency, and run `hadolint` to catch anti-patterns like unpinned versions along the way. Finally I'd verify the result with `dive` to confirm there's no unexpected wasted space, and with `docker images` to confirm the actual size — this pattern typically takes a naive 1.2GB image down to well under 100MB, often in the 40-70MB range depending on dependencies." Mention the security side too if asked to go deeper: none of this touches secrets, so any build-time credentials should go through BuildKit's `RUN --mount=type=secret`, never `ARG`/`ENV`.
