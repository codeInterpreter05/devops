# Day 118 — Quiz: Agentic AI Infrastructure

Try to answer without looking at your notes. Answers are at the bottom.

1. Why is LLM inference described as "memory-bound as much as compute-bound," and what specifically grows with sequence length and concurrent requests?
2. Why does tensor-parallel sharding of a large model across multiple GPUs generally require those GPUs to be on the same physical node?
3. Why are Capacity Reservations more of an availability necessity (not just a cost lever) for `p4d`/`p5`-class GPU instances specifically?
4. What is quantization, and why shouldn't a quantization level be assumed safe across different downstream tasks?
5. What problem does vLLM's PagedAttention specifically solve, and what's the OS-level concept it borrows from?
6. Why does prefix/prompt cache sharing across requests provide a meaningful concurrency win, and what has to be true about how your application structures prompts for it to actually kick in?
7. Name at least three signals an LLM observability system needs to capture that a traditional APM trace typically does not.
8. Why must a vector database be deployed as a stateful workload with persistent storage and backups, rather than treated like a stateless service replica?
9. In a RAG query path, which stage typically dominates total latency, and which typically dominates total cost — are they always the same stage?
10. Why does a larger `top-k` retrieval count in RAG increase cost even if it improves answer quality?
11. Why is rate limiting/budget control especially critical for autonomous agentic systems compared to a simple human-driven chat UI?
12. **Interview question:** Design the infrastructure for a production agentic AI system handling 10k requests/day. What are the unique DevOps challenges?

## Answers

1. Because generating each output token requires attending over all previous tokens' cached keys/values, and this **KV cache** grows linearly with sequence length and with the number of concurrent in-flight requests. At real concurrency, GPU memory capacity/bandwidth for holding and reading this cache is frequently the binding constraint, not raw FLOPs — which is different from a typical compute-bound ML inference workload.
2. Because cross-GPU communication during tensor-parallel forward passes is frequent and latency-sensitive, and only intra-node interconnects (NVLink/NVSwitch) provide low enough latency for this to be efficient. Spreading tensor-parallel shards across nodes over standard networking introduces communication overhead that typically defeats the purpose of parallelizing in the first place (cross-node parallelism generally needs pipeline parallelism plus high-bandwidth networking like EFA instead).
3. Because high-end GPU instance types have genuine, hard regional/AZ-level capacity constraints at AWS — it's realistically possible to be unable to launch an on-demand instance of that type at all during high-demand periods, regardless of budget, unlike general-purpose compute where capacity is rarely a hard blocker.
4. Quantization reduces numeric precision used for model weights (and sometimes activations) to shrink memory footprint and often increase throughput, at some cost to output quality. It shouldn't be assumed safe across tasks because quality degradation is workload-dependent — a level that's fine for casual conversational output might visibly hurt a task requiring precise structured output (code generation, numeric extraction), so it needs validation on your own evaluation data for the specific task.
5. PagedAttention solves the memory waste of naive KV cache allocation, which reserves a worst-case contiguous memory block per request sized for the maximum possible sequence length — wasteful for requests that end up shorter. It borrows the concept of OS virtual memory paging: allocating cache in small, non-contiguous, incrementally-filled blocks instead of one large upfront contiguous reservation, and can share cache blocks across requests with a common prefix.
6. Because a shared prefix (e.g., an identical system prompt across many requests) means its KV cache computation can be reused rather than recomputed for every request, directly increasing achievable concurrency per GPU. For this to actually trigger, the application needs to structure prompts so the shared portion is consistently placed and identical (e.g., same system prompt text and ordering) across calls — inconsistent prompt structuring defeats the cache-sharing optimization.
7. Any three of: separate input/output token counts, per-call dollar cost, the full prompt and completion content (or a sampled/redacted version), multi-step/agentic trace structure (a tree of chained calls, not a flat span list), and retrieval/tool-call outcomes (documents retrieved, tool arguments and results).
8. Because a vector database holds a persistent, expensive-to-rebuild index (embeddings for potentially millions of documents) — treating it as disposable/stateless risks losing that index on a routine pod restart or node replacement, requiring a slow and costly full re-embedding of the source data to recover, unlike a genuinely stateless service replica that can be safely recreated from nothing.
9. LLM generation typically dominates total latency and cost, since it scales with output tokens generated sequentially one at a time and output tokens are usually priced higher than input tokens; vector search is typically fast (tens of ms) if well-tuned. They are not always the same stage in every system — an unindexed or poorly-tuned vector search, or an oversized top-k pulling excessive context into the prompt, can make retrieval a larger latency or cost contributor than expected, so observability needs to measure each stage rather than assume generation always dominates.
10. Because every retrieved chunk included in the assembled prompt counts as additional input tokens sent to the LLM, and cost scales with total tokens processed — a larger top-k directly inflates input token count (and often latency) regardless of whether the additional retrieved context actually improves the final answer's quality.
11. Because an autonomous agent loop can generate calls far faster and in far greater volume than a human typing in a chat UI, especially if it retries, re-plans, or gets stuck in a loop due to a bug in its stopping condition — with no built-in ceiling on LLM API usage, a single such bug can produce a very large, very fast bill with no automatic circuit breaker unless rate limiting/budget enforcement is explicitly built in.
12. Strong answer: "Serve the model(s) via a GPU node pool sized for memory (not just compute), using a serving engine like vLLM with quantization tuned to the task and validated for quality, with a gateway (LiteLLM) in front providing per-team rate limits, budget caps, and provider failover so a single misbehaving agent loop can't produce an unbounded bill. Add a vector database (Qdrant/Weaviate) as a proper stateful, backed-up data store for any RAG component, with hybrid search and a right-sized embedding dimensionality. Instrument observability specifically for LLM/agentic calls — token counts, per-call cost, full prompt/completion, and a tree-structured trace across multi-step agent chains — not just generic APM latency/status codes, so a slow or expensive request can be root-caused to the specific step (a bad retrieval, a failed tool call, an oversized generation) that caused it. The unique DevOps challenges versus a normal microservice: cost that scales with usage in an unbounded way unless deliberately capped, GPU capacity that can be a hard availability constraint rather than just a cost lever, latency that's inherently variable and request-content-dependent rather than roughly constant, and debugging that requires seeing the actual prompt/retrieval/tool-call content, not just status codes and stack traces."
