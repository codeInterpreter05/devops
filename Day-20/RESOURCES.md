# Day 20 — Resources: Phase 0 Review + Mock Interview

## Primary (assigned)

- **Linux troubleshooting scenarios on GitHub (`awesome-linux-resources`)** — a community-curated list of Linux resources, including troubleshooting write-ups and practice scenarios; the assigned starting point for today's review. Search GitHub for `awesome-linux-resources` (or the broader `awesome-linux` family of curated lists) if the exact repo has moved or been renamed — these "awesome" lists are community-maintained and reorganize over time, so treat the search term as more durable than any specific URL.

## Deepen your understanding

- **Brendan Gregg — The USE Method** (brendangregg.com/usemethod.html) — the original write-up of the Utilization/Saturation/Errors framework used throughout this review's troubleshooting methodology. Short, systems-focused, and written by the engineer who popularized it for production performance work.
- **Google SRE Book — "Effective Troubleshooting" chapter** (free at sre.google/books) — the industry-standard reference for structuring a live troubleshooting investigation (hypothesis generation, narrowing, and the discipline of not skipping steps under pressure). Directly complements today's mock-interview walkthrough.

## Reference

- **Dan Luu — postmortem-lessons** (danluu.com) — a collection of patterns pulled from public engineering postmortems across many companies. Useful for building a mental library of "what actually goes wrong in production" beyond the three scenarios in today's lab.

## Practice

- **SadServers.com** — SSH into intentionally broken Linux servers and diagnose/fix real, specific failures for free (with a paid tier for more scenarios). The closest available approximation to today's "given a broken system, diagnose and fix it" activity, run by someone else's scenario design instead of your own.
- **Pramp** (pramp.com) — free, peer-to-peer live mock interview practice with a stranger in the same boat; good for rehearsing the think-aloud/narration skill from today's playbook against a real person instead of a solo timer, since a live audience changes the pressure in ways self-practice can't fully replicate.
