# Day 35 — Quiz: AWS Storage & Databases

Try to answer without looking at your notes. Answers are at the bottom.

1. Once S3 versioning is enabled, can it be fully disabled? What's the actual mechanism for "turning it off"?
2. What does a "delete" actually do to a versioned S3 object under the hood?
3. Why is `NoncurrentVersionExpiration` important specifically for versioned buckets, and what happens if you never configure it?
4. Is S3 cross-region replication retroactive? What tool would you use to replicate objects that existed before a replication rule was enabled?
5. What problem do S3 Access Points solve that a single bucket policy struggles with at scale?
6. What is the fundamental purpose of RDS Multi-AZ, and can you read from the standby under normal operation?
7. What is the fundamental purpose of an RDS read replica, and what kind of replication does it use (sync or async)?
8. If your production RDS primary fails and you only have read replicas (no Multi-AZ), what has to happen to recover, and what's the risk?
9. What's the difference between ElastiCache "cluster mode disabled" and "cluster mode enabled" in terms of what actually scales?
10. What are Redis hash tags for, and why are they needed under cluster mode enabled?
11. What makes a DynamoDB partition key choice "bad," and why doesn't extra provisioned/on-demand capacity fix it?
12. **Interview question:** What is the difference between RDS Multi-AZ and a Read Replica? Can you promote a read replica?

## Answers

1. No — once enabled, versioning can only be **suspended**, not fully disabled. Suspending stops new versions from being created going forward but does not remove or affect any versions already created before suspension.
2. It does not remove the object's data. It adds a **delete marker** — a zero-byte placeholder version that makes the object appear absent from default (non-versioned) listings — while all prior real versions remain fully intact and retrievable by version ID underneath it.
3. Without it, every overwrite or delete of a versioned object leaves the old version in the bucket, consuming storage and being billed indefinitely — the bucket's storage cost grows unbounded over time even though a normal object listing looks unchanged in size. `NoncurrentVersionExpiration` specifically targets and expires old, non-current versions after a configured age, keeping storage cost bounded.
4. No — replication only applies to objects created after the replication rule is enabled; it is not retroactive. To backfill objects that existed beforehand, you need to run **S3 Batch Replication**, a separate, explicit one-time job.
5. A single, shared bucket policy trying to express many different consumers' (teams'/applications') distinct access patterns becomes large, hard to audit, and risky to modify (a change for one consumer risks affecting others). Access Points give each consumer its own named endpoint with its own scoped, independently auditable policy — and can optionally restrict access to a specific VPC.
6. Multi-AZ's purpose is availability: a synchronous, physically-replicated standby in a different AZ that RDS can automatically fail over to (typically in 60-120 seconds) with zero data loss if the primary fails. Under classic Multi-AZ (instance-based), you cannot read from the standby under normal operation — it exists purely as failover insurance.
7. A read replica's purpose is read scaling: an asynchronous, fully queryable copy meant to offload read traffic from the primary. It uses asynchronous replication, meaning there's real (monitorable via `ReplicaLag`) lag between the primary and the replica's data.
8. You must manually run `promote-read-replica`, which detaches the replica and turns it into an independent, standalone writable instance. This is not automatic, takes real time, and because replication to it was asynchronous, any writes that hadn't yet replicated to that replica at the moment of the primary's failure are lost — a real risk that automatic, zero-data-loss Multi-AZ failover does not carry.
9. Cluster mode disabled gives exactly one shard (one primary, up to 5 replicas) — total dataset size is capped by a single node's memory, and replicas exist only for read scaling/HA, not for increasing total data capacity. Cluster mode enabled partitions the keyspace across multiple shards using Redis's hash slot model (16,384 slots via `CRC16(key) mod 16384`), which is what actually lets total data volume and write throughput scale horizontally as shards are added.
10. Hash tags (the `{...}` portion of a key name, e.g., `{user:123}:profile`) let you force multiple related keys to hash to the identical slot. This is needed under cluster mode enabled because multi-key atomic operations (transactions, certain Lua scripts, `MGET` in some clients) only work reliably when all involved keys live on the same shard/slot — without hash tags, related keys could be scattered across different shards, breaking atomicity guarantees for operations spanning them.
11. A partition key is "bad" when it has low cardinality or when a disproportionate share of traffic targets the same key value (e.g., a shared fixed value, or a low-cardinality attribute like a status field shared by millions of items) — DynamoDB hashes the partition key to determine physical placement, so all traffic for that one value lands on a single physical partition. Extra provisioned or on-demand capacity is allocated/spread at the table level, but a single physical partition still has its own hard throughput ceiling — no amount of overall table capacity can push more throughput through one already-saturated physical partition; the fix is a data-modeling change (a higher-cardinality or better-distributed key), not more capacity.
12. Strong answer: "Multi-AZ is an availability feature — a synchronous standby in a different AZ, unreadable under normal operation, that RDS automatically fails over to within roughly 60-120 seconds with zero data loss if the primary fails. A read replica is a read-scaling feature — an asynchronous, always-queryable copy meant to offload read traffic, which does NOT fail over automatically. You can promote a read replica with `aws rds promote-read-replica`, but that's a manual, one-way operation that detaches it into an independent standalone instance; because replication was asynchronous, any not-yet-replicated writes at the moment of a primary failure would be lost, so a promoted replica is a disaster-recovery action with potential data loss and downtime, not an equivalent substitute for Multi-AZ's automatic, lossless failover. In production, I'd run both together — Multi-AZ for availability, one or more read replicas for read scaling — since they solve different problems."
