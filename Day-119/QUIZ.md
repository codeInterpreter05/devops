# Day 119 — Quiz: Zero Trust & mTLS

Try to answer without looking at your notes. Answers are at the bottom.

1. What are the core NIST SP 800-207 zero trust principles, in your own words?
2. What is BeyondCorp, and how is it different from the broader Zero Trust Architecture concept?
3. What happens by default, with zero NetworkPolicy objects applied, to traffic between two pods in different namespaces on a Kubernetes cluster?
4. Why is NetworkPolicy (L3/L4, IP/label-based) not sufficient on its own for strong zero trust guarantees, even when correctly enforced by the CNI?
5. What is a SPIFFE ID, and what is an SVID? Name the two SVID forms and when you'd use each.
6. Describe the two-stage SPIRE attestation chain (node attestation, workload attestation) and explain why workload attestation can't be spoofed by the workload itself.
7. What is SPIFFE Federation, and how is AWS IRSA / GCP Workload Identity Federation conceptually the same pattern?
8. What extra step in the TLS handshake makes mTLS "mutual," and why is a stolen mTLS private key a fundamentally different risk than a stolen bearer token?
9. In Istio, what's the precedence order among mesh-level, namespace-level, and workload-level `PeerAuthentication`, and why does this matter operationally?
10. What is the most common real-world gap in an otherwise "mTLS everywhere" cluster design?
11. Why must a default-deny-egress NetworkPolicy almost always be paired with an explicit DNS egress allow rule?
12. **Interview question:** What is Zero Trust networking? How do you implement it in a Kubernetes cluster?

## Answers

1. No implicit trust based on network location; authenticate and authorize every request based on identity and context, not IP/subnet; enforce least privilege dynamically; design assuming an attacker is already inside (assume breach); verify continuously, not just once at connection time.
2. BeyondCorp is Google's specific, production implementation of zero trust principles for user-to-internal-application access — replacing the corporate VPN with device identity, user identity, and continuous trust scoring evaluated through an access proxy for every request. The broader Zero Trust Architecture (NIST SP 800-207) also explicitly covers workload-to-workload traffic (service mesh, mTLS, workload identity), which BeyondCorp as originally described does not focus on.
3. It's allowed — Kubernetes has no network segmentation by default. Every pod can reach every other pod in every namespace on any port until NetworkPolicy objects are applied, and only if the CNI plugin actually enforces them (Calico/Cilium/Weave do; some minimal/default CNIs silently accept the objects without enforcing anything).
4. NetworkPolicy authorizes traffic based on IP addresses and label selectors resolved to IPs — but pod IPs are ephemeral and recycled, and if an attacker can get a pod scheduled with a matching label (e.g., via compromised CI or over-permissive RBAC on Pod creation), the policy's identity check is satisfied without any real workload compromise being detected. It doesn't verify cryptographic identity or encrypt traffic — it's necessary but not sufficient.
5. A SPIFFE ID is a URI (`spiffe://trust-domain/path`) identifying a workload. An SVID (SPIFFE Verifiable Identity Document) is the cryptographic proof of holding that ID — either an X.509-SVID (a certificate with the SPIFFE ID in the URI SAN, used for mTLS) or a JWT-SVID (a JWT with the SPIFFE ID as the `sub` claim, used where TLS termination isn't practical, e.g., some L7 proxy or serverless scenarios).
6. Node attestation: the SPIRE Agent proves which node it's running on to the Server using a platform-specific unforgeable fact (a Kubernetes projected service account token bound to that node, or a cloud instance identity document), establishing the Agent as a legitimate attestor. Workload attestation: when a local process calls the Workload API, the Agent inspects that process's actual kernel/OS-level attributes (PID → cgroup → container ID → runtime/kubelet metadata) rather than trusting anything the workload claims about itself — the workload would need host-level compromise to forge these facts, which is a categorically bigger problem than SVID theft.
7. SPIFFE Federation lets independent trust domains (separate clusters, clouds, or on-prem environments) exchange trust bundles (root/intermediate CA certs) so a workload in one domain can validate an SVID presented by a workload in another, without merging into a single shared-CA blast radius. AWS IRSA and GCP Workload Identity Federation follow the identical pattern: a cluster's OIDC issuer is registered as a trusted external identity provider in the cloud account, and a projected K8s token is exchanged for short-lived cloud credentials via an STS call — two independent trust domains agreeing to verify each other's tokens instead of sharing a static secret.
8. The server sends a `CertificateRequest` back to the client (instead of the handshake ending after the server presents its own cert), and the client must respond with its own certificate plus a `CertificateVerify` signature proving possession of the matching private key. A stolen bearer token is a value directly reusable by whoever holds it from anywhere; a stolen certificate is useless without the private key, which in a well-designed system never leaves the workload it belongs to (e.g., SPIRE delivers it only over a local Unix socket).
9. Workload-level policy overrides namespace-level, which overrides mesh-level (the `default` `PeerAuthentication` in `istio-system`). This matters because setting a mesh-wide `STRICT` policy doesn't guarantee enforcement everywhere — a more specific namespace- or workload-scoped policy can silently permit plaintext in one spot, and teams often assume the mesh-wide setting is authoritative when it isn't.
10. Non-mesh hops being left out of the mTLS story — most commonly, application-to-managed-database connections secured with TLS-plus-password rather than mutual certificate auth, plus legacy VMs or mesh-less clusters that the service mesh's automatic sidecar mTLS never touches.
11. Because DNS resolution (kube-dns/CoreDNS lookups on UDP/TCP port 53) is itself outbound traffic subject to the default-deny-egress rule; without an explicit allow, every pod in the namespace loses the ability to resolve service/domain names, which typically presents as a generic "connection failed" error rather than an obvious DNS failure, making it a confusing outage to diagnose.
12. Strong answer: "Zero Trust networking means no request is trusted because of where it comes from — not because it's on the internal network, not because it's inside the cluster's pod network — every request is authenticated and authorized on identity and context, continuously, under the assumption that an attacker may already have a foothold somewhere. In a Kubernetes cluster I'd implement this in layers: default-deny NetworkPolicy per namespace with explicit allows (reduces blast radius at L3/L4, though it's IP/label-based and not cryptographically strong on its own); workload identity via SPIFFE/SPIRE or Istio's built-in identity so every workload has a short-lived, auto-rotated, cryptographically verifiable identity instead of static secrets; mTLS enforced everywhere that identity is presented — via Istio `PeerAuthentication` set to STRICT for in-mesh traffic, and cert-manager-issued certificates for anything outside the mesh, like databases or VMs; and L7 `AuthorizationPolicy` rules that authorize specific calls based on the verified peer's SPIFFE identity, not its IP. I'd also make sure RBAC is least-privilege per ServiceAccount, since a compromised pod with an overly broad token undermines all of the above."
