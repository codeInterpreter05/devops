# Day 131-140 — Quiz: GitOps at Scale

Try to answer without looking at your notes. Answers are at the bottom.

1. What are the two parts of every ArgoCD `ApplicationSet`, and what specific mechanism makes it scale to managing hundreds of Applications from one manifest?
2. You register a new cluster into a 50-cluster fleet. Using a Cluster generator with a label selector, what has to change for that new cluster to start receiving the fleet's existing ApplicationSets' Applications? What if you were instead using a hardcoded List generator?
3. Explain what a Matrix generator does, and calculate how many `Application` objects a Matrix of a Git directory generator matching 6 app directories and a Cluster generator matching 40 clusters would render.
4. What specifically happens if an entry disappears from an ApplicationSet's generator source (e.g., a Git directory is deleted, or a cluster Secret is removed)? What setting controls whether this is destructive?
5. How does Rancher Fleet's core architecture (GitRepo → Bundle → BundleDeployment → per-cluster Agent) differ from ArgoCD's Application-centric model, and give one concrete scenario where an org would run both together rather than choosing one.
6. Explain the difference between ArgoCD detecting drift and ArgoCD correcting drift. Which specific `syncPolicy` field controls each, and what command would you run to see the exact field-level delta for an `OutOfSync` Application?
7. Why does a plain Kubernetes `Deployment` with a rolling update strategy structurally lack the ability to do automated canary abort, and what two things does an Argo Rollouts `AnalysisTemplate` need in order to actually gate promotion (not just add a pause)?
8. At 50+ cluster scale, why would you use ApplicationSet's `RollingSync` strategy in addition to (not instead of) an individual Rollout's canary steps within each cluster? What does each layer actually control?
9. Explain, precisely, why "who can merge to this repo" functions as your deployment authorization model in a GitOps setup with `selfHeal: true` enabled — walk through the causal chain from a merged commit to a live cluster change.
10. What's the difference between what an ArgoCD `AppProject`'s `sourceRepos`/`destinations` enforce versus what the Project's `roles`/`policies` enforce? Why do you need both to actually isolate tenants on a shared ArgoCD instance?
11. Why should the ArgoCD service account's Git credential for a config repo be read-only, and what does granting it write access unnecessarily expose you to?
12. **Interview question:** How do you manage GitOps deployments across 100 Kubernetes clusters?

---

## Answers

1. Every ApplicationSet is a **generator** (or generators) plus a **template**. The generator produces a list of parameter dictionaries; the controller renders the template once per entry, substituting `{{key}}` placeholders, and then reconciles the resulting *set* of `Application` objects against what actually exists. The scaling mechanism is that the number of managed Applications is a function of the generator's output, not the number of lines in the ApplicationSet manifest — a fleet of 500 clusters and one of 5 clusters both use the exact same manifest shape.
2. With a Cluster generator + label selector: nothing has to change in the ApplicationSet itself — as long as the new cluster's registration Secret carries the matching labels, the controller's next reconciliation picks it up automatically and renders the Application for it. With a hardcoded List generator: you'd have to manually edit the ApplicationSet's `elements` list to add the new cluster's entry — the entire point of the Cluster generator is eliminating exactly this manual step at fleet scale.
3. A Matrix generator produces the **Cartesian product** of its child generators' outputs — every combination of one entry from each generator. 6 directories × 40 clusters = **240** `Application` objects rendered from one ApplicationSet manifest.
4. The rendered `Application` corresponding to that vanished generator entry gets **deleted** by the ApplicationSet controller on the next reconciliation — and if that Application's own `syncPolicy.automated.prune` is `true`, its deployed cluster resources get cascade-deleted too. The `preserveResourcesOnDeletion` setting (part of `ApplicationSet.spec.syncPolicy` in newer ArgoCD versions) controls whether this cascade actually happens or whether resources are left behind when the Application itself is removed.
5. Fleet's core unit is the **cluster** — a `GitRepo` compiles into a `Bundle`, which fans out into per-cluster `BundleDeployment`s, and an **Agent running inside each downstream cluster** pulls and applies its own BundleDeployment. ArgoCD's core unit is the **Application** (source → destination), typically reconciled by a central controller connecting out to each destination cluster's API (though ArgoCD also supports agent-based/pull patterns). A concrete "run both" scenario: an org uses Rancher for cluster provisioning/lifecycle and Fleet to install the baseline platform layer (namespaces, network policy, monitoring agents) on every cluster the moment it's provisioned, while ArgoCD handles application-level GitOps for business services layered on top of that baseline — different layers, different tools, not a replacement decision.
6. **Detection**: ArgoCD continuously diffs live cluster state against the rendered Git manifests and marks the Application `OutOfSync` if they differ — this happens regardless of any `syncPolicy` setting, it's always-on comparison. **Correction**: whether that detected drift gets automatically fixed depends on `syncPolicy.automated.selfHeal` — `true` means the operator re-applies Git's declared state the moment drift is detected (no waiting for a manual sync); without it, the Application just sits `OutOfSync` until someone runs a manual sync. `argocd app diff <app>` shows the exact field-level delta between live and desired state.
7. A plain rolling-update `Deployment` has no concept of a canary "step" or a metrics query at all — it just replaces pods gradually with no gating logic, so there's nothing to structurally hook an abort decision into. An `AnalysisTemplate` needs (1) a **metric query** against a real signal source (Prometheus/Datadog/CloudWatch/etc., scoped to the specific canary version's traffic, not aggregate service health) and (2) a **`successCondition`/`failureLimit`** that actually gates whether the Rollout proceeds to the next step or aborts — without both, you just have a timed pause with no real signal deciding the outcome, which is not meaningfully different from a slow rolling update.
8. `RollingSync` controls **which clusters** receive an updated Application version and in what order/batch size (e.g., canary-labeled clusters first, 100% at once; standard-tier clusters next, 25% at a time) — it's a fleet-wide wave sequencing mechanism. An individual cluster's `Rollout` canary steps control **how traffic shifts to the new version within that one cluster** (e.g., 20% → pause → analysis → 60% → 100%). You need both because `RollingSync` alone would push a new version instantly and fully to every cluster in a wave with no in-cluster traffic-shaping or metric gating, while per-cluster Rollouts alone (with a plain Cluster generator, no `RollingSync`) would canary every cluster in the fleet simultaneously — a single bad build could still reach all 50+ clusters' canary slices at once instead of being caught on a handful of canary-wave clusters first.
9. With `selfHeal: true`, there is no manual "click deploy" step in the pipeline at all: ArgoCD's controller continuously watches the Git path an Application is configured against, and the moment a commit lands on the watched branch/path (via merge), the next reconciliation (webhook-triggered near-instantly, or on the polling interval otherwise) applies that new state to the live cluster — with no separate approval gate outside Git itself. So the actual authorization question "who can cause a production change" reduces entirely to "who can get a commit merged to that path" — meaning branch protection, required reviews, and CODEOWNERS on the Git repo are not adjacent security hygiene, they are the literal production deployment gate.
10. `sourceRepos`/`destinations` are **scope/blast-radius controls** — they define what any Application assigned to this Project is even *capable* of doing (which repos it may pull from, which cluster/namespace pairs it may target), enforced regardless of who's operating it. `roles`/`policies` are **who-can-do-what-within-that-scope** controls — e.g., which SSO groups can sync vs. only view Applications already inside that Project's boundary. You need both because `roles` alone (with no `sourceRepos`/`destinations` restriction) still lets an authorized user's Application be pointed at another tenant's namespace or an arbitrary external repo; `sourceRepos`/`destinations` alone (with no `roles`) still lets anyone with generic ArgoCD access sync or modify Applications inside that Project even if they shouldn't have tenant-level access at all.
11. ArgoCD only ever needs to **read** from the Git repo it's reconciling against — it never writes back to it. Granting write access serves no functional purpose and directly increases blast radius: if the ArgoCD instance (or its stored credential) is ever compromised, an attacker with only a read-only deploy key can't tamper with the source of truth itself, but one with write access could push malicious commits into the exact repo that drives what's running in every cluster it manages — turning a single compromised component into a compromise of your entire GitOps source of truth.
12. **Worked answer** — structure this as: primitives, then how they compose at scale, then security/tenancy, then tradeoffs.
    - **Cluster registration and labeling first**: every cluster gets registered with ArgoCD (or joins Fleet) with a consistent, meaningful label set (`env`, `region`, `tier`, `provider`) at onboarding time — this is the foundation everything else selects against, and inconsistent labels are the most common reason a fleet-scale setup silently breaks.
    - **ApplicationSet with Cluster/Matrix generators, not hand-maintained Applications**: a Cluster generator filtered by label selector (or a Matrix of a Git directory generator × Cluster generator, for "every app on every matching cluster") means the number of managed Applications scales with the generator's output, not with hand-written YAML — 100 clusters means the same ApplicationSet manifest as 5 clusters, just a bigger selector match.
    - **Git structure drives onboarding**: new clusters/tenants become real via a labeled cluster registration or a new Git directory/file, not a change to any controller-level manifest — this is what keeps "100 clusters" from meaning "100 things a human has to remember to update."
    - **Staged rollout via `RollingSync`**: never push a change to all 100 clusters simultaneously — wave by label (canary tier → standard tier → critical tier), with `maxUpdate` capping how many clusters in a wave move at once, so a bad build is caught on a handful of clusters, not the whole fleet.
    - **Progressive delivery within each cluster**: Argo Rollouts with `AnalysisTemplate`s gates traffic shifting within each cluster on real metrics (scoped per-cluster if metrics infrastructure is per-cluster), giving two independent layers of blast-radius control — which clusters move, and how traffic shifts once they do.
    - **Drift detection and self-healing at scale**: `selfHeal: true` plus `ApplyOutOfSyncOnly=true` (to avoid full-fleet reconciliation cost on every cycle) plus alerting on `argocd_app_info{sync_status="OutOfSync"}` fleet-wide via Notifications, rather than anyone manually watching a UI across 100 clusters' worth of Applications.
    - **Security and multi-tenancy**: repo branch protection/CODEOWNERS is the actual deployment authorization layer; ArgoCD `AppProject`s scope every tenant's `sourceRepos`/`destinations`/`clusterResourceWhitelist`, and instance RBAC (`argocd-rbac-cm`) defaults to read-only with SSO-group-driven role mapping — this is what lets many teams safely share the ArgoCD control plane managing all 100 clusters.
    - **Tradeoffs to state explicitly**: a single ArgoCD control plane managing 100 clusters is an operational scaling concern in its own right (API server load, controller resource limits, `ARGOCD_RECONCILIATION_TIMEOUT` tuning) — at genuinely large scale, some orgs shard into multiple ArgoCD instances (e.g., per region) fronted by a shared ApplicationSet/Git convention, trading a single pane of glass for reduced blast radius and control-plane load per instance; call this out unprompted as the "what changes at 10x this scale" answer.
