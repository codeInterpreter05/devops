# Day 73 — Quiz: Full CI/CD Pipeline Project

Try to answer without looking at your notes. Answers are at the bottom.

1. In a GitHub Actions workflow with `job_b: needs: job_a`, what happens to `job_b` if `job_a` fails? What must you add if you want a job to run anyway (e.g., a notification job)?
2. Why should the `scan` and `sign` jobs operate on the image **digest** rather than the mutable tag produced by `build`?
3. What's the security difference between a push-based deployment model (CI runs `kubectl apply`) and a pull-based GitOps model (ArgoCD reconciles)?
4. In ArgoCD, what's the difference between `syncPolicy.automated` and an empty `syncPolicy: {}`? Which would you use for production and why?
5. What does `selfHeal: true` actually do, and what's a scenario where it surprises an engineer during an incident?
6. Why is per-job `permissions:` scoping (rather than one workflow-wide `permissions:` block) good practice, using `sign` vs. `test` as an example?
7. Why should Slack notifications fire on failure (and recovery) rather than on every run?
8. Name two reasons to use namespace-per-PR for preview environments instead of one shared staging namespace for all open PRs.
9. Why should preview-environment cleanup trigger on `pull_request: closed`, not `pull_request: merged`? What additional safety net should exist regardless?
10. What is the purpose of Kustomize overlays (`base` + per-environment) compared to maintaining separate long-lived Git branches per environment?
11. **Interview question:** Walk me through your complete CI/CD pipeline from git push to production deployment.

---

## Answers

1. By default, `job_b` is **skipped**, not run and not failed — GitHub Actions skips dependents of a failed job unless you override it. To force a job to run regardless of upstream outcome, add `if: always()` (or a more targeted condition like `if: failure()` for a job that should run *only* on failure).
2. Tags are mutable pointers that can be repointed to a different image at any time (including between pipeline steps); the digest is an immutable, content-addressed reference. Scanning/signing by tag leaves a race window where the underlying image content could change between the scan and the sign step (or push and later pull); pinning to digest closes that gap.
3. Push-based CI needs a credential capable of writing to the cluster, meaning cluster-write access is scattered across every CI system/runner that deploys to it. Pull-based GitOps means only the in-cluster controller (ArgoCD) has cluster-write access, and it only ever reads from Git — dramatically shrinking the credential attack surface and making all deployments continuously reconciled against a single Git source of truth.
4. `syncPolicy.automated` makes ArgoCD apply Git changes to the cluster automatically (optionally with `prune`/`selfHeal`). An empty `syncPolicy: {}` means ArgoCD tracks and reports drift (`OutOfSync`) but takes no action until a human runs `argocd app sync` manually. Production should use the manual form so a deliberate, auditable human action gates every production change.
5. `selfHeal: true` makes ArgoCD automatically revert any manual change to a resource it manages back to what's declared in Git, on its next reconciliation. It surprises engineers during an incident when they `kubectl edit` or `kubectl patch` a resource to "quickly fix" something under pressure, only to have ArgoCD silently revert it moments later — the fix has to go through Git, not direct cluster mutation.
6. `permissions:` can be set at the job level, not just workflow level. The `sign` job needs `id-token: write` (for keyless OIDC-based cosign signing) while `test` needs none of that — granting `id-token: write` workflow-wide would give every job (including ones running arbitrary test-dependency code) the ability to mint OIDC tokens, which is unnecessary blast radius.
7. Notifying on every run (including routine successes) creates alert fatigue, and teams mute or ignore the channel within days, which defeats the point of having notifications at all. Notifying only on failure (and on the first success after a failure, i.e., "recovery") keeps the signal meaningful.
8. (a) Isolation — one PR's environment/state can't interfere with another's, and each is torn down independently. (b) Cheap, complete cleanup — `kubectl delete namespace` removes every resource created within it with one command, versus tracking and deleting individual shared-namespace resources for just one PR's changes.
9. A PR can be closed without being merged (abandoned/rejected), and that event still needs cleanup — `merged` alone would miss it, leaving an orphaned environment (and its cloud cost) running indefinitely. Regardless, a scheduled TTL-based cleanup job should exist as a backstop in case the explicit `closed`-triggered cleanup itself fails to run (e.g., a broken workflow file in that same PR).
10. Overlays share one `base/` and each environment overrides only what differs (image tag, replica count, resource limits) — promoting a change is a small, auditable diff in one overlay file. Long-lived environment branches drift over time and require merges/cherry-picks to promote, which risk pulling in unrelated, unintended changes along with the one you meant to promote.
11. Strong answer: "A push to `main` triggers `test`, then `sast` (Semgrep/CodeQL) gated on tests passing, then `build` which produces a container image and outputs its immutable digest. `scan` (Trivy) runs against that digest and fails the pipeline on any critical/high CVE — nothing proceeds past this without a clean scan. `sign` uses cosign in keyless mode (OIDC-based, no long-lived key) to sign that exact digest, and `push` publishes it. From there, CI's job is done — it commits an image-tag bump to a separate GitOps repo's staging overlay, and ArgoCD, running inside the cluster with `automated: {prune: true, selfHeal: true}`, notices the Git change and reconciles staging automatically, typically within a few minutes. Production uses the same overlay pattern but an empty `syncPolicy`, so promotion requires a human to review the diff and run `argocd app sync` deliberately — giving us an audited, deliberate gate between 'works in staging' and 'live in production.' Alongside all this, a `notify` job posts to Slack only when something in the chain fails, with a direct link to the run, and every open PR gets its own namespace-scoped preview environment with a commented URL, torn down on PR close with a scheduled TTL job as a backstop." Follow up with a concrete story about a time this pipeline caught something (a CVE, a bad migration, a broken preview) if you have one.
