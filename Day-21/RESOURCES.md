# Day 21 — Resources: Portfolio & Documentation

## Primary (assigned)

- **GitHub README best practices** ([docs.github.com — About READMEs](https://docs.github.com/en/repositories/managing-your-repositorys-settings-and-features/customizing-your-repository/about-readmes)) — free, the assigned starting point for today. Covers what a README is for on GitHub specifically, how GitHub renders it, and what sections a good one typically includes.

## Deepen your understanding

- **[Make a README](https://www.makeareadme.com/)** — a focused, opinionated guide to what belongs in a project README and why, with a template you can adapt directly.
- **[Awesome README](https://github.com/matiassingers/awesome-readme)** — a curated list of genuinely excellent real-world project READMEs; skim a handful for structure and tone rather than copying content.
- **[MkDocs official documentation](https://www.mkdocs.org/)** — the authoritative reference for `mkdocs.yml` configuration, themes, and the full CLI (`serve`, `build`, `gh-deploy`).
- **[Google Technical Writing courses](https://developers.google.com/tech-writing)** — free, short courses on writing clearly for a technical audience; directly applicable to both README writing and blog posts.

## Reference / lookup

- **[Shields.io](https://shields.io/)** — generate the badge URLs used in a top-level README (build status, license, last commit, etc.).
- **[Choose a License](https://choosealicense.com/)** — plain-language comparison of open-source licenses if you're deciding what to put in `LICENSE`.
- **[gitleaks](https://github.com/gitleaks/gitleaks)** and **[detect-secrets](https://github.com/Yelp/detect-secrets)** — secret-scanning tools to run locally or in CI before anything reaches a public commit.

## Practice

- **[dev.to](https://dev.to)** and **[Hashnode](https://hashnode.com)** — free platforms to publish your technical blog post draft from today's lab, with built-in audience and tagging.
- **The Cloud Resume Challenge** ([cloudresumechallenge.dev](https://cloudresumechallenge.dev/)) — a well-known, free, hands-on project (not required today, but a strong model for what a portfolio-quality, documented, deployed project looks like end-to-end).
- **LinkedIn's own profile-writing guidance** (search "LinkedIn About section tips" via LinkedIn's official help center) — for calibrating the About-section rewrite from file 2 against current LinkedIn norms.
