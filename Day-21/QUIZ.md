# Day 21 — Quiz: Portfolio & Documentation

Try to answer without looking at your notes. Answers are at the bottom.

1. Why does a public GitHub portfolio repo matter more to a hiring manager than a resume bullet claiming the same skills?
2. What's the difference between the "mirror-the-roadmap" and "topic-grouped" repo layout conventions, and when would you pick one over the other?
3. Name three things that make a top-level README actually get read, beyond just prose describing the project.
4. Why should you write `.gitignore` *before* your first commit rather than after?
5. If you accidentally push an AWS access key to a public repo and then delete it in the next commit, is the exposure resolved? Why or why not, and what's the correct remediation?
6. What does `mkdocs gh-deploy` actually do, and what one-time manual step in GitHub's settings is still required after running it?
7. What four-part structure makes a technical blog post readable to a peer audience, and why does "concrete example" carry more weight than prose for this audience?
8. Why is listing "Problem Solving" and "Team Player" in a LinkedIn skills section less effective than listing "AWS," "Kubernetes," or "Terraform," even though both might be true?
9. What's wrong with a per-script folder that has code but no README, even if the top-level repo README is excellent?
10. **Practice exercise (not a scripted interview question — draft your own answer):** Write out, in your own words, a 60–90 second spoken answer to "Walk me through your GitHub portfolio." Use your actual repo, your actual scripts, and your actual blog post — this is meant to be your genuine answer, not a template to memorize.

---

## Answers

1. A resume bullet is a self-reported claim with no way to verify it. A public repo with working scripts, documentation, and commit history is falsifiable evidence — an interviewer can open it, read the code, run a script, and see exactly how you think and work, which is a much stronger and faster signal than any wording on a resume.
2. "Mirror-the-roadmap" organizes folders by day/topic number (`day15-...`, `day16-...`) — good when the repo doubles as a visible learning log and you want reviewers to see progression over time. "Topic-grouped" organizes by function (`aws/`, `kubernetes/`, `linux/`) — better once the repo is meant to read as a general-purpose toolbox rather than a day-by-day journal, or once it outgrows the original roadmap structure. Either is defensible; a flat, ungrouped pile of scripts is the one option to avoid.
3. Badges (build status, license, last commit) that signal active maintenance; a "What this demonstrates" section that explicitly maps folders/scripts to skills instead of making the reviewer infer it; and a table of contents or screenshots/sample output that let a busy reviewer get the gist without reading every line of prose.
4. Because `.gitignore` only prevents files matching its patterns from being staged/tracked *going forward* — it does nothing for files already committed. Writing it first means a secret or cruft file (`.env`, `__pycache__/`) is never accidentally staged in the first place; writing it after the fact means you're relying on remembering to `git rm --cached` anything that slipped through, and if it was already pushed, it's already exposed regardless.
5. No, the exposure is not resolved. Once something is pushed to a public repo, it exists in that commit's history (reachable via `git log`, `git show <sha>`, or anyone who already cloned/forked the repo) even after a later commit removes it from the current file tree. The correct remediation is to treat the credential as compromised immediately and rotate/revoke it at the source (e.g., deactivate/delete the IAM access key and issue a new one, per Day 16) — rewriting history (`git filter-repo`/BFG) only helps for the *current* remote copy, not anything already cloned, so credential rotation is the non-negotiable step.
6. `mkdocs gh-deploy` builds the static site from your `docs/`/`mkdocs.yml` config and force-pushes the built output to a `gh-pages` branch on the remote. The one manual step still required (typically only once) is going into the repo's Settings → Pages and setting the Pages source to build from the `gh-pages` branch — without that, the branch exists but GitHub isn't serving it as a Pages site.
7. Problem/context, explanation, concrete example, takeaway. "Concrete example" (real commands and real output) carries more weight because a technical audience skims prose but reads code blocks closely — an actual terminal capture or diff is verifiable and specific in a way that a paragraph describing the same thing isn't, and it's what lets a reader pattern-match the fix to their own situation.
8. Recruiters search LinkedIn Recruiter by exact keyword/skill tags, not by reading free-form prose — a search for "Kubernetes" or "Terraform" will surface a profile listing those tags and miss one that only lists soft-skill descriptors, even if the candidate is equally strong in both dimensions. Concrete tool names are also things a candidate can be directly quizzed on in an interview, making them a more credible, checkable signal.
9. A missing per-script README means a reviewer who clicks into any individual folder (which they will, since the top-level README should link there) hits an undocumented pile of code with no stated purpose, no usage instructions, and no context — it breaks the impression the polished top-level README created, and suggests the polish was applied only where it's most visible rather than throughout.
10. No single correct answer — this is a practice exercise. A strong answer names the repo's actual layout, picks 2–3 specific scripts to walk through (what problem each solves and why it's built the way it is), mentions the git/documentation discipline behind it (READMEs, no secrets, CI if you built it), and ends by connecting it to a blog post or MkDocs site as further proof of communication skills. The goal is to have said this out loud or written it down at least once *before* an actual interview, so it doesn't come out disorganized under pressure.
