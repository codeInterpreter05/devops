# Day 40 — Quiz: K8s Observability & Debugging

Try to answer without looking at your notes. Answers are at the bottom.

1. What is almost always the correct first command when investigating any broken pod, and why?
2. What does `kubectl logs --previous` do, and why is it the single most-forgotten useful flag for a crash-looping pod?
3. Why does `kubectl exec -it <pod> -- /bin/sh` sometimes fail with "executable file not found," and what's the modern fix?
4. What is the default TTL for Kubernetes Events, and what's the practical consequence of that TTL for post-incident investigation?
5. Is `CrashLoopBackOff` itself a root cause? What is it actually describing?
6. What do exit codes 137, 139, and 143 each indicate?
7. What's the difference between `requests.memory` and `limits.memory`, and which one does the kernel actually enforce to trigger an OOM kill?
8. How do you distinguish "memory limit is genuinely too low" from "there's a real memory leak" when a container keeps getting OOMKilled?
9. Name three distinct root causes of `ImagePullBackOff` and how their Events messages typically differ from each other.
10. Why can `kubectl logs <pod>` (no flags) show nothing useful when a pod is stuck in `Init:Error`?
11. What's a scenario where a `Pending` pod is actually expected/healthy rather than broken?
12. **Interview question:** A pod is stuck in CrashLoopBackOff. Walk me through your complete debugging process.

---

## Answers

1. `kubectl describe pod <name>`. It aggregates container statuses (state, last state, exit codes, restart count), resource configuration, and — most importantly — the Events section, which is a chronological record of what the scheduler/kubelet actually observed (failed pulls, probe failures, OOM kills, scheduling failures). Most real issues are diagnosable, or at least correctly categorized, from this one command alone.
2. It retrieves logs from the **previously terminated** container instance, not the current (freshly restarted) one. It's the most-forgotten flag because a crash-looping container's *current* instance may have crashed again almost instantly with little output, while the actual error (stack trace, exception) sits in the previous instance's logs.
3. Minimal or distroless production images intentionally ship with no shell and no shell utilities for security/size reasons, so there's nothing for `exec` to run. The modern fix is `kubectl debug -it <pod> --image=busybox --target=<container>`, which injects an ephemeral container with real debugging tools into the running pod without requiring the app's own image to have a shell.
4. The default Events TTL is about 1 hour (`--event-ttl` on the kube-apiserver). Practically, this means investigating an incident more than roughly an hour after it happened will often find no events at all, even though they clearly existed at the time — persistent, externally-shipped logging/eventing is required for later post-incident analysis.
5. No — `CrashLoopBackOff` is a status describing the kubelet's exponential-backoff restart behavior after a container has exited repeatedly; it says nothing about *why* the container exited. The actual cause requires further investigation (`describe`'s exit code/Reason, `logs --previous`).
6. 137 = `128+9`, killed by SIGKILL — almost always an OOM kill or a forced kill (e.g., a failed liveness probe past its threshold). 139 = `128+11`, SIGSEGV — a segmentation fault, usually a lower-level/native-code bug. 143 = `128+15`, SIGTERM — a graceful shutdown request, typically normal termination (scale-down, rolling update) rather than a crash.
7. `requests.memory` is used only by the scheduler to decide which node has room for the pod (bin-packing) — it does not cap actual usage. `limits.memory` is the hard ceiling enforced by the Linux kernel's cgroup memory controller; exceeding it triggers an OOM kill regardless of how much memory the node has free elsewhere.
8. Look at the memory usage pattern over the container's uptime (via `kubectl top` sampled over time, or real metrics in Prometheus/Grafana). A pattern that rises and stabilizes under a ceiling, only getting OOMKilled during traffic spikes, suggests the limit is genuinely too tight for peak load. A steady, unbroken upward climb that always ends in an OOM kill (and repeats the same climb after restart) is the signature of an actual memory leak that no limit increase permanently fixes.
9. (a) Typo/nonexistent image tag — Events shows a "not found" style error; (b) missing or incorrect `imagePullSecrets` for a private registry, or expired registry credentials (e.g., ECR's 12-hour token expiry) — Events shows an "unauthorized"/"403" style error, distinct from "not found"; (c) blocked network egress to the registry (firewall/NetworkPolicy/security group) — Events typically shows a timeout rather than an auth or not-found message, which is the tell distinguishing this cause from the other two.
10. Init containers run sequentially and must all complete successfully before any main container ever starts. If an init container is stuck or failing, the main application container never even attempts to start, so `kubectl logs <pod>` (which without `-c` targets a specific default container, often assumed to be the main app) shows nothing relevant — the useful information is in `kubectl logs <pod> -c <init-container-name>` instead.
11. In a cluster with a cluster autoscaler, a pod can be legitimately `Pending` while a new node is being provisioned to accommodate it — this is expected, transient behavior, not a failure, and should be confirmed via `kubectl get nodes` and autoscaler logs/status before treating it as an incident.
12. Strong answer: "First, `kubectl get pods` to confirm the status and check the restart count — a high, still-climbing count confirms an active loop. Then `kubectl describe pod <name>`, specifically the container's Last State, Reason, and Exit Code, plus the Events section for anything Kubernetes itself observed, like an OOM kill or failed liveness probe. Exit code 137 immediately points me toward an OOM kill or a forced kill from a probe; 139 points to a segfault. Next, `kubectl logs <name> --previous` to get the actual application-level error from the terminated instance, since the fresh restart's logs are often unhelpfully sparse. I'd also check resource requests/limits (is this an OOM kill from a limit that's too low, or a real leak — I'd look at the usage trend, not just the fact that it happened once) and readiness/liveness probe configuration, since an overly aggressive probe can make Kubernetes itself kill an otherwise-healthy, slow-starting container, which looks identical to a crash from `kubectl get pods` alone. Finally, if none of that resolves it, I'd check external dependencies the app needs at startup (database, config, secrets) and, if possible, reproduce the exact same image/config outside Kubernetes to isolate whether the problem is the application itself or something Kubernetes-specific."
