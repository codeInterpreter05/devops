# Day 52 — Quiz: AWS Messaging Deep Dive

Try to answer without looking at your notes. Answers are at the bottom.

1. Why do SQS standard queues offer only best-effort ordering and at-least-once delivery instead of exactly-once/strict order?
2. What is a visibility timeout, and how does it explain both SQS's reliability guarantee and why duplicate processing happens?
3. What does a Dead-Letter Queue actually do, and what breaks without one?
4. In an SNS fan-out pattern, why publish to SNS and let it distribute to multiple SQS queues, rather than having the producer publish directly to each queue?
5. What's the fundamental difference in how EventBridge routes events compared to how SNS routes messages?
6. Why do GuardDuty findings (and most native AWS service events) show up on the EventBridge default bus without any producer-side "publish" call?
7. What is the single biggest structural difference between Kinesis Data Streams and SQS in terms of what happens when a consumer reads a record/message?
8. What determines which shard a Kinesis record lands on, and why does that matter for ordering?
9. When would you choose MSK over Kinesis Data Streams for a new AWS-native streaming workload?
10. Give the one-sentence mental model distinguishing RabbitMQ, SQS, and Kafka.
11. A team needs to process uploaded files exactly once each, doesn't need any historical replay, and wants zero infrastructure to manage. Which service fits, and why would Kafka be overkill here?
12. **Interview question:** Compare RabbitMQ, SQS, and Kafka. When would you choose each for a distributed system?

---

## Answers

1. SQS standard queues replicate messages across multiple servers for redundancy and availability; that distributed architecture makes strict global ordering and exactly-once delivery expensive to guarantee without sacrificing throughput, so SQS documents best-effort ordering and at-least-once delivery, pushing idempotent processing responsibility onto the consumer in exchange for near-unlimited throughput.
2. Visibility timeout is the window during which a received-but-not-yet-deleted message is hidden from other consumers. If the consumer deletes the message within that window, it's gone for good (reliability). If the consumer crashes or times out before deleting it, the message becomes visible again and gets redelivered — which is exactly why the same message can be processed twice if the consumer finished the work but crashed before calling `DeleteMessage`.
3. A DLQ is a second queue that automatically receives messages once they've exceeded `maxReceiveCount` failed processing attempts (per the redrive policy). Without one, a single malformed/poison message loops forever — reappearing every visibility timeout, consuming worker capacity indefinitely, and burying real logs under the same repeated failure, without ever being isolated for out-of-band investigation.
4. Publishing to SNS decouples the producer from the evolving set of consumers — adding a new consumer later is just subscribing a new queue to the topic, with zero changes to the producer. Each SQS queue also gives its consumer independent durability, pacing, and retry/DLQ behavior, so a slow or temporarily-down consumer doesn't affect the others.
5. SNS routes based on **topic subscription**, optionally narrowed by a filter policy matched against message attributes. EventBridge routes based on matching the actual event's **content/structure**, including deep matches into the JSON `detail` payload, across roughly 200 native AWS event sources plus SaaS partners, without requiring the producer to explicitly publish to it.
6. Because EventBridge natively ingests events from AWS service actions on its default bus — AWS services push events there as part of their own operation, unlike SNS where a producer must explicitly call `Publish`. This is why GuardDuty, EC2 state changes, and many other AWS service events appear on the bus with zero extra plumbing.
7. In Kinesis, reading a record via `GetRecords` does **not** remove it from the stream — data is retained per a time-based policy regardless of how many consumers have read it, and multiple independent consumers each track their own position/checkpoint. In SQS, receiving and then deleting a message removes it from the queue — it's a claim-and-delete model built around one logical consumer per message.
8. The record's **partition key** determines which shard it's routed to; records with the same partition key land on the same shard and preserve relative order for that key. This matters because ordering in Kinesis is a structural, per-shard property — you only get ordering guarantees within a partition key/shard, not across the whole stream.
9. Choose MSK when you need actual Kafka API/ecosystem compatibility — existing Kafka Connect connectors, Kafka Streams topologies, consumer group semantics, log compaction, or a team that already has deep Kafka expertise and wants to lift-and-shift without rewriting. Choose Kinesis for a simpler, more AWS-native operational model with tighter IAM integration when you don't need Kafka-specific tooling.
10. RabbitMQ is a feature-rich traditional message broker best for complex routing and moderate throughput; SQS is the simplest fully-managed queue for basic decoupling with no replay needs; Kafka is a distributed log built for high-throughput, multi-consumer, replayable event streaming.
11. SQS (standard) fits perfectly — a simple work-queue problem with no need for multiple independent readers or historical replay, and it requires zero infrastructure management. Kafka would be overkill because its value (multi-consumer replay, very high throughput, rich ecosystem) isn't needed here, while its operational complexity (partitions, consumer groups, cluster/broker management even when managed via MSK) would be pure overhead for a simple exactly-once file-processing task.
12. Strong answer: "They sit at genuinely different points in the messaging design space. RabbitMQ is a smart-broker, traditional message broker with rich routing logic built into exchange types (direct, topic, fanout, headers) — I'd choose it for complex routing needs like RPC-style request/reply or priority queues, at moderate throughput. SQS is the simplest fully-managed queue — messages are claimed and deleted once processed, at-least-once delivery on standard queues, exactly-once and strict ordering per group on FIFO queues — I'd choose it for straightforward work-queue decoupling where I don't need replay and want zero operational burden. Kafka is a distributed commit log rather than a queue — reading doesn't remove data, consumer groups independently track their own offsets, and replay is a first-class operation — I'd choose it (or MSK on AWS) for high-throughput event streaming where multiple independent systems need to consume the same data, potentially reprocessing historical data, like clickstream analytics, change-data-capture, or event sourcing. The deciding factor is usually: do I need multiple independent consumers to replay the same data at high throughput (Kafka), do I need complex broker-side routing (RabbitMQ), or do I just need simple, reliable point-to-point decoupling with minimal ops (SQS)."
