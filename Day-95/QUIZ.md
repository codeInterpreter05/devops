# Day 95 — Quiz: Jaeger & Tempo

Try to answer without looking at your notes. Answers are at the bottom.

1. Name Jaeger's core components and what each is responsible for (write path vs. read path vs. storage).
2. Why is jaeger-agent considered legacy, and what typically replaces it in a modern deployment?
3. What are the storage backend options for Jaeger, and what's the key operational tradeoff between Cassandra/Elasticsearch and Tempo?
4. Why is Tempo architecturally similar to Loki? Name the four main Tempo components in write-path order.
5. What is TraceQL, and why is "find by trace_id" fundamentally cheaper/easier for Tempo than "find by arbitrary tag"?
6. Describe the three correlation links that make up "the three pillars" workflow in Grafana (metric → trace → log, and back).
7. What does a trace waterfall look like when a request suffers from the N+1 query problem? Why is this pattern invisible in a DB latency metrics dashboard?
8. What does RED stand for, and why should "Duration" be tracked as a percentile distribution rather than an average?
9. What is a metrics-generator / spanmetrics connector, and what's the catch when it sits downstream of a tail-sampling processor?
10. **Interview question:** A microservice has high p99 latency. Walk me through using distributed tracing to find the root cause.
11. Why should you look at multiple slow traces rather than just one before concluding a root cause?
12. What's one concrete reason to choose Jaeger-on-Elasticsearch over Tempo, and one concrete reason to choose Tempo over Jaeger?

## Answers

1. jaeger-collector (write path — receives and validates spans, writes to storage backend, horizontally scalable), jaeger-query (read path — translates UI/API queries into storage-backend queries), jaeger-ui (frontend — waterfall view, service dependency graph), and the pluggable storage backend (durability layer).
2. jaeger-agent was a UDP-based per-host daemon for receiving spans locally before forwarding to the collector; UDP is unreliable and size-limited. Modern deployments have applications/OTel SDKs send OTLP directly to jaeger-collector, or route through an OTel Collector first, skipping jaeger-agent entirely.
3. Options: Cassandra (battle-tested at very large scale, write-optimized, but you now operate a Cassandra cluster), Elasticsearch/OpenSearch (flexible tag search, good if you already run ELK for logs), in-memory (dev/lab only, no persistence), Badger (embedded, small deployments), or migrating to Tempo. The tradeoff: Cassandra/ES give full secondary indexing (query by any tag) at the cost of operating a stateful, sharded database cluster; Tempo drops most of that indexing to store data in cheap object storage with far simpler operations.
4. Both are built by Grafana Labs on the same philosophy: cheap object storage plus a lightweight ID-based index, instead of a full secondary-indexed database. Tempo's write path: distributor (hash spans by trace_id) → ingester (buffer, flush completed blocks) → object storage (S3/GCS/etc., the actual durable store) → (compactor merges blocks/enforces retention in the background).
5. TraceQL is Tempo's PromQL/LogQL-style query language for searching traces by attributes/duration/status, added to close the gap left by Tempo's ID-lookup-first design; it's still scan-cost-bound over the time range searched, not a full secondary index. "Find by trace_id" is cheap because Tempo only needs a lightweight mapping from trace_id to its object storage block location, rather than maintaining a full distributed index over every possible tag combination.
6. (1) Metric → Trace: a Prometheus/Mimir histogram exemplar carries a sampled trace_id, clickable straight into Tempo. (2) Trace → Logs: given a trace_id, Grafana auto-builds a Loki query to find log lines from that span (requires the app's logs to include trace_id, e.g. via OTel's log integration). (3) Logs → Trace: a log line containing a trace_id field renders as a clickable link back into the corresponding Tempo trace.
7. Many sibling spans with the same operation name and similarly short individual durations, stacked sequentially (or repeated) under one parent span, with the count scaling with input size (e.g., one DB span per row returned). It's invisible in a DB metrics dashboard because each individual query genuinely is fast (p99 for the query type looks fine) — the problem is the sheer count of them per request, which only the trace's parent-span view exposes.
8. RED = Rate (requests/sec), Errors (error rate/fraction), Duration (latency). Duration should be percentiles (p50/p95/p99) because averaging latency (especially averaging pre-aggregated per-host averages) hides tail latency and is mathematically meaningless when combined across hosts — percentiles require the actual distribution (histogram), not just a mean.
9. It's a component (Tempo's metrics-generator, or the OTel Collector's spanmetrics connector) that aggregates the span stream into RED-style metrics automatically, without hand-instrumenting separate metrics. The catch: if it runs after a tail-sampling processor that drops most traces, the derived metrics (like request rate) will undercount reality proportional to the sampling rate, unless you place the generator before sampling or account for the ratio.
10. Strong answer: first check the service's RED dashboard to confirm scope (is p99 actually elevated for this service, and is error rate also up); pull several representative slow traces from the tracing backend for the affected endpoint/time window (not just one, to avoid outlier bias); read the waterfall to find which child span consistently dominates duration across those traces; check specifically for an N+1 pattern (many small repeated spans vs. one large slow span); jump from the dominant span to correlated logs for exact error detail; determine whether the slow span is inside your own service or a downstream dependency (routes the investigation to the right owner); and correlate the timing of the latency increase with recent deploys/config/feature-flag changes.
11. A single trace could be an outlier (e.g., a one-off GC pause, a cold cache, a transient network blip) unrelated to the systemic issue; looking at multiple slow traces confirms the bottleneck span is a consistent pattern rather than a coincidence, and reduces the risk of chasing the wrong root cause.
12. Choose Jaeger-on-Elasticsearch when you need rich, flexible ad-hoc tag search and already operate ES for logs (reusing operational investment). Choose Tempo when you want to avoid running a dedicated stateful trace database at all, minimize storage cost at scale via object storage, and want native Grafana-based cross-signal correlation (exemplars, trace-to-logs) out of the box.
