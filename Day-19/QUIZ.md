# Day 19 — Quiz: Performance & Debugging

Try to answer without looking at your notes. Answers are at the bottom.

1. What do the three numbers in `load average` represent, and over what time windows?
2. Why is a load average of `8` meaningless on its own? What additional piece of information do you need before judging it?
3. A system shows `free -h` output of `used: 14Gi / 15Gi total` — why can this be perfectly healthy? Which column should you actually watch instead of `used`?
4. What do `vmstat`'s `r` and `b` columns tell you, and how do they help you disambiguate what a high load average actually means?
5. What's the difference between `vmstat`'s `swpd` column and its `si`/`so` columns? Why doesn't a nonzero `swpd` necessarily indicate an active problem?
6. What does `strace -c` show you, and why is it usually the first `strace` invocation to run against an unfamiliar slow process?
7. Describe the specific syscall pattern (as it would appear in raw `strace` output) that's the signature of unbuffered, byte-at-a-time file I/O.
8. Beyond simply "making things slower," why can `strace`'s own tracing overhead actively mislead a debugging session?
9. What's the fundamental difference between what `strace` and `ltrace` each intercept? Why can `ltrace` produce useless output against a statically linked Go binary while `strace` still works fine on it?
10. Explain the USE method: what do Utilization, Saturation, and Errors each mean, and why is Saturation often the more urgent of the two numeric signals?
11. What does the `%steal` column in `sar -u` indicate, and in which kind of environment does it actually matter?
12. **Interview question:** A server is slow. Walk me through your systematic diagnosis from CPU → memory → disk → network.

## Answers

1. They're exponentially-decayed moving averages of the number of processes that are either **runnable** (waiting for CPU) or in **uninterruptible sleep** (`D` state, usually blocked on I/O), averaged over the last **1, 5, and 15 minutes** respectively.
2. Load average is not a percentage and has no inherent ceiling — its significance depends entirely on how many CPU cores the box has. A load of `8` is fully saturating on a 4-core box but only 50%-loaded on a 16-core box. You need `nproc` (or `/proc/cpuinfo`/`lscpu`) before the number means anything.
3. Linux uses otherwise-idle RAM as page cache (`buff/cache`) to speed up future disk reads — that memory is reclaimed instantly under pressure, so a high `used`/`buff/cache` combination is the kernel doing its job, not a leak. The column to actually watch is `available`, which already accounts for reclaimable cache and estimates what's usable by new processes without swapping.
4. `r` counts processes currently runnable and competing for CPU right now; `b` counts processes blocked in uninterruptible sleep, almost always waiting on I/O. If a load-average spike is paired with high `r` (well above core count), the bottleneck is CPU; if it's paired with high `b`, the bottleneck is I/O (disk, NFS, etc.) even though load average alone can't tell you which.
5. `swpd` reports how much swap space is currently occupied — this can be nonzero and completely harmless if something was swapped out long ago and simply never needed since. `si`/`so` (swap in/swap out per second) reflect *active* swapping happening right now — sustained nonzero values there mean the system is genuinely out of RAM and paging to disk, a real performance problem, whereas nonzero `swpd` alone is not.
6. `strace -c` aggregates syscalls into a summary table showing call count, total time, and errors per syscall, without producing a full line-by-line trace. It's the first command to run because it answers "which syscall dominates" in one shot — e.g., a huge `read` count immediately points at unbuffered/byte-at-a-time I/O — before you invest time drilling into a full trace.
7. Something like `read(3, "a", 1) = 1` repeated an enormous number of times, one line per byte, all against the same file descriptor — each call requesting and returning exactly 1 byte instead of a program issuing far fewer calls for large buffered chunks.
8. `strace` attaches via `ptrace`, which stops the traced process at every syscall entry/exit — this overhead (often 2x-20x) doesn't just make the program run slower overall, it can change the *relative* timing between threads/processes enough to mask a race condition that normally reproduces, or introduce artifacts (e.g., timeouts firing or not firing) that wouldn't happen untraced. Absolute timing captured under `strace -T` should be treated as approximate for this reason.
9. `strace` intercepts calls at the kernel boundary (syscalls) — something every program does regardless of language or how it's compiled/linked. `ltrace` intercepts calls into dynamically linked shared libraries by hooking the dynamic linker's symbol resolution (PLT/GOT). A statically linked binary (common for Go binaries, `musl`-static builds, and many minimal/distroless container images) has no such dynamic-linking indirection for `ltrace` to hook into, so it reports nothing useful — but it still makes syscalls, which is why `strace` keeps working regardless of linking.
10. **Utilization** is the percentage of time a resource was busy servicing work (or percentage of capacity used). **Saturation** is the degree to which a resource has extra work queued that it can't service right now. **Errors** are the count of error/retry/drop events for that resource. Saturation is often the more urgent signal because a resource can sit at 100% utilization with zero backlog (perfectly efficient, no problem) or be saturated even below 100% utilization (bursty arrivals queuing up) — saturation means work is actively being delayed *right now*, which is closer to what users actually feel.
11. `%steal` is the percentage of time a virtual CPU wanted to run but the hypervisor gave the physical CPU to something else instead (another tenant/VM). It only applies in virtualized/cloud environments — a nonzero, rising `%steal` on a "slow" cloud instance points at contention on the underlying host, not at anything wrong with your workload's own code.
12. Strong answer: "First I'd confirm the time window — is it slow now (live tools) or was it slow at a specific past time (`sar` replaying `/var/log/sa/`)? Then I go resource by resource, cheapest first. **CPU**: check `nproc` for core count, `uptime` for load average, then `vmstat 1 5`'s `r`/`b` columns to see whether processes are runnable (CPU-bound) or blocked (I/O-bound) — and `mpstat -P ALL` to catch a single-threaded hot core hiding behind a healthy aggregate. **Memory** next, because swapping can make everything else look slow as a side effect: `free -h`'s `available` column, and `vmstat`'s `si`/`so` for active swapping, plus checking `dmesg` for OOM kills. **Disk** next, especially if `b` was elevated: `iostat -x 1 5` for `%util` and queue depth (`avgqu-sz`/`aqu-sz`), correlated with the blocked processes I already saw — and if disk is implicated, I'd use `strace -e trace=read,write,openat` on the specific process to find which file it's stuck on. **Network** last, since it's more often the cause of slow *calls to a dependency* than a single box feeling broadly sluggish: `ss -s` for retransmits, `sar -n DEV` for throughput, `ip -s link show` for drop/error counters. At each resource I'm applying the USE method — utilization, saturation, *and* errors, not just utilization — because a resource can degrade performance through errors or queuing even while looking fine on raw busy-percentage. And I'd treat this ordering as a sensible default, not a rule — if the reported symptom already points somewhere specific (e.g., 'only slow on file uploads'), I'd start there instead."
