# Day 77 — Quiz: Multi-cloud CI/CD

Try to answer without looking at your notes. Answers are at the bottom.

1. In a multi-cloud CI/CD pipeline, why should you build the artifact once and deploy the same digest to every cloud, rather than rebuilding per cloud?
2. What do `aws-actions/configure-aws-credentials` (with `role-to-assume`) and `google-github-actions/auth` (Workload Identity Federation) have in common as a security pattern, and why does it matter more as you add cloud providers?
3. What does a Terraform workspace actually isolate? What does it NOT do, despite a common misconception?
4. What mechanism actually lets one Terraform configuration manage resources across multiple cloud providers?
5. Give a concrete reason a team might prefer separate root modules per environment over heavy `terraform.workspace ==` conditionals in a multi-cloud setup.
6. Why is "avoiding vendor lock-in" often a weak justification for adopting multi-cloud as company-wide policy?
7. List three concrete, ongoing (not one-time) costs of running genuine multi-cloud infrastructure.
8. Name two circumstances where multi-cloud is a legitimate, well-justified architectural choice.
9. What is "cloud repatriation," and what kind of workload is the classic candidate for it?
10. Why doesn't "we run Kubernetes so we're cloud-agnostic" fully solve the multi-cloud portability problem?
11. **Interview question:** What are the real costs and benefits of a multi-cloud strategy? When is it justified?

---

## Answers

1. Building once and deploying the identical digest everywhere guarantees byte-for-byte the same artifact is running on every cloud, eliminating any risk of subtle behavioral drift that could arise if separate builds (even from the same source) somehow differ (different base image layer caching, timing-dependent build steps, etc.). It's also more efficient — you don't pay the build cost N times for N cloud targets.
2. Both are OIDC-based, workload-identity federation mechanisms that let CI obtain short-lived, scoped credentials for a specific cloud without ever storing a long-lived static credential (an IAM access key or a service-account JSON key) in CI secrets. This matters more as you add cloud providers because without it, you'd be accumulating a growing pile of long-lived, high-privilege static secrets — one per cloud — each an independent, permanent attack surface.
3. A workspace isolates **state** — it lets the same `.tf` configuration be applied against multiple separate, named instances of infrastructure (e.g., dev/staging/production of the same shape), each tracked in its own state file. It does NOT switch which providers or resource types are used — a common misconception is trying to use "one workspace per cloud," which doesn't work because workspaces have no built-in mechanism to conditionally include/exclude provider blocks or resources based on the active workspace.
4. Declaring multiple `provider` blocks (e.g., both `provider "aws" {}` and `provider "google" {}`) within the same Terraform configuration, with resources for each cloud declared alongside each other and tracked in one state file via one `plan`/`apply`.
5. Heavy `terraform.workspace == "x"` conditionals scattered through resource definitions become hard to read and easy to get wrong (forgetting to add a conditional to a newly added resource). Separate root modules per environment, each calling shared modules with environment-specific inputs, make the actual differences between environments explicit and reviewable in each environment's small entrypoint file, at the cost of a bit more directory structure.
6. Because in practice, most organizations that cite "avoiding lock-in" as their reason for multi-cloud never actually shift meaningful workloads or spend between providers in response to pricing or leverage opportunities — the benefit is largely theoretical — while the costs (doubled operational complexity, forfeited volume discounts from splitting spend, cross-cloud egress fees) are real, continuous, and start accruing immediately.
7. Any three of: roughly doubled on-call/engineering cognitive load (different IAM, networking, and failure models per cloud); cross-cloud data-transfer/egress costs; forfeited volume/committed-use discounts from splitting spend across providers instead of concentrating it; duplicated tooling and observability configuration (parsers, dashboards, alerting rules) per cloud even when using nominally "multi-cloud" tools.
8. Any two of: a genuine regulatory/data-residency requirement mandating specific providers in specific jurisdictions; an M&A situation where you've inherited infrastructure on a different cloud and immediate full migration isn't worth the disruption; a narrow, bounded need for one cloud's genuinely superior best-of-breed service for a specific workload; catastrophic-risk hedging for a small set of truly critical systems where a full provider-level outage would be existential.
9. Cloud repatriation is moving workloads back from public cloud to on-premises/colocated infrastructure (or consolidating from multiple clouds down to one), driven by the realization that steady-state, pay-as-you-go cloud costs can exceed owned/colocated infrastructure costs over time for certain workloads. The classic candidates are workloads with stable, predictable, high-volume usage — commonly cited examples are large-scale storage- and bandwidth-heavy workloads — not variable/bursty workloads, which still benefit from cloud elasticity.
10. Kubernetes makes the workload/compute layer (Deployments, Services, application manifests) largely portable across clouds, but cluster provisioning, storage classes, load-balancer/ingress integration, and IAM-to-Kubernetes-RBAC bridges (e.g., IRSA on EKS vs. Workload Identity on GKE) still differ meaningfully per cloud and require explicit, cloud-specific handling — the portability is real but partial, not total.
11. Strong answer: "The benefits typically cited are avoiding vendor lock-in, negotiating leverage, and resilience against a full provider outage — but in practice, the costs are continuous and structural: roughly double the on-call cognitive load since every cloud has different IAM, networking, and failure modes; real cross-cloud egress costs; forfeited volume/committed-use discounts since spend gets split instead of concentrated; and duplicated tooling and observability configuration even when using nominally cloud-agnostic tools. I'd only justify multi-cloud for a specific, bounded reason — a genuine regulatory data-residency requirement, an M&A-inherited situation, a narrow best-of-breed need for one specific service, or catastrophic-risk hedging for a small number of truly critical systems — not as a blanket company-wide policy for vague lock-in-avoidance, because that ongoing tax is rarely offset by benefits that, in most organizations I've read about or seen, are never actually exercised in practice." Follow with a concrete example if you have one — even the "build once, deploy to AWS and GCP in parallel" exercise from today's lab is a legitimate small example to reference.
