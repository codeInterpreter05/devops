# Day 51 — Quiz: K8s Upgrades & Cluster Management

Try to answer without looking at your notes. Answers are at the bottom.

1. Why must the control plane always be upgraded before the worker nodes, never the other way around?
2. Can you skip Kubernetes minor versions during an EKS control plane upgrade (e.g., go straight from 1.27 to 1.29)?
3. Besides the control plane and nodes themselves, what else commonly needs a version-compatibility check during an upgrade, and what breaks if you skip it?
4. What's the difference between a rolling in-place node group upgrade and a blue/green node group upgrade? Which one gives you an easier rollback?
5. What does `kubectl cordon` do, and specifically what does it *not* do?
6. Why does `kubectl drain` use the Eviction API instead of just deleting pods, and what practical difference does that make?
7. What do `--ignore-daemonsets` and `--delete-emptydir-data` do, and why are they needed for most real drains?
8. A PodDisruptionBudget sets `minAvailable: 3` on a Deployment with exactly 3 replicas. What happens when you try to drain a node hosting one of those pods, and is this a bug?
9. What's the difference between a "voluntary" and "involuntary" disruption, and which one does a PodDisruptionBudget actually protect against?
10. What does `pluto` do, and what's the practical difference between its `DEPRECATED` and `REMOVED` findings?
11. What does Velero back up, and what does it explicitly not replace as a disaster-recovery mechanism?
12. **Interview question:** How do you upgrade an EKS cluster with zero downtime and minimal risk?

---

## Answers

1. Kubernetes' version skew policy requires kubelets to be at the same version as, or some bounded number of minor versions behind, the control plane — never ahead of it. If nodes were upgraded first, kubelets could end up speaking a newer API dialect than the control plane supports, causing incompatibilities; EKS enforces this by refusing to let a node group upgrade past the current control plane version.
2. No — EKS control plane upgrades must proceed one minor version at a time (e.g., 1.27 → 1.28 → 1.29); you cannot skip directly from 1.27 to 1.29 in a single operation.
3. Add-ons: VPC CNI, CoreDNS, kube-proxy, and CSI drivers (EBS/EFS). If left outdated after a control plane bump, symptoms include new pods stuck unable to get networking (outdated CNI) or DNS resolution failures — issues that look unrelated to "the upgrade" but are a direct consequence of it.
4. A rolling in-place upgrade replaces nodes within the existing node group directly, modifying live infrastructure with no separate fallback set of old nodes once the process starts. A blue/green node group creates an entirely new node group at the target version alongside the old one, migrates workloads over gradually, and only decommissions the old node group after verifying health — giving you a trivial rollback (just don't cut over, or scale the old group back up) if something's wrong with the new one.
5. `kubectl cordon` marks a node `Unschedulable`, meaning the scheduler won't place any new pods there. It does **not** affect pods already running on that node — they keep running untouched until something else (like `drain`) acts on them.
6. The Eviction API subresource explicitly checks any applicable PodDisruptionBudget before allowing a pod to be removed — if eviction would violate the PDB, the request is rejected (retried by `drain` until it's safe or it times out). A plain `kubectl delete pod` bypasses this check entirely and terminates the pod regardless of any PDB, which is why manual deletion should never substitute for `drain` during planned maintenance.
7. `--ignore-daemonsets` skips DaemonSet-managed pods during drain, since they're tied to that specific node and can't meaningfully be "evicted" elsewhere — without it, drain would otherwise fail/block on pods it can never successfully relocate. `--delete-emptydir-data` explicitly acknowledges that pods using `emptyDir` volumes will lose that (inherently ephemeral, node-local) data on eviction, which is expected but requires explicit confirmation.
8. The drain will hang/fail on that pod — the PDB requires all 3 replicas to remain available at all times, so evicting even one pod (which would drop availability to 2) is refused by the Eviction API. This is not a bug; it's the PDB doing exactly what it was configured to do. The fix is either to scale the Deployment up before draining, or to temporarily loosen the PDB.
9. Voluntary disruptions are ones initiated through the Eviction API by Kubernetes' own orchestration mechanisms (node drains, cluster-autoscaler scale-downs) — these are the only kind a PodDisruptionBudget can constrain. Involuntary disruptions (node hardware failure, kernel panic, spot instance reclamation, out-of-resource kill) are outside the Eviction API entirely, and a PDB has no ability to prevent or delay them.
10. `pluto` scans a live cluster, Helm releases, or manifest directories for Kubernetes API versions relative to a target Kubernetes version. `DEPRECATED` means the API still works on the target version but is scheduled for future removal (fix soon); `REMOVED` means the API is already gone on the target version and the manifest will fail to apply immediately after upgrading (fix before you upgrade).
11. Velero backs up Kubernetes API objects (as JSON to an external object store) and, if configured with the appropriate provider snapshot support or its file-system backup mode, the contents of persistent volumes too. It explicitly does not replace cluster-level infrastructure backup (it can't recreate the EKS control plane, IAM roles, or VPC configuration from scratch) — it restores workloads/objects into an existing cluster, not the surrounding infrastructure.
12. Strong answer: "First, check for deprecated or removed Kubernetes APIs in everything running on the cluster using a tool like `pluto`, and fix anything that would break on the target version using `kubectl convert`, reviewing the diff manually. Take a Velero backup as a safety net. Then upgrade the control plane one minor version at a time — EKS enforces this ordering and won't let you upgrade nodes past the current control plane version, because of Kubernetes' version skew policy. After the control plane, update add-ons (VPC CNI, CoreDNS, kube-proxy, CSI drivers) for compatibility. Then upgrade the node groups — for a routine, low-risk bump I'd do a rolling in-place node group upgrade, which respects PodDisruptionBudgets automatically via cordon/drain; for a customer-facing, high-stakes cluster I'd prefer a blue/green node group (or even a full blue/green cluster for a major version jump) so there's a trivial rollback path if the new nodes misbehave. Throughout, PodDisruptionBudgets on critical Deployments make sure drains never take availability below a safe floor, and I'd validate with a canary or a subset of traffic before considering the upgrade complete."
