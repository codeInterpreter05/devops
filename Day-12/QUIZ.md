# Day 12 — Quiz: Container Security

Try to answer without looking at your notes. Answers are at the bottom.

1. Why does running a container process as root matter more than it might seem, given that containers (unlike VMs) share the host kernel?
2. What's the difference between what `--cap-drop ALL --cap-add NET_BIND_SERVICE` restricts and what a seccomp profile restricts? Are they redundant?
3. What does Docker's default seccomp profile actually do, and how would you confirm it's active on a running container?
4. Explain how `--read-only` and `--tmpfs` work together. Why should a well-designed twelve-factor app need very few (or zero) tmpfs mounts?
5. How does a scanner like Trivy actually determine that an installed package is vulnerable? Name the two categories of vulnerability data it draws from.
6. Why is `trivy image myapp:latest` on its own **not** sufficient as a CI gate? What flag turns it into an enforced gate?
7. What's the architectural difference between how Grype and Trivy approach scanning (SBOM-first vs. combined)?
8. Why is scanning an image only once, at build time, not sufficient?
9. Why must Cosign signing and verification always be done against the image's content digest rather than its tag?
10. What's the difference between traditional key-pair signing and Sigstore's keyless signing with Cosign? What replaces the long-lived private key in the keyless flow?
11. Signing an image with Cosign is not, by itself, a security control. What's the missing piece that makes it one, and what happens without it?
12. **Interview question:** How do you prevent a container breakout? Name 4 hardening steps you always apply.

---

## Answers

1. A VM has its own kernel, isolated by a hypervisor; a container is a regular Linux process isolated only by namespaces and cgroups, running on the *same* kernel as the host and every other container on that node. UID 0 inside a container is the kernel's real UID 0 unless a user namespace remaps it — so if a container process escapes its namespace boundary (a kernel exploit, `--privileged`, a mounted Docker socket, a bad host-path mount), root inside the container becomes root on the host, and from there every other container on that node is reachable too. Even without an escape, running the app itself as root means any application-level RCE hands the attacker root over everything the container can touch.
2. Capabilities restrict *what a privileged operation is allowed to do* (e.g., without `CAP_NET_RAW`, a process can't open raw sockets, regardless of whether it's "root"). Seccomp restricts *which syscalls can be invoked at all*, independent of privilege level. They're complementary, not redundant — a process could hold zero extra capabilities and still be able to call a dangerous syscall if seccomp doesn't block it (and vice versa); real hardening uses both together.
3. It blocks around 44 syscalls considered dangerous or unnecessary inside a container — `mount`, `umount2`, `reboot`, `swapon`, `keyctl`, `add_key`, and restricts dangerous flag combinations on calls like `clone`. Confirm it's active with `grep Seccomp /proc/<pid>/status` inside or against the container — `Seccomp: 2` means filtering is active.
4. `--read-only` mounts the container's root filesystem read-only; `--tmpfs <path>` adds back a specific writable, RAM-backed mount at exactly the paths the app needs (e.g., `/tmp`), wiped on container stop. A twelve-factor app writes logs to stdout/stderr rather than files and avoids writing persistent state to its own container filesystem (state belongs in an external store), so it typically needs at most `/tmp` or a socket directory — not broad write access to its own filesystem.
5. Trivy walks the image's layers to build an inventory of installed package name + exact version (from OS package databases like `dpkg`/`rpm`/`apk`, and from language-ecosystem lockfiles like `package-lock.json`/`requirements.txt`/`go.sum`), then matches that inventory's version ranges against vulnerability databases: the NVD (CVE source of record) and distro-specific security advisories (Debian Security Tracker, Red Hat OVAL/RHSA, Alpine secdb, Ubuntu USN), plus ecosystem advisory databases like GitHub's for language packages.
6. By default, Trivy always exits `0` regardless of what it finds — it's a reporting tool until you opt into gating. `--exit-code 1` (combined with a `--severity` filter, typically `HIGH,CRITICAL`) makes Trivy exit nonzero when a matching finding exists, which is what a CI pipeline step actually checks to decide pass/fail.
7. Grype is built around a two-step pipeline: generate a Software Bill of Materials with its sibling tool Syft first, then match that SBOM against vulnerability feeds — the SBOM is a reusable, durable artifact decoupled from the scan itself. Trivy does both steps (inventory + match) in one combined pass by default, though it can also emit SBOMs. Grype can also scan an image directly, running Syft internally, but its design philosophy treats the SBOM as the first-class artifact.
8. New CVEs are discovered against already-published package versions constantly — an image that was completely clean the day it was built can have a CRITICAL CVE disclosed against one of its dependencies a month later, with nothing about the image itself having changed. A build-time-only scan is a snapshot; it says nothing about the image's status after that point. That's why scanning needs to also happen at the registry (scan-on-push and periodic re-scans) and at runtime (admission-time checks or continuous in-cluster scanning).
9. Image tags (`myapp:latest`) are mutable pointers — anyone with push access can repoint a tag to different content without anything visibly changing. The content digest (`sha256:...`) is immutable and derived directly from the image's bytes. Signing/verifying a tag would mean an attacker who repoints the tag after signing defeats verification entirely; binding to the digest is what makes the guarantee mean anything.
10. Traditional key-pair signing requires generating, protecting, and rotating a long-lived private key (ideally KMS-backed) that an attacker could steal or that could be mismanaged. Keyless signing removes that long-lived key entirely: Cosign requests a short-lived certificate from Fulcio based on an OIDC identity (e.g., a CI pipeline's native workflow identity token), signs with an ephemeral key discarded immediately after use, and records the event in Rekor's public transparency log so the signing action is independently auditable. OIDC identity + a transparency log replace the long-lived key as the root of trust.
11. The missing piece is admission-time (or deploy-time) enforcement — something that actually checks the signature before letting the image run and rejects anything unsigned or signed by an unexpected identity. In Kubernetes this is typically a Kyverno or OPA/Gatekeeper policy calling Cosign's verification logic at admission. Without it, signing is just documentation: nothing stops an unsigned or tampered image from being deployed anyway.
12. Strong answer: "A container breakout means a process escapes its namespace/cgroup boundary and reaches the host or other containers — since containers share the host kernel, this is always a real risk, not a theoretical one. Four hardening steps I always apply: (1) never run the container process as root — set `USER` in the Dockerfile and enforce `runAsNonRoot: true` in Kubernetes so a missing `USER` line fails closed instead of silently running as root; (2) drop all Linux capabilities and add back only the specific ones proven necessary (`--cap-drop ALL --cap-add <specific>`) instead of running with Docker's full default set; (3) mount the root filesystem read-only (`--read-only` plus targeted `--tmpfs` for the few paths that need write access), so even a successful RCE can't persist a backdoor or modify application code on disk; (4) never run with `--privileged`, and keep Docker's default seccomp profile enabled (or a tighter custom one) rather than disabling it — `--privileged` alone reverses nearly every other hardening step at once." A strong follow-up mentions that these are defense-in-depth layers, not a substitute for keeping the image itself free of known CVEs via scanning.
