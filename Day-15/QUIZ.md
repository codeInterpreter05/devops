# Day 15 — Quiz: Python for Automation I

Try to answer without looking at your notes. Answers are at the bottom.

1. What's the difference between `subprocess.run()`, `Popen`, and `check_output()`? When would you reach for `Popen` instead of `run()`?
2. Why is `shell=False` with a list of arguments safer than `shell=True` with a string? What specific attack does `shell=True` expose you to?
3. **Interview question:** How do you run a shell command from Python safely? What are the risks of `shell=True`?
4. What does passing `check=True` to `subprocess.run()` actually do, and what silently happens if you omit it and the command fails?
5. Why can reading a `Popen` object's `stdout` and writing to its `stdin` directly — without `communicate()` — cause a deadlock?
6. What's the practical difference between `os.path` and `pathlib.Path`? Give one concrete example of something `pathlib` makes easier.
7. Why is `click` generally preferred over `argparse` for real, maintained CLI tools? Name two concrete features that justify the extra dependency.
8. Why is `print()` insufficient for a production automation tool's diagnostics?
9. What's the difference between a logger's `setLevel()` and a handler's `setLevel()`? Which one wins if they disagree?
10. What bug can occur if you attach handlers directly to a non-root logger without setting `propagate = False`, especially in combination with `logging.basicConfig()`?
11. Why do virtual environments matter, and what's specifically wrong with running `sudo pip install <package>` against system Python?
12. What problem does `pyproject.toml` solve that the older `setup.py` + `requirements.txt` combination did not?

---

## Answers

1. `subprocess.run()` is a synchronous, blocking call that runs a command, waits for it to finish, and returns a `CompletedProcess` with `.returncode`/`.stdout`/`.stderr` — the right default for "run this and wait." `check_output()` is an older, narrower helper that returns just captured stdout and raises on failure; it's essentially what `run(..., stdout=PIPE, check=True).stdout` gives you. `Popen` is the lower-level primitive both are built on — reach for it directly when you need to stream output line-by-line while a process runs, manage multiple concurrent processes, pipe between two Python-managed processes, or interact with a process's stdin/stdout as it runs rather than waiting for it to finish.
2. With `shell=False` (the default) and a list of arguments, Python executes the program directly via `exec`-family syscalls with no shell parsing involved — so shell metacharacters (`;`, `|`, `&&`, `` ` ``, `$()`) in any argument are treated as literal characters, not commands. With `shell=True`, Python hands your string to `/bin/sh -c "<string>"`, so any of those metacharacters are live and interpretable. If any part of that string comes from untrusted input (user input, an API response, a config file), an attacker can inject and run arbitrary commands — this is OS command injection, CWE-78.
3. Strong answer: default to `shell=False` with the command passed as a **list** of arguments (e.g., `subprocess.run(["ping", "-c", "1", hostname])`) — this is immune to shell injection by construction because no shell ever parses the string, so metacharacters in `hostname` are inert. The risk of `shell=True` is that it hands your string to `/bin/sh -c`, so if any part of the command is built from untrusted input via string formatting, an attacker can append `; malicious-command` (or use `|`, backticks, `$()`) and have it executed with your process's privileges. If shell features (pipes, globbing) are genuinely required, prefer restructuring with `Popen` pipelines, or use `shlex.split()`/`shlex.quote()` rather than trusting raw string interpolation — but the list-argument, `shell=False` form should be the default for everything else. Also mention always setting `timeout=` and `check=True`/handling `CalledProcessError` as part of "running safely," since an interviewer may extend the question to reliability, not just injection.
4. `check=True` makes `run()` raise `subprocess.CalledProcessError` if the command exits with a non-zero return code, and that exception carries `.returncode`, `.stdout`, and `.stderr` for diagnostics. Without it, a failing command returns normally with a non-zero `.returncode` that you must remember to check yourself — omit that check and a failed command (host unreachable, permission denied, command not found via the shell) is silently treated as if it succeeded, which is a common source of automation that "completes" while having done nothing.
5. The OS pipe connecting parent and child has a limited buffer (commonly ~64KB on Linux). If you write to the child's stdin and read its stdout manually and sequentially (rather than concurrently), you can end up with the child blocked writing to a full stdout pipe (because your parent hasn't read it yet — it's still busy writing to stdin) while your parent is blocked writing to the child's stdin (because the child isn't reading it yet, or the child's own output pipe is full) — both sides wait on each other forever. `Popen.communicate()` avoids this by reading/writing via internal threads (or select-based multiplexing) so both streams are serviced concurrently.
6. `os.path` represents paths as plain strings and requires separate function calls for each operation (`os.path.join`, `os.path.dirname`, `os.path.splitext`, etc.); `pathlib.Path` represents a path as an object with chainable methods and operators. Concretely, `pathlib` makes joining and decomposing paths far more readable: `Path("/var/log") / "app.log"` versus `os.path.join("/var", "log", "app.log")`, and getting the extension is `path.suffix` instead of `os.path.splitext(path)[1]`. `pathlib` is also OS-separator-correct automatically and lets you read/write whole files (`.read_text()`/`.write_text()`) without manual `open()`/`close()` handling.
7. `click` uses declarative decorators instead of imperative parser-building, which scales much better once a tool has multiple subcommands (`@click.group()` and nested commands stay clean; `argparse`'s `add_subparsers()` gets verbose past 2-3 levels). Two concrete justifying features: (1) richer built-in types for validation — `click.Path(exists=True)`, `click.Choice([...])`, `click.IntRange(...)` validate and convert input at parse time without hand-written checks; (2) `click.testing.CliRunner`, which lets you invoke a command in-process and assert on `exit_code`/`output` without spawning a real subprocess or manipulating `sys.argv`, making CLI tools far easier to unit test than raw `argparse`-based ones.
8. `print()` has no concept of severity (no way to distinguish routine output from a warning or error without ad hoc string prefixes), can't be selectively silenced or made more verbose without editing code, and always goes to stdout with no built-in way to also (or instead) write to a file, syslog, or a rotating log with a timestamp — all of which the `logging` module provides out of the box. It also doesn't integrate with third-party libraries' own diagnostics the way `logging` does, since libraries are expected to log via the stdlib `logging` module, not print.
9. `logger.setLevel()` sets the logger's own floor — any record below that level is discarded before it's ever dispatched to a handler. Each handler can also call its own `setLevel()`, which filters further, but only in the direction of raising the effective floor for that specific destination — a handler can never see a level lower than what the logger already let through. So if the logger is set to `INFO`, a handler set to `DEBUG` still only sees `INFO` and above; the logger's level always wins if it's the stricter (higher) one.
10. Loggers, by default, propagate their records up to their ancestor loggers (ultimately the root logger) *in addition* to firing their own handlers. If `logging.basicConfig()` (or some imported library) has configured a handler on the root logger, and your own logger also has a handler attached without `propagate = False`, every log record gets emitted twice — once via your handler, once via the root's. This shows up as confusing duplicate log lines and is a very common real-world logging bug.
11. A virtual environment gives each Python project its own isolated `site-packages`, so different projects can depend on different (even conflicting) versions of the same package without interfering with each other, and so a project's exact dependency set is reproducible and discoverable rather than mixed in with everything else installed globally. `sudo pip install` into system Python risks breaking OS-level tools that themselves depend on Python and specific package versions (many Linux distros ship Python-based system utilities), and it makes it impossible to tell, just by looking at global `site-packages`, which packages any particular project actually needs.
12. `setup.py` is executable Python code that runs arbitrary logic just to install a package — not declarative, harder to statically inspect, and a minor security/reproducibility smell. `requirements.txt` only lists dependencies (often unpinned) with nowhere to put per-tool configuration. `pyproject.toml` consolidates project metadata, build-system declaration (PEP 518), dependency declarations (PEP 621), and per-tool configuration tables (`[tool.ruff]`, `[tool.mypy]`, `[tool.pytest.ini_options]`, etc.) into one static, declarative TOML file that any tool can read its own section from — no code execution required just to discover a package's metadata.
