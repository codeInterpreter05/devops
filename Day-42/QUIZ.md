# Day 42 — Quiz: K8s + IaC + AWS Mock Interview

Try to answer without looking at your notes. Answers are at the bottom.

1. In a system design interview, why is spending the first few minutes clarifying requirements graded as heavily as the final architecture?
2. Name the two separate layers of auto-scaling needed for a Kubernetes-hosted API, and what each one actually scales.
3. Why can CPU-based HPA scaling be a weak choice for a request-heavy but CPU-light service?
4. What's the key access-pattern trade-off between Aurora and DynamoDB that should be named explicitly in a data-layer discussion?
5. Why can a plain Kubernetes rolling update still cause an outage even if every pod's readiness probe is configured correctly?
6. What is the first command you should always run when Terraform's plan output looks wrong, before touching any infrastructure?
7. What does `terraform plan -refresh-only` do, and why should it be run before a normal `apply` when console drift is suspected?
8. What's the difference between `terraform state rm` and `terraform destroy` for a given resource?
9. Does granting `get`/`list` on `pods` also grant access to `pods/log`? Why or why not?
10. What's the difference in effective scope between binding a `ClusterRole` via a `RoleBinding` versus via a `ClusterRoleBinding`?
11. What tool gives you the fastest, most direct answer to "would this specific ServiceAccount be allowed to do this specific action," without manually tracing RBAC YAML?
12. **Interview question:** Design the complete AWS architecture for a globally available microservices app.

---

## Answers

1. Because designing the wrong thing efficiently is worse than designing the right thing slowly — an architecture built for global multi-region HA when the real requirement was a single-region API (or vice versa) signals that a candidate follows a memorized template rather than reasoning from actual constraints (RPS shape, latency SLA, consistency needs, budget). Interviewers explicitly watch for this.
2. Node-level scaling (Karpenter or Cluster Autoscaler — adds/removes actual EC2 nodes to match aggregate pending pod resource requirements) and pod-level scaling (Horizontal Pod Autoscaler — adds/removes pod replicas within existing node capacity, based on CPU/memory or custom metrics). Both are necessary; addressing only one is an incomplete answer.
3. A service that spends most of its time waiting on I/O (database calls, downstream API calls) rather than doing CPU-bound work can be handling a very high request rate while CPU usage stays low — HPA configured only on CPU won't trigger new replicas fast enough (or at all) even as request queues/latency degrade. Custom metrics (e.g., requests-per-second via Prometheus Adapter) address this gap.
4. Aurora offers strong relational consistency and flexible ad-hoc querying/joins but has a comparatively harder horizontal write-scaling ceiling; DynamoDB offers near-infinite horizontal scaling and single-digit-millisecond latency but requires access patterns to be designed up front, with no ad-hoc joins — retrofitting DynamoDB after query patterns are already relational is expensive.
5. Because zero-downtime deploys are only as safe as the riskiest accompanying database schema migration — a breaking (non-backward-compatible) migration deployed alongside new application code can cause errors or downtime regardless of how gracefully the pods themselves roll over. The fix is an expand/contract migration strategy (add new schema additively, deploy code that handles both old and new shapes, backfill, then remove the old shape later), not just a Kubernetes-level deployment strategy alone.
6. `terraform plan` — always look at exactly what Terraform is proposing (modify vs. destroy-and-recreate vs. no-op) before taking any further diagnostic or corrective action; the specific diff shape drives which root cause to investigate next.
7. `-refresh-only` updates Terraform's recorded state to match the real infrastructure's current values, without changing any actual infrastructure. It should be run first when console/out-of-band drift is suspected because it separates "sync state to reality" from "change reality to match code" — running a normal `apply` directly in that situation risks silently reverting someone's legitimate manual change instead of surfacing it as a deliberate decision.
8. `terraform state rm` removes a resource from Terraform's state file only — the real infrastructure object is left completely untouched, and Terraform simply "forgets" it manages that resource. `terraform destroy` (or destroying a specific resource) actually deletes the real infrastructure object. Confusing the two is dangerous in both directions — using `destroy` when you meant to just stop tracking something, or using `state rm` when you actually intended to tear the resource down.
9. No — `pods/log` is a separate subresource from `pods` in Kubernetes' RBAC model, and must be explicitly listed in a `Role`'s/`ClusterRole`'s `resources` field. Permission on the parent resource (`pods`) does not automatically extend to its subresources (`pods/log`, `pods/exec`); this is a very commonly-missed detail when troubleshooting "why does `kubectl logs` say forbidden even though I can `kubectl get pods` fine."
10. A `ClusterRole` bound via a namespaced `RoleBinding` grants those permissions **only within that RoleBinding's namespace** — even though the `ClusterRole` object itself is cluster-scoped, its effect is scoped down by the binding. A `ClusterRoleBinding` grants the same `ClusterRole`'s permissions **cluster-wide**, across every namespace at once. The former is the standard least-privilege pattern for reusing one permission definition narrowly; the latter is a much broader grant.
11. `kubectl auth can-i <verb> <resource> --as=<subject> -n <namespace>` — it directly evaluates and answers the exact authorization question against the live RBAC configuration, far faster and more reliably than manually tracing through Role/ClusterRole/Binding YAML by hand.
12. Strong answer: "I'd start by clarifying scope — how many regions, what latency requirements drive 'globally available' (e.g., sub-100ms in each major region), and whether data needs to be globally consistent or can be eventually consistent per-region. At a high level: Route 53 with latency-based or geo-routing sends users to the nearest healthy region; each region runs an independent EKS cluster behind an ALB, fronted by CloudFront for static assets and edge caching; each microservice is a separate Deployment with its own HPA and dedicated data store or shared regional database with cross-region replication (e.g., Aurora Global Database, or DynamoDB Global Tables for simpler access patterns) depending on consistency needs. For truly global data, I'd flag the CAP-theorem trade-off explicitly: strong cross-region consistency costs latency, so most globally available designs accept eventual consistency for non-critical data and reserve strong consistency for specific critical paths (e.g., payments) routed to a single 'home' region per user/account. Cross-service communication would go through an API gateway or service mesh per region, secrets managed via Vault or Secrets Manager replicated per region, and deployments via GitOps (ArgoCD/Flux) per cluster, promoted through a pipeline rather than manually per region. I'd close by naming the operational cost of this design honestly — running N independent regional clusters multiplies operational complexity, monitoring surface, and cost roughly N-fold compared to a single-region design, which is exactly the trade-off 'globally available' is buying." A weak answer lists AWS service names without addressing the CAP trade-off or the operational cost of true multi-region operation.
