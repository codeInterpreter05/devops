# Day 71 — Quiz: Testing in CI Pipelines

Try to answer without looking at your notes. Answers are at the bottom.

1. Why does the test pyramid have a specific shape (many unit, some integration, few e2e) rather than just "have some of each"?
2. What's the "ice cream cone" anti-pattern, and why does it hurt CI reliability specifically?
3. In `pytest`, what does `@pytest.mark.parametrize` actually save you from doing, and why does it also improve edge-case coverage?
4. Why mock `boto3.client` in a unit test instead of letting it make a real AWS call?
5. What can Terratest verify that `terraform validate` and Conftest cannot, and why?
6. Where should `defer terraform.Destroy(...)` be placed in a Terratest test function, and why does the placement matter?
7. What specific problem does consumer-driven contract testing (Pact) solve that full e2e testing does not solve well?
8. Walk through the Pact flow: what does the consumer side produce, where does it go, and what does the provider side do with it?
9. What is a smoke test meant to catch that no amount of pre-deploy testing (in a different environment) could have caught?
10. Why does `if: always()` matter on a test-report-publishing CI step?
11. What genuinely can't be reliably tested about Infrastructure as Code through static analysis alone, and why does Terratest's approach address that gap?
12. **Interview question:** How do you test Infrastructure as Code? What can and can't be unit-tested?

---

## Answers

1. Because each layer trades speed/reliability against realism differently: unit tests are fast and precise but tell you nothing about component interaction; e2e tests are realistic but slow and prone to flakiness unrelated to real bugs. Weighting the suite toward fast, precise tests (many unit) and reserving the slow, brittle layer (e2e) for a small number of critical end-to-end validations keeps CI both fast and reliable.
2. The "ice cream cone" anti-pattern is mostly e2e/manual tests with very few unit tests — inverted from the intended pyramid shape. It hurts CI reliability because e2e tests are slow and flaky for reasons unrelated to real bugs (network timing, environment quirks), so a CI suite dominated by them becomes slow and produces frequent false-failure noise, which teams learn to ignore — masking real regressions when they do occur.
3. It saves you from writing multiple nearly-identical test functions that differ only in input/expected-output values — you write the assertion logic once and supply a table of cases. It improves edge-case coverage because adding a new edge case (zero, boundary value, remainder) becomes "add one row to the table" instead of "write a whole new test function," lowering the friction to actually including edge cases people might otherwise skip.
4. So the unit test runs in milliseconds with no real AWS credentials needed and zero risk of accidentally creating/modifying/deleting real AWS resources — keeping the unit-test layer fast and dependency-free, which is the entire point of it sitting at the base of the pyramid. Real AWS interaction (or a realistic mock like `moto`) belongs at the integration layer instead.
5. Terratest actually applies real Terraform and asserts against real created cloud resources (often via the cloud provider's own SDK), proving actual runtime behavior — e.g., that private subnets genuinely have no route to the internet. `terraform validate` only checks HCL syntax/internal consistency, and Conftest only checks the *declared* plan against known-bad patterns; neither creates real infrastructure or observes actual runtime behavior.
6. Immediately after constructing `terraformOptions`, before calling `terraform.InitAndApply`. Placing it there (via `defer`) guarantees the destroy/cleanup runs even if the apply, an assertion, or any later code in the test panics or fails — critical since real, billable cloud resources are involved and leaked resources are a real cost/security risk.
7. Full e2e tests require both the consumer and provider services to actually be running together to validate their integration, which couples every team's testing to every other team's service being deployed and correctly configured — slow, brittle, and requires cross-team coordination. Contract testing lets each side test independently against a shared, versioned contract (the pact file), with neither team needing the other's service running at test time.
8. The consumer writes a test against Pact's mock provider, asserting its own code correctly handles an expected request/response shape; this produces a pact file (JSON contract) as a byproduct. The pact file is published to a Pact Broker. The provider team's own CI pulls that pact file and runs provider verification — replaying the consumer's expected requests against the real provider code and asserting the real responses match what the consumer expects.
9. Environment-specific failures that are unique to *this* particular deploy in *this* particular real environment — a missing/wrong secret in that environment, a database migration that silently didn't run, a misconfigured environment variable. No pre-deploy test run in a different environment (dev, CI's own test environment) can catch a problem that's specific to production's actual current state.
10. Without `if: always()`, the report-publishing step is skipped whenever the preceding test step fails (the default CI behavior on failure is to skip subsequent steps) — meaning exactly the run you most need a structured, browsable report for (a failing one) is the one where it's least likely to get published, forcing debugging through raw console logs instead.
11. Emergent, cross-resource runtime behavior — for example, whether a security group rule, route table, and NACL combination actually results in the network reachability you intended. Static analysis (HCL syntax checks, policy checks against a plan) can verify declared configuration values, but cannot observe actual behavior because nothing real has been created yet. Terratest addresses this by actually applying the Terraform and then making real assertions (often via the cloud provider's SDK) against the live, created resources — proving behavior rather than just declared intent.
12. Strong answer: "Infrastructure as Code testing happens in layers, similar to the application test pyramid. Static checks — `terraform validate`, `terraform fmt`, and policy tools like Conftest against the JSON plan — are fast, free, and catch syntax errors and known-bad configuration patterns (like an overly permissive security group) without creating any real infrastructure; that's the cheap, high-volume layer. What can't be verified statically is actual runtime behavior — whether a private subnet genuinely has no route to the internet, whether an IAM role's effective permissions actually match what was intended once combined with other policies, whether a Lambda's VPC config actually lets it reach an RDS instance through the real subnet/route-table/NACL chain. That requires actually creating the infrastructure and observing it, which is what Terratest does: it applies real Terraform against a disposable test account, asserts against the real created resources — often via the cloud provider's own SDK, not just Terraform's outputs — and tears everything down afterward, with `defer terraform.Destroy(...)` placed immediately so cleanup happens even on failure. So the honest answer to 'what can't be unit-tested' is: emergent, cross-resource runtime behavior can't be verified without real infrastructure; everything about declared configuration correctness and known-bad patterns can and should be caught earlier and more cheaply, before you ever get to that point."
