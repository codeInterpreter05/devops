# Day 24 — Quiz: Networking — Services & Ingress

Try to answer without looking at your notes. Answers are at the bottom.

1. What actually determines a Service's set of backend Pods, and where does that live?
2. What's the difference between `ClusterIP`, `NodePort`, `LoadBalancer`, and `ExternalName` — specifically, which ones proxy traffic and which one is purely a DNS alias?
3. Why does creating a `LoadBalancer` Service for every microservice in a cluster become expensive and unwieldy, and what's the standard fix?
4. Is the `Ingress` API object itself capable of routing any traffic on its own? What has to exist for it to do anything?
5. What does `ingressClassName` control, and when does it become important?
6. Walk through the full path a request takes from the public internet to a specific Pod, through a cloud load balancer and an Ingress Controller.
7. What role does cert-manager play, and what does it actually write into the cluster once a certificate is issued?
8. Why should you generally use a staging ACME issuer while testing cert-manager configuration rather than the production Let's Encrypt endpoint?
9. What is CoreDNS, concretely, in terms of Kubernetes objects — and what happens if it's scaled to zero replicas?
10. What does `ndots:5` in a pod's `resolv.conf` cause, and why is it a real performance consideration for services that call external APIs often?
11. What's required for a headless Service + StatefulSet combination to produce meaningful per-pod DNS entries?
12. **Interview question:** Explain how a request travels from the internet to a Pod through an Ingress.

---

## Answers

1. A Service's backend set is determined by its label **selector**, continuously matched against Pods by the Endpoint/EndpointSlice controller — the live result (actual `podIP:port` pairs currently considered ready) is stored in the Service's **Endpoints** (or **EndpointSlice**) object, not in the Service object itself.
2. `ClusterIP` (internal-only virtual IP, proxied via kube-proxy), `NodePort` (opens the same port on every node, proxied), `LoadBalancer` (provisions an external cloud LB in front of a NodePort, proxied) all involve real traffic proxying through kube-proxy's iptables/IPVS rules. `ExternalName` is different in kind — it has no selector, no virtual IP, no Endpoints, and does no proxying at all; it's purely a DNS CNAME alias that CoreDNS answers, after which the client talks directly to the external endpoint.
3. Each `LoadBalancer` Service provisions a real, separately billed cloud load balancer resource — with dozens of microservices, that's dozens of cloud LBs, which is costly and operationally unwieldy (managing DNS/certs per LB). The standard fix is `Ingress`: one (or a small number of) shared Ingress Controller, fronted by a single `LoadBalancer` Service, that does host/path-based routing to many internal Services.
4. No — `Ingress` is inert configuration stored in etcd like any other object. It only becomes meaningful once an **Ingress Controller** (a separate workload, e.g. ingress-nginx or the AWS Load Balancer Controller) is deployed, watches `Ingress` objects, and translates them into actual proxy/load-balancer configuration.
5. `ingressClassName` specifies which Ingress Controller should handle a given `Ingress` object. It becomes important as soon as a cluster has more than one Ingress Controller installed (e.g., a multi-tenant cluster running both ingress-nginx and the AWS Load Balancer Controller) — without it, routing can be ambiguous, picked up by an unintended controller, or ignored entirely depending on default-class configuration.
6. DNS resolves the hostname to a cloud load balancer's IP -> the cloud LB forwards the connection to one of the Ingress Controller's pods (reached via the controller's own Service, typically type `LoadBalancer`) -> the Ingress Controller (e.g., Nginx inside ingress-nginx) terminates TLS if configured, inspects the Host header/path, and matches it against its compiled routing rules from watched `Ingress` objects -> it proxies to the matched **Service's ClusterIP** -> kube-proxy's iptables/IPVS rules forward to one specific backend **Pod IP** from that Service's current Endpoints -> the Pod handles the request.
7. cert-manager watches `Ingress`/`Certificate` objects referencing an `Issuer`/`ClusterIssuer`, automatically performs the ACME challenge (HTTP-01 or DNS-01) against Let's Encrypt (or another ACME CA) to prove domain ownership, and writes the resulting TLS certificate and private key into the Kubernetes **Secret** named in the Ingress's `tls.secretName` field. It also automatically renews the certificate before the (typically 90-day) expiry.
8. Let's Encrypt's production endpoint enforces strict rate limits (e.g., duplicate certificate limits per domain per week); repeated testing/iteration against production while debugging configuration can exhaust those limits and lock you out of getting a real certificate for that domain for a period of time. The staging endpoint has much higher rate limits (its certs just aren't trusted by browsers), making it safe for iterative testing.
9. CoreDNS is an ordinary Deployment (commonly 2 replicas) in `kube-system`, fronted by a `ClusterIP` Service (`kube-dns`), configured via a `Corefile` stored in a ConfigMap. If scaled to zero, all new DNS lookups cluster-wide fail (existing already-established connections using previously resolved IPs keep working, but anything needing a fresh lookup — including retries, new connections, and most application startup — breaks), making it a critical, must-be-highly-available component despite looking like invisible infrastructure.
10. `ndots:5` means any queried name with fewer than 5 dots gets the pod's `search` domains appended and tried first, in order, before being tried as an absolute name. For short external hostnames (e.g., `api.stripe.com`, 2 dots), this means several guaranteed-to-fail internal DNS lookups happen before the real external lookup succeeds — extra latency and DNS query volume on every such call. Fixing it means using a trailing-dot FQDN (`api.stripe.com.`) or tuning `dnsConfig.options` to lower `ndots` for that workload.
11. The Service must be **headless** (`clusterIP: None`) and the StatefulSet's `spec.serviceName` must match that Service's name. Only then does CoreDNS's `kubernetes` plugin produce one DNS A record per individual pod (`pod-name.service.namespace.svc.cluster.local`) instead of resolving the Service name to a single load-balanced virtual IP.
12. Strong answer: "DNS resolves the hostname to a cloud load balancer, which forwards the connection to a pod running the Ingress Controller — itself just an ordinary workload exposed by its own Service. The controller terminates TLS if configured, inspects the Host header and path against the routing rules compiled from every Ingress object it watches, and proxies the matched request to the target Service's ClusterIP. From there it's normal Service routing: kube-proxy's iptables or IPVS rules forward to one specific backend pod IP drawn from that Service's current Endpoints. The important point is that Ingress doesn't replace or bypass the Service model — it's an HTTP-aware reverse proxy layered in front of it, adding host/path-based routing and centralized TLS termination that a bare Service type can't provide on its own."
