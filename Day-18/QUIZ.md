# Day 18 — Quiz: Storage & Filesystems

Try to answer without looking at your notes. Answers are at the bottom.

1. What's the fundamental difference between block, file, and object storage — in terms of how an application actually addresses and writes data to each?
2. Why is it a bad idea to run a database's data directory directly on S3 (or an `s3fs`-style mount of S3)?
3. Name the three layers of an LVM stack, in order, and explain what problem LVM solves that a plain disk partition doesn't.
4. You just ran `lvextend -L +10G /dev/vg_data/lv_data`. `df -h` still shows the old, smaller size. What did you forget, and what's the exact command for an ext4 filesystem vs. an XFS filesystem?
5. List the six fields of an `/etc/fstab` line, in order, and explain what `pass` (the sixth field) controls.
6. Why should you prefer `UUID=` over a raw `/dev/sdX` path in `/etc/fstab`?
7. What do the `nofail` and `_netdev` mount options each do, and why do NFS/iSCSI fstab entries typically need both?
8. What is a "stale file handle" in NFS, and how do you actually recover from one (as opposed to just retrying the failing command)?
9. In `iostat -xz 1` output, which two columns together give you the strongest signal that a disk is genuinely saturated, and why is `await` more trustworthy than `%util` alone?
10. `df -h` shows plenty of free space on a filesystem, but you still get "No space left on device" when creating a new file. What's the likely cause, and which command confirms it?
11. **Interview question:** When would you choose EBS over EFS on AWS? What are the tradeoffs?
12. What's the difference between `iostat` and `iotop`, and in what order would you typically use them when investigating a slow application?

---

## Answers

1. Block storage addresses data as fixed-size raw blocks with no inherent structure — the OS/application must format it with a filesystem or address it directly; only one host normally attaches to it at a time. File storage exposes a hierarchical namespace (directories/files) with POSIX-like operations (open/read/write, locking) shared concurrently across multiple clients over a network protocol (NFS/SMB). Object storage stores whole blobs in a flat namespace addressed by key, accessed via an HTTP API (PUT/GET/DELETE) with no partial in-place writes and no native locking.
2. S3 (and `s3fs`-style mounts approximating a filesystem over it) has no real file locking and only supports whole-object overwrites, not efficient small random writes — a database needs both to maintain data integrity and reasonable performance under concurrent access, so it will either behave incorrectly or perform very poorly.
3. Physical Volume (PV) → Volume Group (VG) → Logical Volume (LV). LVM solves the problem of plain partitions being fixed in size and position — resizing a raw partition traditionally required unmounting and repartitioning; LVM lets you add disks to a pool (VG) and grow logical volumes carved from that pool without repartitioning or downtime.
4. Growing the LV does not grow the filesystem living inside it — that's a separate step. For ext4: `resize2fs /dev/vg_data/lv_data`. For XFS: `xfs_growfs /mount/point` (note XFS takes the *mount path*, not the device, and can only grow, never shrink). `lvextend -r` combines the LV resize and filesystem resize into one step.
5. Device, mount point, filesystem type, options, dump, pass — in that order. `pass` controls `fsck` ordering at boot: `0` means never check that filesystem, `1` is reserved for the root filesystem (checked first), and `2` means checked after root, potentially in parallel with other `pass=2` filesystems.
6. Device names like `/dev/sdb` are assigned by enumeration order at boot and can shift — adding, removing, or reattaching disks (especially common on cloud VMs) can change which name maps to which physical disk. A UUID is generated with the filesystem and stays constant regardless of enumeration order, so fstab entries keep pointing at the correct disk.
7. `nofail` tells the boot process not to halt/drop to an emergency shell if that device is missing or the mount fails — critical for anything optional. `_netdev` tells the boot process this is a network filesystem, so it should wait until networking is up before attempting the mount. NFS/iSCSI entries need both because the network may not be ready yet when local mounts are normally attempted, and because the remote server being temporarily unreachable shouldn't be able to block the whole system from booting.
8. A stale file handle happens when the client is holding a cached reference to server-side file/export state that no longer matches reality — e.g., the export was recreated, remounted, or the underlying file was deleted while a client had it open. It shows up as a "Stale file handle" error on further operations. You can't fix it by retrying — you have to unmount and remount the client's view of that export (`umount -f` then `mount` again), since the invalid handle is cached client-side.
9. `%util` (percentage of time the device was busy) and `await` (average I/O completion time in milliseconds, including time spent queued) together. `await` is more trustworthy because a device — especially an SSD/NVMe with internal parallelism — can show high `%util` while still servicing requests with low, acceptable latency; sustained, climbing `await` reflects the latency an application would actually feel, which is the thing that matters.
10. The filesystem has likely run out of **inodes**, not space — this happens when a directory accumulates a very large number of small/empty files. `df -i` confirms it by showing inode usage at or near 100% even while `df -h` shows free bytes.
11. Choose EBS when a single instance needs low-latency, high-IOPS block access and you control capacity directly — the classic case is a database data directory or a boot volume; you pay for provisioned GB/IOPS whether or not you use them, and EBS volumes are AZ-scoped and normally single-attach. Choose EFS when multiple instances (potentially across multiple AZs) need concurrent, POSIX-correct access to the same shared file data — shared uploads, CMS content, fleet-wide home directories — accepting a bit more per-operation latency and a higher per-GB price in exchange for elastic capacity and not managing where the storage "lives." Reaching for EFS on a single-instance workload wastes money and adds latency for multi-AZ sharing capability you'll never use; reaching for EBS when multiple instances need the same live data simply doesn't work without a cluster filesystem, which EBS multi-attach rarely provides in practice.
12. `iostat` reports I/O statistics **per device** — it tells you whether a disk is saturated (via `%util`, `await`, throughput) but not which process is responsible. `iotop` reports I/O **per process**, like `top` for disk I/O, but doesn't give you the device-level saturation context on its own. The typical order is `iostat -xz 1` first to confirm a device is actually the bottleneck, then `iotop` to identify which specific process is driving that load.
