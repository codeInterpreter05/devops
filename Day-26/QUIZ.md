# Day 26 — Quiz: RBAC & Security

Try to answer without looking at your notes. Answers are at the bottom.

1. What's the difference between authentication and authorization in the context of a Kubernetes API request, and which one does RBAC handle?
2. What identity does a Pod use when it makes calls back to the Kubernetes API, and where does that token live?
3. What's the structural difference between what a `Role` can grant versus what a `ClusterRole` can grant?
4. Why is a `RoleBinding` that references a `ClusterRole` a common, useful pattern rather than a strange edge case?
5. What is the fastest way to check whether a specific ServiceAccount can perform a specific action, without actually attempting the real request?
6. Describe the full mechanism by which IRSA gets a pod real AWS credentials, from OIDC provider to the AWS SDK, without ever storing a static key.
7. Give three concrete reasons static AWS keys in a Kubernetes Secret are worse than IRSA.
8. What's the difference between what RBAC controls and what Pod Security Standards control? Why do you need both?
9. Name the three built-in Pod Security Standard profiles and briefly describe what each allows/blocks.
10. What's the difference between a mutating and a validating admission webhook, and which one runs first?
11. What does a webhook's `failurePolicy: Fail` vs `failurePolicy: Ignore` actually control, and what's the tradeoff?
12. **Interview question:** How does IRSA work? Why is it better than putting AWS keys in a Secret?

---

## Answers

1. Authentication answers "who is making this request" (verified via client certs, bearer tokens, OIDC, etc.); authorization answers "is this already-identified subject allowed to perform this specific action on this resource." **RBAC is entirely an authorization mechanism** — it has no role in proving identity, it only decides what an already-authenticated identity is permitted to do.
2. A Pod authenticates using its **ServiceAccount's token** — a projected, auto-rotated JWT mounted by default at `/var/run/secrets/kubernetes.io/serviceaccount/token`. If the Pod doesn't specify `serviceAccountName` explicitly, it silently uses the namespace's `default` ServiceAccount.
3. A `Role` can only grant permissions on **namespaced** resources, and only within its own namespace. A `ClusterRole` can grant permissions on **cluster-scoped** resources (Nodes, PersistentVolumes, Namespaces) that a `Role` structurally cannot touch at all, and can also be reused across multiple namespaces via separate namespace-scoped `RoleBinding`s.
4. Because it lets you define a permission set once as a reusable `ClusterRole` (e.g., Kubernetes' built-in `view`/`edit`/`admin` ClusterRoles) and then grant it in a specific, limited namespace via a `RoleBinding` — the `RoleBinding` scopes the ClusterRole's permissions down to just that one namespace, rather than requiring you to duplicate an identical `Role` definition in every namespace that needs the same permission set.
5. `kubectl auth can-i <verb> <resource> --as=system:serviceaccount:<namespace>:<sa-name>` (or `--as=<user>` for a human identity) — this evaluates the exact authorization path the apiserver itself uses, without needing to attempt and fail the actual request.
6. The EKS cluster has an IAM OIDC provider registered; you create an IAM role whose trust policy restricts `sts:AssumeRoleWithWebIdentity` to a specific ServiceAccount via a `sub` claim condition; you annotate that ServiceAccount with the role's ARN; the EKS Pod Identity mutating webhook automatically injects `AWS_ROLE_ARN` and a projected, auto-rotated OIDC token file into any pod using that ServiceAccount; the AWS SDK inside the app detects those environment variables automatically and exchanges the token for short-lived (~1 hour), auto-refreshed temporary AWS credentials via STS — no static key is ever created, stored, or transmitted.
7. Any three of: (1) static keys are indefinite-lived until manually rotated, versus IRSA's ~1-hour auto-refreshed credentials; (2) Kubernetes Secrets are base64-encoded, not encrypted, in etcd by default, so a leaked/backed-up etcd snapshot exposes the raw keys, whereas IRSA never stores a long-lived credential anywhere; (3) static keys are often shared broadly across pods/namespaces rather than scoped tightly to one workload, whereas IRSA's trust policy `sub` condition ties permissions to one specific ServiceAccount; (4) IRSA gives a cleaner audit trail (CloudTrail assumed-role sessions tied to the originating ServiceAccount) versus a shared IAM user/key showing no workload-level attribution.
8. RBAC controls **who is allowed to perform which API operations** (create/get/list/delete on which resource types, in which namespaces). Pod Security Standards/admission control **what a Pod object is allowed to actually contain** (whether it can run privileged, use host namespaces, run as root, mount arbitrary `hostPath`s). You need both because a user can have perfectly legitimate RBAC permission to create Pods and still create a dangerously configured one — RBAC says nothing about the *contents* of the object being created.
9. `Privileged` — no restrictions at all, appropriate only for trusted infra workloads. `Baseline` — blocks the most obvious escalations (no privileged containers, no host namespaces, restricted `hostPath`) while staying broadly compatible with typical apps. `Restricted` — the hardened profile, additionally requiring non-root, `allowPrivilegeEscalation: false`, dropping all Linux capabilities by default, and a seccomp profile.
10. A mutating webhook can **modify** the object being admitted (e.g., injecting a sidecar container or environment variables); a validating webhook can only **accept or reject** it, never change it. Mutating webhooks always run **before** validating webhooks, so validation sees the fully mutated final object, not the original submitted one.
11. `failurePolicy` controls what happens if the webhook service itself is unreachable when the apiserver tries to call it. `Fail` (fail closed) blocks the matching request entirely if the webhook can't be reached — safer for security-critical policies, but risks a broad outage (e.g., all Pod creation blocked) if the webhook itself has an incident. `Ignore` (fail open) lets the request through unchecked if the webhook is unreachable — keeps the cluster available, but means a webhook outage silently becomes a policy bypass window.
12. Strong answer: "IRSA uses OIDC federation instead of static credentials. The EKS cluster has an OIDC identity provider registered in IAM; an IAM role's trust policy is scoped to a specific Kubernetes ServiceAccount via a condition on the OIDC token's `sub` claim; the ServiceAccount is annotated with that role's ARN; and a mutating admission webhook automatically injects a projected, short-lived OIDC token plus the role ARN into any pod using that ServiceAccount. The AWS SDK inside the application automatically exchanges that token for temporary AWS credentials via `sts:AssumeRoleWithWebIdentity` — credentials that last about an hour and refresh automatically. It's better than static keys in a Secret because no long-lived credential ever exists to leak: Secrets are only base64-encoded in etcd by default (not encrypted), keys don't expire on their own and require manual rotation, and they're typically scoped far more broadly than the one-ServiceAccount-to-one-IAM-role mapping IRSA enforces — so a compromised pod or leaked Secret with IRSA yields, at worst, a short-lived, narrowly-scoped credential instead of an indefinite, broadly-scoped one."
