# Day 27 — Quiz: Autoscaling

Try to answer without looking at your notes. Answers are at the bottom.

1. What's the fundamental difference between what HPA does and what VPA does?
2. HPA reports `<unknown>` for a Deployment's current CPU metric. What's the most likely cause?
3. Why is running HPA and VPA both targeting CPU on the same workload, both in active/auto mode, a risky combination?
4. Why can't a plain CPU/memory-based HPA effectively autoscale a queue consumer that's I/O-bound and sits at low CPU usage even under a large backlog?
5. What does KEDA actually do under the hood when you create a `ScaledObject` — does it replace HPA?
6. What capability does KEDA provide that plain Kubernetes HPA cannot do at all?
7. What are `pollingInterval` and `cooldownPeriod` in a KEDA `ScaledObject`, and what does each control?
8. What is the fundamental architectural difference between Cluster Autoscaler and Karpenter in how they provision new nodes?
9. What's the specific limitation of Cluster Autoscaler that Karpenter's approach avoids?
10. Why are scale-up and scale-down cooldowns typically asymmetric (fast up, slow down) across HPA, KEDA, Cluster Autoscaler, and Karpenter alike?
11. What Kubernetes Pending-pod signal is what triggers a node autoscaler (CA/Karpenter) to act, and how is that different from what triggers HPA/KEDA to act?
12. **Interview question:** How would you auto-scale pods based on SQS queue depth? What are the tradeoffs vs HPA?

---

## Answers

1. HPA changes the **number of replicas** of a workload to respond to a metric; VPA changes the **resource requests/limits of each individual pod** based on observed usage history. They operate on orthogonal axes — "how many" versus "how big each one is."
2. Almost always a missing `resources.requests.cpu` on the target container(s) — HPA's CPU utilization percentage is calculated relative to the pod's CPU request, and with no request set there's no baseline to compute a percentage against, so the current metric shows as unknown and no scaling decision can be made.
3. Because VPA raising the CPU request changes the denominator HPA uses to compute utilization percentage, which can suppress HPA's scale-up signal even under genuinely higher absolute load (or vice versa) — the two controllers' independent adjustments interact in a feedback loop rather than complementing each other cleanly. The safer pattern is HPA active on the metric while VPA runs in `Off`/recommendation-only mode.
4. Because CPU utilization simply isn't correlated with the real demand signal for an I/O-bound consumer — it can be nearly idle on CPU while a queue backlog grows into the tens of thousands of unprocessed messages, and HPA has no visibility into that backlog at all unless it's explicitly fed as a custom/external metric (which is exactly the gap KEDA fills).
5. KEDA does not replace HPA — when you create a `ScaledObject`, KEDA's operator automatically creates and manages a real Kubernetes HPA object under the hood, acting as a custom/external metrics API server that feeds that HPA the value from whichever trigger (SQS, RabbitMQ, Kafka, cron, etc.) you configured. It's a metrics adapter and control layer on top of vanilla HPA, not a competing autoscaling engine.
6. **Scale-to-zero** — KEDA supports `minReplicaCount: 0`, letting a Deployment scale all the way down to zero running pods when there's no work, then scale back up when work appears. Plain Kubernetes HPA's `minReplicas` cannot go below 1, structurally.
7. `pollingInterval` is how often (in seconds) KEDA checks the external metric source (e.g., queries SQS for current queue depth). `cooldownPeriod` is how long the metric must indicate low/no activity before KEDA scales the workload back down to `minReplicaCount` — it's the KEDA-level equivalent of HPA's `scaleDown.stabilizationWindowSeconds`.
8. Cluster Autoscaler scales pre-defined node groups/Auto Scaling Groups by adjusting their desired capacity via the ASG API — it picks *which existing group* to scale, not the instance type itself. Karpenter has no node-group concept at all; it evaluates pending pods' exact requirements directly and calls the cloud API (EC2) to launch a dynamically chosen, best-fit instance type and size for that specific workload.
9. Cluster Autoscaler can only satisfy pending pods whose resource shape matches one of its pre-configured node groups — if a pod needs an instance shape (e.g., unusual memory-to-CPU ratio, a specific instance family) that doesn't match any existing node group, CA cannot help regardless of how long it waits, and operators typically end up maintaining a sprawling set of node groups to cover every anticipated shape. Karpenter avoids this by choosing from the entire allowed instance-type universe per pending pod, with no pre-provisioned groups required.
10. Because the cost of under-provisioning (dropped requests, latency spikes during a real load spike) is generally worse than the cost of over-provisioning slightly longer than strictly necessary (extra spend, but no user-facing impact) — so every layer is deliberately designed to react immediately to "need more" signals while requiring sustained, confirmed evidence before reacting to "need less" signals, to avoid removing capacity prematurely and then immediately needing it back (flapping).
11. A node autoscaler reacts to pods that are **unschedulable** — a `FailedScheduling` event with a reason like `Insufficient cpu`/`Insufficient memory`, meaning no existing node has room. HPA/KEDA react to a **metric crossing a target threshold** (CPU utilization, queue depth, etc.) to decide how many replicas *should* exist in the first place — a completely earlier and different decision point in the chain (HPA/KEDA decide "we need more pods"; the node autoscaler reacts only once those pods can't be scheduled anywhere).
12. Strong answer: "I'd use KEDA with an `aws-sqs-queue` trigger on a `ScaledObject`, targeting the consumer Deployment, with `queueLength` set to the number of messages one replica can comfortably process concurrently, and `minReplicaCount` potentially at 0 if the workload can tolerate cold-start latency and the queue is genuinely idle sometimes. Under the hood, KEDA creates and manages a real HPA fed by the queue depth as an external metric, rather than CPU/memory. The tradeoff versus plain HPA is that HPA alone can't see queue depth at all without a custom/external metrics adapter — CPU utilization for an I/O-bound consumer doesn't correlate with backlog size, so HPA would never scale appropriately on its own. KEDA's tradeoffs are the added authentication surface (I'd back it with IRSA rather than static AWS keys), tuning `pollingInterval`/`cooldownPeriod` against the queue's real arrival pattern to avoid flapping, and — if using scale-to-zero — accepting the cold-start latency for the first message after an idle period. I'd also make sure a node-level autoscaler like Karpenter is in place so the extra replicas KEDA/HPA request actually have node capacity to land on."
