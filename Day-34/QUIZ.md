# Day 34 — Quiz: AWS Compute (EKS, ECS, Lambda)

Try to answer without looking at your notes. Answers are at the bottom.

1. What does EKS's managed control plane actually take off your plate, and what does it explicitly NOT decide for you?
2. What's the difference between self-managed nodes and managed node groups in EKS? What real operational task does a managed node group still leave to you?
3. Why can't you run a DaemonSet on EKS Fargate?
4. Why might Fargate end up *more* expensive than a managed node group at high, steady utilization, despite feeling "serverless-simple"?
5. Name three concrete factors that should drive an ECS vs. EKS decision, beyond "Kubernetes is more powerful."
6. What happens during a Lambda cold start, specifically, before your handler code executes?
7. Why should you initialize a DB connection or SDK client outside the handler function rather than inside it?
8. What does provisioned concurrency actually do, and for what class of workload is it usually NOT worth the cost?
9. What is a Lambda layer, and what's the tradeoff it doesn't remove (i.e., what limit does it still count against)?
10. How does EventBridge's routing model differ from SQS's, and give one scenario where you'd prefer each.
11. Why is a `REPORT` log line's `Init Duration` field a reliable way to detect whether a specific invocation was a cold start?
12. **Interview question:** When would you choose ECS over EKS? What are the operational tradeoffs?

---

## Answers

1. EKS's managed control plane takes over running, scaling, patching, and multi-AZ availability of the API server, etcd, scheduler, and controller manager — you never operate these directly. It explicitly does NOT decide how worker nodes are provisioned/managed — that's a separate choice among self-managed nodes, managed node groups, and Fargate.
2. Self-managed nodes require you to provision the EC2 instances, install/join the correct kubelet and container runtime, and personally handle OS patching and coordinated version upgrades. Managed node groups have AWS provision/manage the underlying Auto Scaling Group using EKS-optimized AMIs, and automate the drain-and-replace upgrade workflow respecting Pod Disruption Budgets. What's still left to you with managed node groups: choosing/right-sizing instance types, capacity planning (including Spot vs. On-Demand mix), and setting up autoscaling (cluster autoscaler/Karpenter) — AWS automates lifecycle operations, not capacity decisions.
3. A DaemonSet requires a persistent node it can run one copy of itself on per node. Fargate has no persistent, visible node — each pod gets its own isolated micro-VM provisioned on demand — so there's no stable node target for a DaemonSet to schedule onto.
4. Fargate bills per-pod based on requested vCPU/memory, and can't share a machine's spare capacity across many small pods the way node-level bin-packing does on a managed node group. At high, steady, well-utilized density, a densely bin-packed set of EC2 nodes is often cheaper than paying for each pod's isolated Fargate allocation individually.
5. Any three of: whether there's a genuine multi-cloud/hybrid requirement (favors EKS); whether the team already has Kubernetes expertise (reduces EKS's learning-curve cost); whether a specific CNCF/Kubernetes-ecosystem tool (Helm chart, operator, service mesh) is actually needed (favors EKS); team size/availability of dedicated platform engineering effort (favors ECS if small); whether the workload is a standard "run containers behind a load balancer" case that doesn't need Kubernetes's extensibility (favors ECS).
6. AWS downloads the deployment package/image, starts a new Firecracker micro-VM, initializes the language runtime, and then runs any top-level/init code in your function file (SDK client construction, config loading, DB connection setup) — all of this happens before your handler function itself starts executing.
7. Code outside the handler (at module/top level) runs once per cold start and is reused across all subsequent warm invocations of that same execution environment, avoiding the cost of re-establishing a DB connection or reconstructing an SDK client on every single invocation. Code inside the handler runs fresh on every invocation regardless of warm/cold status.
8. Provisioned concurrency pre-initializes and continuously keeps warm a specified number of execution environments, so invocations routed to them skip the cold-start sequence entirely — at the cost of paying for that idle capacity even when there's no traffic. It's usually not worth it for asynchronous/batch workloads (like an SQS-triggered processor) where occasional extra cold-start latency has no user-facing impact; it's worth it mainly for latency-sensitive synchronous endpoints.
9. A layer is a versioned ZIP of shared code/dependencies mounted into a function's execution environment at `/opt`, letting multiple functions reference the same shared code without each bundling its own private copy — keeping individual deployment packages smaller. It still counts toward Lambda's combined function-code-plus-layers unzipped size limit (250MB) — layers reduce duplication, they don't grant extra size headroom beyond that shared limit.
10. EventBridge routes events via content-based pattern-matching rules to potentially multiple targets/consumers, well suited when several unrelated consumers need to react differently to the same class of event, or when reacting to native AWS service state changes. SQS is a simpler point-to-point queue with per-message retry/DLQ semantics, well suited for a single (or a small known set of) consumer processing a stream of your own application's messages reliably. Prefer EventBridge for "many different things might care about this event type"; prefer SQS for "I need a durable, retryable buffer between a producer and a specific consumer."
11. The `Init Duration` field only appears in the `REPORT` log line when AWS had to provision and initialize a fresh execution environment for that specific invocation — a warm invocation reuses an already-initialized environment and skips that step entirely, so the field is simply absent from its `REPORT` line, making its presence/absence a direct, per-invocation signal of cold vs. warm.
12. Strong answer: "I'd choose ECS when the team is AWS-only with no near-term multi-cloud or hybrid requirement, when there's no existing deep Kubernetes expertise on the team, and when the workload is a standard 'run stateless containers behind a load balancer' case that doesn't need Kubernetes-specific ecosystem tools like Helm charts, operators, or a service mesh. The operational tradeoff is real: ECS has a much smaller conceptual surface (Task Definitions, Services, Clusters, no separate control plane to pay for or reason about) and a small team can run it reliably without a dedicated Kubernetes specialist, at the cost of being AWS-only with no portability and a smaller ecosystem. EKS pays off specifically when you need genuine multi-cloud portability, a specific CNCF-ecosystem tool with no ECS equivalent, or when the team already has sunk Kubernetes expertise that makes the learning-curve cost effectively zero."
