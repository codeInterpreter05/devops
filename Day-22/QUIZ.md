# Day 22 — Quiz: K8s Architecture Deep Dive

Try to answer without looking at your notes. Answers are at the bottom.

1. Which component is the *only* one that talks directly to `etcd`? What does everything else do instead?
2. Why do production `etcd` clusters almost always have an odd number of members (3 or 5), never an even number?
3. What exactly does `kube-scheduler` do — and what does it explicitly *not* do?
4. Name three examples of controllers bundled inside `kube-controller-manager`, and describe the general pattern they all follow.
5. What is a "static pod," and how does the kubelet start one differently from a normal pod?
6. Since Kubernetes 1.24, what happened to Docker Engine as a supported container runtime, and what do clusters use instead?
7. What is the difference between what `kube-proxy` does and what a CNI plugin does? Why is this distinction important when debugging connectivity?
8. What is the difference between a failing liveness probe and a failing readiness probe, in terms of what the kubelet actually does?
9. Are Kubernetes Secrets encrypted at rest by default? What do you need to configure if not, and why does it matter?
10. What is `resourceVersion` used for, and what HTTP status code do you get if you try to update an object using a stale one?
11. If the entire control plane goes down for 10 minutes, what happens to Pods that were already running and scheduled? What stops working?
12. **Interview question:** Walk me through exactly what happens when you run `kubectl apply -f pod.yaml`.

---

## Answers

1. `kube-apiserver` is the only component that reads/writes `etcd` directly. Every other component (scheduler, controllers, kubelets) interacts only with the apiserver, typically via long-lived **watch** connections, never with `etcd` itself.
2. Raft consensus requires a majority (quorum) to commit writes: for `N` members you can tolerate `(N-1)/2` failures. Going from 3 to 4 members doesn't increase the failures tolerated (still 1) but does increase the number of votes needed for quorum — an even number adds cost without adding resilience, so odd numbers are always preferred.
3. The scheduler only **decides which node a Pod should run on** — it runs Filter (eliminate infeasible nodes) then Score (rank the rest) and writes the chosen node name back via a `Binding` API call. It never pulls images, starts containers, or talks to nodes directly — that's entirely the kubelet's job.
4. Examples: the Deployment controller (manages ReplicaSets to match desired Deployment state), the ReplicaSet controller (creates/deletes Pods to match `replicas:`), the Node controller (marks nodes NotReady/evicts pods after timeouts), the Endpoint/EndpointSlice controller (keeps Service backend lists in sync with matching Pods). All follow the same level-triggered "observe current state -> compare to desired state -> act to close the gap" reconciliation loop, running continuously, not just once on a change event.
5. A static pod is a pod the kubelet starts directly from a manifest file in a local directory (default `/etc/kubernetes/manifests`), completely bypassing the apiserver — no scheduler decision, no `etcd` write initiated by anyone else. This is how self-hosted control planes (`kubeadm`, minikube) bootstrap their own `kube-apiserver`/`etcd`/`kube-scheduler`/`kube-controller-manager`, solving the chicken-and-egg problem of "how do you schedule the scheduler."
6. `dockershim` (the shim that let the kubelet talk to Docker Engine via CRI) was removed in Kubernetes 1.24. Clusters now use CRI-native runtimes directly: `containerd` (most common) or `CRI-O`. Docker Engine builds/images still work fine (they're OCI-compliant), but the kubelet no longer manages containers *through Docker itself*.
7. `kube-proxy` implements the **Service** abstraction — routing a virtual ClusterIP to one of several backend Pod IPs, via iptables or IPVS rules. The CNI plugin (Calico, Cilium, AWS VPC CNI, etc.) is responsible for the more fundamental problem of giving every Pod a real, routable IP address and making pod-to-pod traffic work across nodes. If pod-to-pod connectivity is broken, that's a CNI problem; if Service routing specifically is broken but direct pod-to-pod `curl` works, that's a kube-proxy problem — conflating the two wastes debugging time.
8. A failing **liveness** probe causes the kubelet to **restart the container** (it assumes the process is stuck/deadlocked). A failing **readiness** probe causes the kubelet/Endpoint controller to **remove the pod from Service endpoints** — the container keeps running, it's just temporarily taken out of load-balancing rotation until it passes again. No restart happens for a readiness failure.
9. No — by default Secrets are stored in `etcd` as base64, which is an encoding, not encryption; anyone with access to raw `etcd` data or an unencrypted snapshot can read them trivially. To fix this, configure an `EncryptionConfiguration` with a KMS provider (envelope encryption) so the apiserver encrypts Secret values before writing to `etcd` — on EKS this is the "secrets encryption" option using a KMS key.
10. `resourceVersion` is a monotonically increasing marker on every Kubernetes object that enables optimistic concurrency control and efficient watches (a watch is "give me all changes since resourceVersion X"). If you try to update an object using a stale `resourceVersion` (someone else updated it in the meantime), the apiserver rejects your request with `409 Conflict` rather than silently overwriting the other change.
11. Already-running Pods keep running and keep serving traffic — the kubelet on each node doesn't need the control plane to keep an already-started container alive, and existing `iptables`/IPVS routing rules keep working. What stops: scheduling new Pods, any `kubectl apply`/`get`/`edit` operations, autoscaling reactions, and any reconciliation that requires a fresh decision (e.g., a crashed pod on a healthy node *can* still be restarted locally by the kubelet, but a pod on a node that goes down cannot be rescheduled elsewhere until the control plane recovers).
12. Strong answer: "`kubectl` sends an authenticated HTTPS request to the apiserver, which runs authentication, then RBAC authorization, then mutating and validating admission webhooks, then persists the object to etcd with `Pending` status and no assigned node. The scheduler, which watches for unscheduled pods, runs its filter-then-score algorithm and writes back a Binding assigning a node — this write also goes through the apiserver to etcd. The kubelet on that node, watching for pods bound to itself, then pulls the image via the CRI-compliant runtime (containerd), sets up the pod sandbox and any volumes, starts the containers respecting init-container ordering, and continuously reports status back to the apiserver. Once the pod passes its readiness probe, the Endpoint controller adds it to any matching Service's endpoints, and kube-proxy on every node updates its routing rules so traffic can reach it. The whole thing is asynchronous and watch-driven — `kubectl apply` returning success only means the object was accepted and stored, not that the pod is running yet."
