# Day 49 — Quiz: AWS Security Services

Try to answer without looking at your notes. Answers are at the bottom.

1. Does GuardDuty require an agent installed on your EC2 instances for its core detection capability? What data sources does it analyze instead?
2. What does the GuardDuty finding naming pattern `ThreatPurpose:ResourceType/ThreatFamilyName` tell you about a finding like `CryptoCurrency:EC2/BitcoinTool.B!DNS` even if you've never seen it before?
3. What is the actual relationship between GuardDuty and Security Hub — does Security Hub detect its own threats?
4. What is ASFF and why does it matter that Security Hub normalizes findings into it?
5. Why should you always deploy a new WAF rule in `Count` mode before switching it to `Block`?
6. What's the concrete difference between Shield Standard and Shield Advanced — what do you actually get by paying for Advanced?
7. What does Macie scan for, and give an example of the kind of real-world incident it's designed to catch before an external party does.
8. What's the difference between a CloudTrail management event and a data event? Which one is enabled by default?
9. Why is a single-region, single-account CloudTrail trail considered a weak security posture?
10. Explain envelope encryption in KMS: why doesn't KMS directly encrypt a 10GB file with your CMK?
11. What's the difference between an AWS managed key and a customer managed key (CMK), and name two things you can only do with a CMK?
12. **Interview question:** How would you detect and respond to a compromised EC2 instance automatically?

---

## Answers

1. No — GuardDuty's core detection is agentless. It analyzes VPC Flow Log metadata, DNS query logs, CloudTrail management/data events, and EKS audit logs that AWS/your account already generate. (Optional add-ons like Malware Protection and Runtime Monitoring do use lightweight agents/scanners, but the baseline threat detection does not.)
2. Even without recognizing the specific finding, the naming pattern tells you: this is about cryptocurrency-related malicious purpose (`CryptoCurrency`), affecting an EC2 instance (`EC2`), detected via a known "BitcoinTool" malware family signature observed through DNS query analysis (`BitcoinTool.B!DNS`) — i.e., the instance is likely resolving domains associated with cryptomining infrastructure.
3. No. Security Hub does not detect threats itself — it's an aggregation and normalization layer that ingests findings from GuardDuty, Macie, Inspector, IAM Access Analyzer, Firewall Manager, and third-party tools, and additionally runs automated compliance checks (CIS, FSBP, PCI DSS, NIST) on your account configuration.
4. ASFF (AWS Security Finding Format) is the normalized JSON schema Security Hub converts every ingested finding into. It matters because it lets one dashboard, one set of EventBridge rules, and one set of downstream integrations (ticketing, SIEM) work uniformly across findings from completely different source products, instead of needing custom parsing logic per source.
5. Because a new rule (especially pattern/regex-based ones like SQLi detection) can false-positive on legitimate traffic. `Count` mode logs what *would* have been blocked without actually blocking it, letting you measure the false-positive rate against real production traffic before risking an outage by switching to `Block`.
6. Shield Standard is free, automatic on every account, and covers common network/transport-layer (L3/L4) DDoS attacks. Shield Advanced is a paid subscription adding: stronger detection for larger/more sophisticated attacks (including some Layer 7 coverage when paired with WAF), 24/7 access to the AWS DDoS Response Team (DRT), and cost protection credits for scaling costs incurred during an attack.
7. Macie scans S3 object contents (using managed + custom data identifiers) for sensitive data like PII, credit card numbers, SSNs, and accidentally-committed credentials, and also flags bucket-level exposure risk (public access, missing encryption). Example: catching an S3 bucket that was accidentally made public and contains a database backup with customer PII in it — before an external researcher or attacker discovers it.
8. Management events are control-plane API calls (creating/modifying/deleting resources, e.g. `RunInstances`, `CreateBucket`) and are enabled by default with 90 days of free Event history. Data events are high-volume data-plane operations (S3 `GetObject`/`PutObject`, Lambda `Invoke`) and require you to explicitly configure a trail to capture them — they're off by default due to volume/cost.
9. A single-region trail misses activity if an attacker (or a misconfigured automation) operates in a different, unmonitored region. A single-account trail with logs stored in the same account can be tampered with or deleted by an attacker who gains sufficient privilege in that account — best practice is a multi-region, organization-wide trail with logs shipped to a separate, locked-down log-archive account.
10. Because KMS's HSM boundary is designed to handle small cryptographic material (keys), not bulk data throughput. Envelope encryption has KMS generate a data key (plaintext + CMK-encrypted versions); the calling service encrypts the actual (possibly huge) data locally using the plaintext data key, then discards it and stores only the small encrypted data key alongside the ciphertext. Decryption sends just the small encrypted data key back to KMS to unwrap.
11. AWS managed keys (`aws/service` aliases) are created automatically, free, auto-rotated yearly with no control, and you cannot edit their key policy or share them cross-account. CMKs cost a monthly fee, and only with a CMK can you: (a) share the key cross-account via its key policy, and (b) schedule the key for deletion (or fully control its rotation and policy).
12. Strong answer: "GuardDuty continuously analyzes VPC Flow Logs, DNS logs, and CloudTrail events with no agent required, and would flag anomalous behavior like C2 beaconing or brute-force attempts as a finding with a severity score. That finding is automatically published to the default EventBridge bus. An EventBridge rule filtering on severity routes it to two targets: an SNS topic that pages the on-call engineer, and a Lambda function that performs automated containment — for example, swapping the instance's security group to an isolation SG with no egress except to a forensics subnet, snapshotting its EBS volume for analysis, and tagging it as quarantined. Security Hub gives the aggregated, cross-account view and lets a human trigger further action via a custom action if the automation didn't fully resolve it. I'd gate any destructive automation behind a severity/confidence threshold so a Low-severity false positive doesn't cause a self-inflicted outage."
