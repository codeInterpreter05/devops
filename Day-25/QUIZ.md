## Answers

1. Referencing a PVC instead of a PV decouples the Pod spec from any specific underlying storage implementation â€” the same Pod YAML works whether the PVC ends up bound to an EBS volume, a local disk, or NFS, because the Pod only ever needs to know "give me the storage this claim represents," not which physical/cloud resource satisfies it.
2. Nothing â€” deleting a Pod never touches its PVC or the PVC's underlying data. The PVC remains `Bound` and the data is untouched; a new Pod referencing the same PVC will see exactly the same data. This is the entire reason PVCs exist as a separate lifecycle from Pods.
3. The **`persistentVolumeReclaimPolicy`** set on the bound PersistentVolume (not the PVC). `Delete` policy destroys the underlying storage resource when the PVC is deleted (common default for dynamically-provisioned storage). `Retain` policy preserves the data â€” the PV moves to `Released` status and requires manual admin cleanup before reuse.
4. `Released` means the bound PVC was deleted but the PV (and its data) was preserved per a `Retain` reclaim policy â€” it is **not** automatically reusable because it still carries a reference (`claimRef`) to the old, now-deleted PVC, and Kubernetes won't silently repurpose storage that might contain someone's important leftover data. An admin must manually intervene (e.g., clear `claimRef` or recreate the PV) before it can move to `Available` and be bound to a new PVC.
5. It delays actual volume provisioning until a Pod using the PVC is scheduled, so the storage gets created in the **same availability zone** the scheduler already picked for the Pod. Without it (`Immediate` binding mode), a zonal volume like EBS could be provisioned in a different AZ than where the Pod ends up scheduled, leaving the Pod permanently unable to mount its volume (a `volume node affinity conflict`).
6. `ReadWriteOnce` (RWO): read-write by one node at a time (multiple pods on that same node can share it since Kubernetes 1.22+). `ReadOnlyMany` (ROX): read-only by many nodes simultaneously. `ReadWriteMany` (RWX): read-write by many nodes simultaneously â€” requires a network filesystem (EFS/NFS/Azure Files); block storage like **EBS can never support RWX** because an EBS volume can only be attached to a single EC2 instance at a time, fundamentally. `ReadWriteOncePod` (RWOP): the strictest â€” only one single Pod cluster-wide can mount it read-write, even stricter than "one node."
7. A **controller plugin** (usually a Deployment) that talks to the cloud provider's API to create/delete/attach/detach volumes, and a **node plugin** (usually a DaemonSet, one per node) that handles the actual local mount/unmount/formatting operations the kubelet delegates to it on each specific node.
8. `emptyDir` provides scratch storage created fresh when a Pod is scheduled and permanently deleted when the Pod is removed from the node â€” useful for temporary cache/processing space and for sharing files between containers within the same Pod. Its two key limitations versus a PVC: (1) data does not survive Pod deletion at all, and (2) it's not portable/independent â€” it only exists as long as that specific Pod instance exists, with no separate lifecycle or binding model.
9. `hostPath` mounts the actual underlying node's filesystem into a Pod. A compromised or misconfigured container with a `hostPath` mount (especially a broad or writable one) has a direct path to reading or modifying files on the node's own operating system â€” a classic container-escape/node-compromise vector â€” which is why Pod Security Standards restrict or forbid it, rather than because of persistence concerns (which it also handles poorly, since the data is tied to one specific node, not portable across reschedules).
10. `VolumeSnapshotClass` (like a StorageClass, names the CSI driver/snapshot parameters), `VolumeSnapshot` (namespaced request referencing a source PVC), and `VolumeSnapshotContent` (cluster-scoped, analogous to a PV, representing the actual snapshot). Restoring never modifies the original PVC â€” you create a **brand-new** PVC with a `dataSource` referencing the `VolumeSnapshot`, which provisions a new volume pre-populated from the snapshot's data.
11. Missing or insufficient **IAM permissions** for the EBS CSI controller's ServiceAccount (typically via IRSA) â€” the controller pod can't call the AWS EBS API to actually create the volume, resulting in an `AccessDenied`-type error visible in `kubectl describe pvc` events, which looks like a Kubernetes storage problem but is actually an AWS IAM configuration gap.
12. Strong answer: "Deleting a Pod doesn't touch the PVC at all â€” the claim stays bound and the data is untouched, which is exactly why a new Pod referencing the same PVC sees the same data; this is how StatefulSet pods keep their identity's data across restarts. Deleting the PVC itself is different: what happens next depends on the bound PersistentVolume's reclaim policy. With `Delete` (the common default for dynamically-provisioned storage), the underlying cloud volume is destroyed along with the PVC, generally irreversibly unless you have a snapshot. With `Retain`, the PV moves to `Released` status, keeping the data intact but requiring manual admin cleanup before that storage can be reused by anything else. So the operative rule is: Pod deletion is always safe for data, PVC deletion's safety depends entirely on the reclaim policy configured on the PV."
