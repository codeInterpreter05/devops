# Day 25 — Lab: Storage in K8s

**Goal:** Provision real persistent storage dynamically, prove data survives Pod deletion but not PVC deletion (with `Delete` policy), and explore `emptyDir`/`hostPath`/snapshots hands-on.

**Prerequisites:** A running cluster. This lab is written primarily for **EKS with the AWS EBS CSI driver** (matching the assigned hands-on activity), with a minikube-equivalent noted for each step so you can practice locally first before touching a real cloud account.

```bash
# EKS-specific prerequisite (skip if practicing on minikube first)
eksctl create addon --cluster <your-cluster> --name aws-ebs-csi-driver \
  --service-account-role-arn arn:aws:iam::<account-id>:role/AmazonEKS_EBS_CSI_DriverRole
kubectl get pods -n kube-system -l app.kubernetes.io/name=aws-ebs-csi-driver
```

---

### Lab 1 — StorageClass and dynamic provisioning basics

1. Inspect the default StorageClass (minikube ships one called `standard`; EKS with the EBS add-on typically needs one created):
   ```bash
   kubectl get storageclass
   ```
   On EKS, if none exists yet:
   ```bash
   cat <<'EOF' | kubectl apply -f -
   apiVersion: storage.k8s.io/v1
   kind: StorageClass
   metadata:
     name: gp3
     annotations: { storageclass.kubernetes.io/is-default-class: "true" }
   provisioner: ebs.csi.aws.com
   parameters: { type: gp3, encrypted: "true" }
   volumeBindingMode: WaitForFirstConsumer
   allowVolumeExpansion: true
   EOF
   ```
2. Create a PVC and watch it bind:
   ```bash
   cat <<'EOF' | kubectl apply -f -
   apiVersion: v1
   kind: PersistentVolumeClaim
   metadata: { name: data-pvc }
   spec:
     accessModes: ["ReadWriteOnce"]
     resources: { requests: { storage: 5Gi } }
   EOF
   kubectl get pvc data-pvc -w      # Pending -> Bound (may stay Pending until a Pod consumes it, if WaitForFirstConsumer)
   ```
3. Confirm the PV that was auto-created:
   ```bash
   kubectl get pv
   kubectl describe pv <auto-generated-name>
   ```

**Success criteria:** You've created a PVC with no manually pre-created PV, and confirmed a real PV (and, on EKS, a real EBS volume in the AWS console) was created automatically by the CSI driver.

### Lab 2 — The core hands-on activity: write data, delete pod, verify persistence

This is the assigned hands-on activity — do it for real.

1. Start a Pod using the PVC and write a marker file:
   ```bash
   cat <<'EOF' | kubectl apply -f -
   apiVersion: v1
   kind: Pod
   metadata: { name: writer }
   spec:
     containers:
       - name: app
         image: busybox:1.36
         command: ["sh", "-c", "sleep 3600"]
         volumeMounts: [{ name: data, mountPath: /data }]
     volumes:
       - name: data
         persistentVolumeClaim: { claimName: data-pvc }
   EOF
   kubectl wait --for=condition=Ready pod/writer --timeout=120s
   kubectl exec writer -- sh -c 'echo "persisted-at-$(date -u +%FT%TZ)" > /data/marker.txt'
   kubectl exec writer -- cat /data/marker.txt
   ```
2. Delete the Pod (not the PVC) and start a fresh Pod on the same PVC:
   ```bash
   kubectl delete pod writer
   kubectl apply -f - <<'EOF'
   apiVersion: v1
   kind: Pod
   metadata: { name: reader }
   spec:
     containers:
       - name: app
         image: busybox:1.36
         command: ["sh", "-c", "sleep 3600"]
         volumeMounts: [{ name: data, mountPath: /data }]
     volumes:
       - name: data
         persistentVolumeClaim: { claimName: data-pvc }
   EOF
   kubectl wait --for=condition=Ready pod/reader --timeout=120s
   kubectl exec reader -- cat /data/marker.txt      # should show the SAME marker written earlier
   ```
3. Now delete the PVC itself and observe the reclaim-policy behavior:
   ```bash
   kubectl delete pod reader
   kubectl get pv <pv-name> -o jsonpath='{.spec.persistentVolumeReclaimPolicy}{"\n"}'
   kubectl delete pvc data-pvc
   kubectl get pv     # Delete policy -> PV is gone; Retain policy -> PV shows Released
   ```

**Success criteria:** You've proven, with your own command output, that deleting a Pod leaves data intact (new Pod on the same PVC reads the same file), and that deleting the PVC triggers the underlying PV's reclaim policy — either destroying the volume (`Delete`) or leaving it `Released` for manual cleanup (`Retain`).

### Lab 3 — Access modes, `emptyDir`, and `hostPath`

1. Try requesting `ReadWriteMany` on the EBS-backed StorageClass and watch it fail (EBS cannot do RWX):
   ```bash
   cat <<'EOF' | kubectl apply -f -
   apiVersion: v1
   kind: PersistentVolumeClaim
   metadata: { name: rwx-test }
   spec:
     accessModes: ["ReadWriteMany"]
     storageClassName: gp3
     resources: { requests: { storage: 1Gi } }
   EOF
   kubectl describe pvc rwx-test | tail -10    # expect a provisioning failure/event explaining the mismatch
   kubectl delete pvc rwx-test
   ```
2. Demonstrate `emptyDir` sharing between two containers in one Pod:
   ```bash
   cat <<'EOF' | kubectl apply -f -
   apiVersion: v1
   kind: Pod
   metadata: { name: shared-scratch }
   spec:
     containers:
       - name: writer
         image: busybox:1.36
         command: ["sh", "-c", "echo hello from writer > /scratch/msg.txt; sleep 3600"]
         volumeMounts: [{ name: scratch, mountPath: /scratch }]
       - name: reader
         image: busybox:1.36
         command: ["sh", "-c", "sleep 5; cat /scratch/msg.txt; sleep 3600"]
         volumeMounts: [{ name: scratch, mountPath: /scratch }]
     volumes:
       - name: scratch
         emptyDir: { sizeLimit: 100Mi }
   EOF
   kubectl logs shared-scratch -c reader
   ```

**Success criteria:** You've seen an RWX request against block storage fail with a clear reason, and confirmed two containers in the same Pod can read/write the same `emptyDir` volume.

### Cleanup

```bash
kubectl delete pod writer reader shared-scratch --ignore-not-found
kubectl delete pvc data-pvc rwx-test --ignore-not-found
kubectl get pv    # manually delete any left in Released status if reclaim policy was Retain
```

### Stretch challenge

If your CSI driver supports snapshots (EBS CSI driver does, with the snapshot controller installed), create a `VolumeSnapshot` of `data-pvc` before deleting it, then restore it into a brand-new PVC via `dataSource`, and confirm `marker.txt` is present in the restored volume without ever touching the original PVC.
