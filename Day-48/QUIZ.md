# Day 48 — Quiz: EKS Deep Dive

Try to answer without looking at your notes. Answers are at the bottom.

1. What's the difference between a self-managed node group and a Managed Node Group in EKS?
2. Name three capabilities a pod loses when scheduled onto Fargate instead of EC2-based nodes.
3. What are the three core EKS add-ons every cluster needs, and which one runs as a Deployment rather than a DaemonSet?
4. Fundamentally, how does AWS VPC CNI's pod networking model differ from a typical overlay CNI plugin like Calico or Flannel?
5. What real, distinct scaling constraint does VPC CNI's design introduce, and what are the two standard mitigations?
6. Why can ALB "IP target mode" route directly to pod IPs, and what hop does that skip compared to "instance target mode"?
7. What is the fundamental architectural limitation of Cluster Autoscaler that Karpenter was built to remove?
8. What does Karpenter's "consolidation" feature do that Cluster Autoscaler does not do at all?
9. What's the difference between EKS Distro and EKS Anywhere?
10. What was the core operational problem with the old `aws-auth` ConfigMap approach to cluster access, and how do Access Entries solve the "chicken-and-egg" bootstrapping issue specifically?
11. What does setting a cluster's authentication mode to `API_AND_CONFIG_MAP` allow you to do, and why would you use it during a migration?
12. **Interview question:** Explain the EKS networking model. How does the VPC CNI differ from other CNI plugins?

---

## Answers

1. A self-managed node group is a plain EC2 Auto Scaling Group you configure, patch, and bootstrap entirely yourself (custom launch template, AMI choice, bootstrap script). A Managed Node Group is AWS-orchestrated — AWS handles AMI selection/updates, graceful draining during updates, and lifecycle integration with the EKS control plane — while you still choose instance types and scaling bounds.
2. Any three of: DaemonSets (no shared node to run them on), `hostNetwork`/`hostPort`/privileged pods (each pod is isolated in its own micro-VM), GPU-backed workloads (unsupported on Fargate for EKS).
3. CoreDNS (cluster DNS), kube-proxy (Service-to-pod routing via iptables/IPVS), and VPC CNI (pod IP assignment). CoreDNS runs as a Deployment (shared cluster service); kube-proxy and VPC CNI's node agent run as DaemonSets (per-node).
4. VPC CNI assigns pods real, routable IP addresses directly from the VPC's own CIDR block (via ENIs and secondary IPs attached to worker nodes) — the same address space as EC2 instances and everything else in the VPC. Overlay CNIs like Calico/Flannel create a separate, virtual pod network with its own CIDR that doesn't exist in the underlying physical network, encapsulating (e.g., VXLAN) traffic between pods on different nodes and decapsulating it at the destination.
5. Pod density per node is bounded by ENI/secondary-IP limits (instance-type dependent) rather than purely by CPU/memory — a node can have plenty of free compute but still be unable to schedule more pods due to IP exhaustion. The two mitigations are prefix delegation (assigning a /28 prefix, 16 IPs, per ENI slot instead of one IP, multiplying effective density) and custom networking (assigning pod IPs from a secondary, non-primary CIDR to avoid exhausting the VPC's primary address space).
6. Because VPC CNI gives pods real, directly routable VPC IP addresses, a load balancer can target pod IPs directly. This skips the NodePort/kube-proxy iptables redirection hop that instance target mode requires (where traffic hits a node's NodePort first, then gets redirected to a pod, possibly on a different node), reducing latency and avoiding an extra network hop.
7. Cluster Autoscaler can only scale existing, pre-defined node groups/ASGs up or down — it cannot choose a different instance type on the fly; you must pre-define node groups for every instance-type/AZ combination you might need, leading to either over-provisioned node group sprawl or pods left Pending when no predefined group fits an unusual resource request.
8. Consolidation proactively and continuously looks for opportunities to replace multiple underutilized nodes with fewer, better-fitting ones, or remove empty nodes entirely, as an ongoing bin-packing optimization. Cluster Autoscaler only reactively scales down (removing clearly underutilized nodes within existing groups) — it doesn't proactively repack workloads onto more efficient node combinations.
9. EKS Distro (EKS-D) is AWS's open-source Kubernetes distribution — the actual tested components (Kubernetes, etcd, CoreDNS, etc.) AWS uses internally for managed EKS, released independently as binaries/images. EKS Anywhere (EKS-A) is the deployment/lifecycle-management tooling built on top of EKS-D that lets you actually run a full cluster using those components on your own infrastructure (bare metal, vSphere, Snow, Docker, CloudStack).
10. The `aws-auth` ConfigMap was a manually-edited, unvalidated Kubernetes object with no clean audit trail or Terraform-native management, and a real risk that a malformed edit could lock out cluster access entirely. It also created a bootstrapping problem: only someone already granted Kubernetes access could edit it to grant access to anyone else. Access Entries solve this because they're a first-class EKS API operation (external to the cluster, gated by IAM permissions like `eks:CreateAccessEntry`) rather than an in-cluster object edit — the cluster creator can be granted admin access automatically at cluster-creation time via the EKS API, with no prior Kubernetes-level access required.
11. It runs both the old `aws-auth` ConfigMap and the new Access Entries API simultaneously, so existing IAM-to-RBAC mappings configured via `aws-auth` keep working while you incrementally recreate them as Access Entries — avoiding a risky big-bang cutover where you'd have to migrate every mapping at once before anyone could lose access.
12. Strong answer: "Every EKS node runs the VPC CNI plugin, which assigns pods real IP addresses directly from the VPC's CIDR block using ENIs and their secondary IP addresses attached to each worker node — the pod network is the VPC network, not a separate one. This differs fundamentally from typical CNI plugins like Calico or Flannel in overlay mode, which create a virtual pod network with its own CIDR, invisible to the underlying infrastructure, and encapsulate traffic between nodes (e.g., via VXLAN) to move it across the real network. VPC CNI's approach means pods are natively routable everywhere in the VPC with no encapsulation overhead, and tools like VPC Flow Logs and Security Groups for Pods can operate on pod traffic directly — but it also means pod density per node is limited by ENI and secondary-IP capacity rather than just compute resources, which is why AWS added prefix delegation and custom networking as mitigations for IP exhaustion at scale."
