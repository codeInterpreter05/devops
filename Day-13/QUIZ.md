# Day 13 — Quiz: TCP/IP & DNS

Try to answer without looking at your notes. Answers are at the bottom.

1. Why does the TCP handshake require three packets instead of two?
2. Why are TCP initial sequence numbers (ISNs) randomized rather than starting at a fixed value?
3. Walk through the 4-way connection teardown. Why can it sometimes look like only 3 packets on the wire?
4. What is `TIME_WAIT`, which side holds it, and why does it matter for a high-throughput server?
5. In the OSI model, at which layer does a Kubernetes `ClusterIP` Service operate, and at which layer does an `Ingress` operate? Why does that distinction matter practically?
6. What's the difference between a recursive query and an iterative query in DNS? Give an example of each.
7. Why should you lower a DNS record's TTL *before* a cutover instead of during or after it?
8. What's the difference in round trips between a TLS 1.2 and a TLS 1.3 handshake, and what protocol change makes that possible?
9. What is forward secrecy, and how does ephemeral Diffie-Hellman provide it?
10. What specific problem does HTTP/2 solve relative to HTTP/1.1, and what problem does HTTP/2 *not* solve that HTTP/3 does?
11. What's the difference between a certificate's Common Name and its Subject Alternative Name, and which one do modern browsers actually check for hostname matching?
12. **Interview question:** What happens at the network level when a pod in K8s talks to a service in another namespace?

---

## Answers

1. TCP is full-duplex — the client and server each maintain independent byte-stream sequence numbers. A two-packet handshake (SYN, SYN-ACK) would only prove to the client that the server received its SYN; it would never prove to the *server* that its own reply can actually reach the client, since the server has no way of knowing the SYN-ACK arrived until it sees a response to it. The third packet (the client's final ACK) is what confirms bidirectional reachability, not just one-way delivery.
2. Randomized ISNs (RFC 6528) prevent an attacker from predicting sequence numbers and blindly injecting or spoofing packets into a connection without ever seeing the actual traffic. Early TCP stacks used predictable, often sequential ISNs, which was a real, exploited vulnerability (off-path TCP spoofing/session hijacking).
3. `FIN → ACK → FIN → ACK`. It's four packets because each direction of the connection closes independently: the initiator's `FIN` only means "I have no more data to send" — the peer may still have data in flight, so it ACKs the FIN (entering "half-closed") and sends its own `FIN` once *it's* done, which the initiator then ACKs. It can look like 3 packets when the peer has nothing left to send at the moment it processes the first FIN, letting it combine its ACK and its own FIN into a single packet.
4. `TIME_WAIT` is a state the **active closer** (whichever side sends the final ACK of the 4-way teardown) holds a socket in for roughly 2×MSL before fully releasing it. It exists so that (a) if that final ACK is lost, the peer's retransmitted FIN can still be ACKed, and (b) any stray, delayed duplicate segments from the now-dead connection expire before a new connection could reuse the same 4-tuple and get confused by them. It matters for high-throughput servers because large numbers of short-lived connections accumulate `TIME_WAIT` sockets that consume ephemeral ports (and socket-table entries) faster than they're released, potentially exhausting the port range and causing new connections to fail.
5. A `ClusterIP` Service is an **L4** construct — kube-proxy programs `iptables`/IPVS rules that DNAT a `ClusterIP:port` to a backend `PodIP:port`; it never looks past the TCP/UDP header. An `Ingress` (or any L7 proxy/API gateway) is an **L7** construct — it terminates/parses HTTP and can route by hostname, path, or header. This matters practically because only an L7 router can do things like `/api` → service A, `/` → service B on the same IP/port; an L4 Service structurally cannot, since it never inspects the payload.
6. A **recursive** query says "give me the final answer or an error, don't just point me elsewhere" — this is what a client sends to its configured recursive resolver (e.g. `dig example.com` querying `8.8.8.8`). An **iterative** query gets either the final answer or a referral to another server, and the *querier itself* must follow that referral — this is how the recursive resolver talks to root, TLD, and authoritative servers (e.g. root server says "I don't know, ask the `.com` TLD servers"). `dig +trace` performs and displays this entire iterative chain directly.
7. TTL controls how long *already-cached* answers remain valid without being re-checked. If you lower the TTL at the moment of cutover, resolvers that cached the record under the *old*, longer TTL will keep serving the old value for however long was left on that old TTL — your new short TTL only applies to answers cached *after* the change. Lowering it well in advance and waiting out the old TTL's duration ensures that, by the time you actually cut over, everyone is honoring the new short TTL and will pick up the change quickly.
8. TLS 1.2 needs 2 round trips before application data can flow (`ClientHello` → `ServerHello`/`Certificate`/`ServerKeyExchange`/`Done` → `ClientKeyExchange`/`Finished` → `Finished`). TLS 1.3 needs only 1 (or 0 with session resumption), because the client speculatively sends its ephemeral key share in the `ClientHello` itself (guessing which group the server supports), letting the server derive handshake keys and reply with its own key share plus the rest of the handshake (certificate, Finished) all encrypted in a single flight.
9. Forward secrecy means that even if an attacker later compromises a server's long-term private key, they cannot decrypt traffic they recorded in the past. Ephemeral (EC)DHE provides this because the actual session key is derived from a key pair generated fresh for that one handshake and discarded immediately afterward — the long-term certificate key is only used to *sign*/authenticate the exchange, not to encrypt the premaster secret directly. With no ephemeral key exchange (old static RSA key exchange), the premaster secret was encrypted with the server's long-term public key, so stealing that key later retroactively decrypts every past session recorded under it.
10. HTTP/2 solves **application-layer head-of-line blocking**: by multiplexing many streams over a single TCP connection with a binary framing layer, one slow/stalled request no longer blocks others from being sent/received on the same connection. What it does *not* solve is **transport-layer head-of-line blocking**: because it's still one TCP connection, a single lost packet forces TCP's in-order delivery to block *all* multiplexed streams until that packet is retransmitted and arrives. HTTP/3, by running over QUIC (per-stream reliability over UDP) instead of TCP, fixes that remaining problem — a lost packet only stalls its own stream.
11. The Common Name (CN) is a legacy subject field historically used to identify the certificate's primary hostname; the Subject Alternative Name (SAN) is an X.509v3 extension listing every DNS name (and/or IP) the certificate is actually valid for. Modern browsers and TLS clients have ignored the CN for hostname validation since roughly 2017 and check **only the SAN list** — a certificate with a correct CN but no matching SAN entry will fail validation in any current browser.
12. **Interview question — full walkthrough:**

    Say Pod A in namespace `ns-a` sends a request to `curl backend.ns-b.svc.cluster.local`, where `backend` is a Service in namespace `ns-b`.

    - **DNS resolution:** Pod A's `/etc/resolv.conf` points at CoreDNS's ClusterIP (e.g. `10.96.0.10`). Since the name is already close to fully qualified, CoreDNS's `kubernetes` plugin resolves `backend.ns-b.svc.cluster.local` directly to the Service's `ClusterIP` (say `10.96.5.5`) by looking up the `Service`/`Endpoints` objects it watches from the API server.
    - **The app opens a TCP connection to the ClusterIP:** the kernel routes the outgoing packet, which first passes through netfilter. Since this is a new connection, conntrack has no existing entry, so the packet hits the `iptables` (or IPVS) rules kube-proxy programmed for that Service: the `KUBE-SERVICES` chain matches the Service's `ClusterIP:port`, jumps to a per-Service chain, and then to a per-endpoint chain that performs **DNAT** — rewriting the destination from `ClusterIP:port` to one specific backend `PodIP:port` (chosen via random probability in iptables mode, or a real load-balancing algorithm in IPVS mode). Conntrack records this translation so return traffic gets un-DNATed automatically, and Pod A's application never sees the real pod IP — only the ClusterIP it originally dialed.
    - **The rewritten packet now needs to physically get to the backend pod**, which happens to live in `ns-b` — but from this point on, the namespace is completely irrelevant to routing. The packet leaves Pod A's network namespace through its `veth` pair (the host-side end plugged into the node's CNI bridge, e.g. `cni0`, or handled by pure L3 routing in CNIs like Calico). If the backend pod is on the **same node**, it's usually just an L2 hop across the bridge (or a direct route) straight to the destination pod's `veth`. If it's on a **different node**, the packet leaves via the node's physical NIC: an overlay CNI (Flannel VXLAN, Weave) encapsulates it in a UDP packet to the destination node's IP and decapsulates it there; a native-routing CNI (Calico in BGP mode, cloud-native CNIs like the AWS VPC CNI) instead routes the real pod IP directly across the underlying network/VPC route tables with no encapsulation at all.
    - **Only once the packet physically arrives at the backend pod's `veth`** does the actual TCP three-way handshake happen — SYN, SYN-ACK, ACK — between Pod A's real IP:port and the backend pod's real IP:port. TCP has no concept of Kubernetes namespaces whatsoever; it only ever sees two IP:port pairs.
    - **Namespace isolation is not automatic.** By default, any pod can route to any other pod's IP across any namespace — Kubernetes namespaces are purely an API-server/RBAC scoping construct (who can list/create/delete objects, and a boundary for short DNS names), not a network segmentation boundary. The only thing that can actually block this traffic based on namespace is a `NetworkPolicy` object, and only if the cluster's CNI enforces `NetworkPolicy` at all (Calico, Cilium, etc. do; plain Flannel silently ignores them, so the traffic simply isn't blocked with no error or warning).
    - **The response** reverses through the same path: routed/encapsulated back to Pod A's node, and conntrack reverses the DNAT so Pod A's kernel delivers the reply as if it came from the `ClusterIP` it dialed, keeping the whole Service abstraction transparent to the application.

    A strong interview answer hits all of these beats explicitly: DNS resolves the Service name to a ClusterIP via CoreDNS; kube-proxy's iptables/IPVS rules DNAT that ClusterIP to a real backend pod IP; the packet then physically traverses veth → bridge/CNI routing (or overlay encapsulation cross-node) to reach that pod, with the TCP handshake happening between the real pod IPs; and namespaces are an API/RBAC construct, not a network isolation boundary, so cross-namespace traffic is not blocked unless a NetworkPolicy explicitly says so.
