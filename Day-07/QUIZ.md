## Answers

1. `.service` = a process to run and supervise (e.g., a Python app or daemon). `.timer` = a schedule that activates a matching `.service` unit ŌĆö systemd's replacement for cron. `.target` = a synchronization/grouping point for other units, similar to an old SysV runlevel (e.g., `multi-user.target` = normal boot-to-login state).
2. `After=`/`Before=` only control **start ordering** ŌĆö they say nothing about whether the referenced unit is required or even present. `Wants=`/`Requires=` control **dependency strength**: `Wants=` pulls in the target unit as a soft dependency (this unit still starts even if the wanted one fails), while `Requires=` is a hard dependency (this unit fails to start if the required one fails). You typically need an ordering directive *and* a dependency directive together to get the behavior you actually want.
3. Forgot `sudo systemctl daemon-reload`. Systemd parses unit files into an internal cached representation when they're loaded; it does not watch `/etc/systemd/system/` for live changes, so edits are invisible until you explicitly tell systemd to re-read unit files with `daemon-reload`.
4. `enable` only creates the boot-time symlink (in `<target>.wants/`) so the unit starts automatically on the next boot ŌĆö it does not start the unit right now. `start` runs the unit immediately but does not persist across a reboot unless it's also enabled. `enable --now` does both in one command: persists for future boots and starts it immediately.
5. `no` (default, never auto-restart), `on-success` (restart only on clean exit), `on-failure` (restart on non-zero exit, signal kill, or timeout), `on-abnormal` (signal/timeout/watchdog, not plain non-zero exit), `on-watchdog` (only on missed watchdog keepalive), `on-abort` (only on uncaught fatal signal), `always` (restart regardless of exit reason). Use `on-failure` for the standard "bring it back up if it crashes" behavior on a service that's expected to exit cleanly sometimes (e.g., during a deliberate `systemctl stop`). Use `always` only when the process should never legitimately stay down even after a clean exit ŌĆö be aware it will also restart it if someone kills it outside of `systemctl stop`.
6. They act as a circuit breaker on the restart loop: if a unit fails and restarts more than `StartLimitBurst` times within `StartLimitIntervalSec`, systemd stops trying and marks the unit failed instead of retrying forever. Without this, a service that fails instantly on every start (e.g., due to a missing dependency or bad config) would restart-loop at the rate allowed by `RestartSec`, burning CPU and flooding the journal indefinitely.
7. Journal priority filters match the given level **and everything more severe**, not an exact level ŌĆö `-p err` includes `err`, `crit`, `alert`, and `emerg`. To get exactly one level, use the range syntax: `journalctl -p err..err`.
8. Because journal persistence depends on whether `/var/log/journal/` exists. If it doesn't, journald falls back to the volatile, RAM-backed `/run/log/journal/`, which is wiped on every reboot ŌĆö so logs from any prior boot are simply gone, regardless of how many times the machine has rebooted.
9. `*/15 * * * 1-5 /opt/scripts/sync.sh`. Fields left to right: minute = `*/15` (every 15 minutes), hour = `*` (every hour), day-of-month = `*` (any), month = `*` (any), day-of-week = `1-5` (Monday through Friday, where 1=Monday).
10. Most likely causes: (a) cron runs jobs with a minimal `PATH` and no shell profile sourced, so a binary that works from an interactive shell (e.g., a pyenv/nvm-managed interpreter) isn't found; (b) the script uses relative paths that resolve against an unexpected working directory (cron typically starts jobs in the user's home directory, not wherever you were standing when you wrote the script), or relies on environment variables that were never set. Standard fix: use absolute paths everywhere, set `PATH` (and any other required env vars) explicitly at the top of the crontab, and redirect stdout/stderr to a log file (`>> file.log 2>&1`) rather than relying on cron's default mail-based output delivery, which usually goes unread.
11. Anacron is a scheduler designed for machines that aren't always powered on (laptops, workstations) ŌĆö instead of exact wall-clock scheduling like cron, it tracks, per job, how many days have elapsed since it last ran, and catches up any job whose period has elapsed the next time the machine (or anacron itself) runs, even if the "ideal" time was missed while the machine was off. Cron has no such catch-up: if the machine is off at the scheduled time, the run is simply skipped forever. A systemd timer with `Persistent=true` achieves the same catch-up behavior natively ŌĆö systemd records the last trigger time and fires the timer once at the next boot if a scheduled run was missed ŌĆö meaning you don't need anacron as a separate mechanism when you're using systemd timers instead of cron.
12. Strong answer: "Auto-restart on crash is a systemd feature, configured per-unit in the `[Service]` section via `Restart=` and `RestartSec=`. `Restart=on-failure` is the standard choice ŌĆö it restarts the service if it exits with a non-zero code, is killed by a signal, or times out, but not on a clean/intentional stop. Other policies exist for narrower cases: `on-abnormal` (signal/timeout/watchdog only), `on-watchdog` (only if a `WatchdogSec=` keepalive is missed), `on-abort` (only on an uncaught fatal signal), `always` (restart no matter how it exited, including clean exits), and `no` (the default ŌĆö never restart). `RestartSec=` controls the delay between restart attempts, and `StartLimitIntervalSec=`/`StartLimitBurst=` cap how many restarts are allowed in a time window so a service that fails instantly doesn't restart-loop forever ŌĆö after the burst limit trips, systemd marks the unit failed instead of continuing to retry. You verify it's actually working by checking `systemctl status` for a new `Main PID` after a crash, and `journalctl -u <service>` for the exit/restart sequence." Mention a concrete example if you have one ŌĆö e.g., killing the process with `kill -9` and watching a new PID come up within `RestartSec` seconds.
