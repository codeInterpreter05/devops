# Day 8 — Quiz: Git Deep Dive I

Try to answer without looking at your notes. Answers are at the bottom.

1. What does a Git commit object actually store, and what does it deliberately *not* store? Why does that distinction matter for how expensive operations like `checkout` and `rebase` are?
2. In one sentence each: what is a blob, a tree, and a commit, and how do the three connect to form a full project snapshot?
3. What is a ref, concretely (in terms of what's actually on disk)? What does creating a new branch cost, and what's different about `HEAD` in a normal checkout versus a detached `HEAD`?
4. What does `git log --graph --oneline --decorate --all` show that a plain `git log` doesn't, and how do you tell a merge commit apart from a fast-forward or a rebase just by looking at the graph shape?
5. What's the difference between Git's plumbing and porcelain commands? Name one of each, and explain how the porcelain command is actually built from plumbing underneath.
6. In an interactive rebase (`git rebase -i`), what's the difference between the `squash` and `fixup` verbs, and when would picking the wrong one cost you something?
7. Why does replaying a commit during a rebase give it a brand-new SHA, even though the diff it introduces is byte-for-byte identical to before?
8. What is the "golden rule of rebasing"? Concretely, what breaks for a teammate if you rebase and force-push a branch they've already pulled?
9. When would you reach for `git cherry-pick` instead of `merge` or `rebase`? Give one concrete, real-world scenario.
10. What does `git reflog` actually track, what are its two key limitations, and how is a commit that's recoverable via reflog/`fsck` different from one that's truly, permanently gone?
11. What's the difference between `HEAD@{2}` and `HEAD~2`? Why can they point at completely different commits right after a reset or a rebase?
12. **Interview question:** What's the difference between `git merge` and `git rebase`? When would you use each in a team?

---

## Answers

1. A commit stores a pointer to one **tree** (a complete snapshot of every file in the project at that point), pointer(s) to its parent commit(s), and metadata (author, committer, timestamp, message). It does **not** store a diff — diffs are computed on demand by comparing two trees. This matters because `git checkout <commit>` is "point at this tree and check it out," not "replay a chain of patches," while rebase genuinely does have to reconstruct and reapply a diff for every commit it replays — which is exactly why rebase is comparatively more work than a checkout and why it can conflict per-commit.

2. A **blob** is raw file content with no name or permissions attached. A **tree** is a directory listing — a set of `(mode, type, sha, name)` entries pointing at blobs (files) or other trees (subdirectories). A **commit** points to exactly one tree (the root of the project at that moment) plus its parent commit(s) and metadata. Together: commit -> tree -> (blobs and/or subtrees, recursively) -> a complete, addressable snapshot of the whole project.

3. A ref is just a small file containing a 40-character SHA-1 (or, for `HEAD`, usually a pointer to another ref, e.g. `ref: refs/heads/main`). Creating a branch (`git branch foo`) costs almost nothing — it's writing one new small file under `.git/refs/heads/`, not copying the repository. In a normal checkout, `HEAD` is a **symbolic ref** pointing at a branch, so committing updates that branch automatically. In a **detached HEAD** state, `HEAD` points directly at a commit SHA instead of at a branch name — commits made there aren't on any branch, and can become unreferenced (recoverable only via reflog/fsck) the moment you check out something else.

4. Plain `git log` only follows the current branch's ancestry, linearly, one commit per section. `--graph` draws the actual DAG shape (branch points, merge points) using ASCII connectors; `--decorate` labels commits with whatever branch/tag refs point at them; `--all` includes every ref, not just the current branch, so branches that haven't been merged yet are still visible. A **merge commit** shows as a node with two parent lines converging into it (`|/` or similar) — proof two histories were combined non-destructively. A straight, unbranching line of commits with no converging parents means the history was produced by a **fast-forward** or a **rebase** (both are linear), not a merge.

5. **Porcelain** commands (`git commit`, `git merge`, `git rebase`, `git log`, `git status`) are the user-facing, friendly layer with safety rails and readable output. **Plumbing** commands (`git cat-file`, `git hash-object`, `git write-tree`, `git commit-tree`, `git update-ref`) are the low-level primitives, designed for scripting rather than humans. Concretely, `git commit` is built from plumbing: it takes the index, runs `write-tree` to turn it into a tree object, runs `commit-tree` with that tree and the current `HEAD` as parent to produce a commit object, then `update-ref`s the current branch to point at the new commit. Porcelain is just plumbing with guardrails and better output.

6. Both `squash` and `fixup` combine a commit's changes into the commit directly above it in the rebase todo list. The difference is entirely about the **commit message**: `squash` stops to let you edit and merge both commits' messages together; `fixup` silently discards the fixup commit's message and keeps only the message of the commit it's being folded into. Picking `squash` when you meant `fixup` leaves noisy "fix typo" text baked into your final message; picking `fixup` when you meant `squash` silently throws away a message that might have had useful context.

7. A commit's SHA is a hash of its own content, and its content includes a pointer to its **parent commit's SHA**. When rebase replays a commit onto a new base, the parent pointer necessarily changes (it now points at the new base or the previous replayed commit, not the original parent) — so even though the tree/diff is identical, the commit object's content is different, and therefore its hash is different. The author, message, and diff can be byte-identical and the SHA will still change, because the parent link is part of what's being hashed.

8. The golden rule: never rebase (and rewrite) commits that others may have already pulled or built work on top of. If you do, and force-push, anyone who already fetched the old commits now has a branch that has diverged in a way Git can't auto-reconcile: their next ordinary `git pull` (which does a merge by default) will either create a confusing merge combining "old" and "new" versions of what are logically the same commits, or duplicate every rewritten commit in their local history. They'd have to know to discard their old copy and reset onto your new history — not something you can assume without explicit coordination.

9. Cherry-pick is for pulling one **specific** commit onto a branch without bringing along everything else that branch has done since diverging. The canonical scenario: a critical bug fix lands as a single commit on `main`, and a `release-2.3` branch — which is otherwise frozen and shouldn't get all of `main`'s unrelated feature work — needs exactly that fix. `git cherry-pick -x <sha>` (checked out on `release-2.3`) applies just that commit's diff as a new commit there, with `-x` recording where it came from for traceability.

10. The reflog is a local, per-ref (mainly `HEAD` and each branch), chronological log of every point that ref has pointed to, along with what operation moved it (commit, reset, rebase, checkout, merge, etc.) — stored as plain-text files under `.git/logs/`. Its two key limitations: it is **local only** (never pushed, fetched, or cloned — useless once the repo/clone is gone), and it has an **expiry** (90 days by default for entries still reachable from a ref, 30 days for entries that have become unreachable — actually removed only when `git gc` runs). A commit recoverable via reflog or `git fsck --unreachable`/`--lost-found` is one whose objects are still physically present in `.git/objects`, just unreferenced by any ref — "deleted" in Git almost always means "unreferenced." A commit is **truly, permanently lost** only once it's unreachable from every ref *and* every reflog entry (past its expiry, or explicitly expired), **and** an actual `git gc`/`git prune` has run to remove it from disk — losing a pointer alone is never enough by itself.

11. `HEAD@{2}` is a **reflog-relative** reference: "wherever `HEAD` was, 2 ref-movements ago" (per the log in `.git/logs/HEAD`). `HEAD~2` is an **ancestry-relative** reference: "2 commits back along the current commit's first-parent chain," computed purely from parent pointers in the object graph, with no reference to the reflog at all. They usually agree right after a plain sequence of commits, but diverge sharply the moment you `reset`, `rebase`, or `checkout` something else — at that point, "where HEAD used to be" (reflog) and "N commits back in the current graph" (ancestry) are answering genuinely different questions, which is exactly why reflog recovery uses `HEAD@{n}`, not `HEAD~n`.

12. **Merge** creates a new commit with two parents, combining two histories without altering any existing commit — always safe on shared/pushed branches, and it preserves the true, literal shape of what happened (including the fact that a merge occurred). **Rebase** replays one branch's commits, one at a time, onto a new base, producing new commits (new SHAs) with the same diffs and messages — it rewrites history to look linear, as if the work had been done sequentially on top of the latest base, but it does not touch anyone else's copy of those commits; it just makes your local ones obsolete. In a team: use **rebase** to clean up and update your own local, not-yet-shared feature branch before opening a PR (interactive rebase to squash/fixup WIP commits, or `rebase main` to stay current without merge-commit noise). Use **merge** to integrate a finished branch into a shared branch like `main`, and always once a branch has been pushed and others might have based work on it — rebasing shared history forces every collaborator to reconcile diverged SHAs, which is exactly the "golden rule of rebasing" violation to avoid.
