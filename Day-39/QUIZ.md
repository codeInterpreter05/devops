# Day 39 — Quiz: Secrets in AWS & K8s

Try to answer without looking at your notes. Answers are at the bottom.

1. What's the core functional difference between AWS Secrets Manager and SSM Parameter Store, beyond "one costs money and one doesn't"?
2. Is a `SecureString` parameter in SSM Parameter Store cryptographically weaker than a Secrets Manager secret? Why or why not?
3. Why is a native Kubernetes `Secret` object not, by itself, a secrets management solution?
4. What does External Secrets Operator actually do, end to end, from AWS Secrets Manager to a running pod?
5. If the upstream secret in AWS Secrets Manager rotates, does the application automatically start using the new value? What's missing if not?
6. How does Sealed Secrets make it safe to commit an encrypted secret to a public (or widely-readable) git repository?
7. What's the key operational trade-off between ESO and Sealed Secrets?
8. What does SOPS encrypt versus leave in plaintext within a YAML file, and why does that matter for code review?
9. Why has `age` become the preferred keypair mechanism for SOPS over GPG for many new projects?
10. If a real API key is accidentally committed and pushed to a git repo, is deleting it in the next commit sufficient remediation? What is?
11. Why is baking a secret into a Docker image via `COPY .env .` or a plain `ARG` dangerous even if a later build stage deletes the file?
12. **Interview question:** How would you rotate database passwords in production with zero downtime?

---

## Answers

1. Secrets Manager provides built-in automatic rotation (native Lambda integrations for RDS/Aurora/Redshift/DocumentDB), versioning with staging labels for safe cutover, and per-secret resource policies, at a per-secret monthly cost. SSM Parameter Store's Standard tier is free and excels at hierarchical configuration (`get-parameters-by-path`) but has no built-in rotation — you'd have to build that yourself.
2. No — both use KMS encryption under the hood, so the cryptographic strength is comparable. The real differences are rotation tooling, API shape, and pricing, not encryption strength.
3. Kubernetes Secrets are base64-encoded, not encrypted — trivially reversible by anyone with RBAC read access to that namespace's secrets, or access to an unencrypted etcd datastore. Real protection requires separate etcd encryption at rest and careful control of who can `get`/`list` secrets — the Secret object itself is a storage/RBAC construct, not a security boundary.
4. ESO's controller (authenticated via workload identity like IRSA) periodically polls AWS Secrets Manager per the `ExternalSecret`'s `refreshInterval`, fetches the current secret value, and writes/updates a native Kubernetes `Secret` object in the cluster. A pod then mounts/reads that Secret exactly like any other Kubernetes Secret, with zero AWS-specific code or awareness in the application itself.
5. Not automatically at the running-pod level — ESO updates the Kubernetes Secret object on its next poll, but a pod that already started with the old value as an environment variable (set once at container start) keeps using the stale value until it's restarted. A reload/restart mechanism (e.g., Reloader, a rolling restart trigger, or an app that watches its mounted volume for changes) is needed to actually pick up the new value.
6. `kubeseal` encrypts a Secret manifest client-side using the Sealed Secrets controller's **public** key, which is safe to distribute. Only the controller's private key (which never leaves the specific cluster) can decrypt it back into a real Secret — so anyone reading the encrypted manifest in git (even publicly) gains nothing without also compromising that specific cluster's controller.
7. ESO requires the external secrets backend to be reachable at sync time and gives you automatic rotation propagation; Sealed Secrets requires no external dependency at decrypt time (just the in-cluster controller) but needs manual re-sealing and re-committing whenever the secret value changes.
8. SOPS encrypts only the **values** in a structured file, leaving the **keys** in plaintext. This means a git diff/PR review shows exactly which configuration keys changed, even though the new secret values themselves remain unreadable — a meaningful usability advantage over whole-blob encryption approaches.
9. `age` has a dramatically simpler operational model than GPG — generating a keypair and distributing the public key is one line each, with no keyservers, trust webs, or subkey expiration management, while still using strong modern encryption (ChaCha20-Poly1305) underneath.
10. No. Deleting it from the latest commit does not revoke the credential — anyone who already cloned the repo, or an automated scraper that indexed the public exposure window, still has the working key. The only real remediation is rotating/revoking the actual credential at its source (the cloud provider, database, or third-party API); cleaning git history is a secondary step, not the fix itself.
11. Docker images are built in layers, and each layer is a snapshot of the filesystem changes at that point — a file added in one layer still physically exists in that layer even if a later layer deletes it from the "final" visible filesystem. Anyone with the image (`docker save`, `docker history`, or extracting layer tarballs) can recover the secret from the earlier layer regardless of later deletion. The safe alternative is BuildKit's `--mount=type=secret`, which makes the secret available only during that specific `RUN` step and never persists it into any layer.
12. Strong answer: "Use dynamic, short-lived credentials wherever possible (Vault's database secrets engine, Day 38) so 'rotation' is continuous by construction rather than a risky cutover event. For a database that must use a long-lived credential (e.g., the app's primary connection user, or when adopting Secrets Manager's native RDS rotation), zero-downtime rotation relies on having **two valid credentials simultaneously** during the transition: create/enable a new credential first, update the application's configuration or have it re-fetch from the secrets store, confirm it's using the new credential (e.g., via connection metrics or logs), and only then revoke the old one — never revoke the old credential before confirming the new one is actually in active use, since in-flight connections and connection pools may still be relying on it. AWS Secrets Manager's native RDS rotation implements exactly this pattern using `AWSCURRENT`/`AWSPENDING`/`AWSPREVIOUS` staging labels, keeping the old password valid until the new one is fully verified working." A weak answer just says "change the password" without addressing the overlap/cutover problem that causes downtime.
