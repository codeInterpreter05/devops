# Day 9 — Quiz: Git Deep Dive II — Workflows

Try to answer without looking at your notes. Answers are at the bottom.

1. What are the five branch types in GitFlow, and what is each one for?
2. What mechanism makes trunk-based development safe for continuous deployment, and why is trunk-based development without it considered reckless?
3. Name three branch protection rules and, for each, describe a concrete incident it prevents.
4. Why does "require status checks to pass" need to be paired with "require branches to be up to date" (or a merge queue) to actually be effective?
5. What is the difference between a client-side Git hook and a server-side Git hook, and why can't a client-side hook be treated as a hard enforcement boundary?
6. What problem does the `pre-commit` framework (or `husky`) solve that raw scripts in `.git/hooks/` don't?
7. Why is pinning an exact `rev:` in `.pre-commit-config.yaml` a security control and not just a stability nicety?
8. Write the Conventional Commits format for a breaking change to an API's `scope`, and explain what a `semantic-release`-style tool would do differently for a `fix:` commit versus a `feat:` commit versus one with a `BREAKING CHANGE` footer.
9. Why is `git commit`'s `Author` field spoofable by default, and what specifically does GPG signing add that a plaintext `Author` field does not have?
10. What does a green "Verified" badge on GitHub actually prove, and what does it not prove? What causes a cryptographically valid signature to still show as "Unverified"?
11. What is one concrete advantage of SSH-based commit signing over GPG-based signing?
12. **Interview question:** How do you enforce code quality and commit standards across a team without slowing them down?

## Answers

1. `main` (production-reflecting, every commit is a release), `develop` (integration branch where finished features land before a release), `feature/*` (branched from `develop`, one feature, merged back to `develop`), `release/*` (cut from `develop` when feature-complete, only stabilization changes, merged into both `main` and `develop`), `hotfix/*` (branched from `main` for urgent production fixes, merged into both `main` and `develop`).
2. Feature flags. They decouple **merging** code into `main` from **releasing** it to users — code can be merged (even if incomplete) behind a flag that's off by default, then activated later without another deploy. Without flags, "trunk-based" just means merging unfinished or risky code directly into the branch that tracks production, which is exactly the failure mode long-lived feature branches were isolating against.
3. Examples: (a) "no direct push, PR required" — prevents someone bypassing review entirely by pushing straight to `main`; (b) "no force-push" — prevents history being silently rewritten out from under teammates who've already based work on the old commits; (c) "required status checks" — prevents code that fails tests/lint from being merged just because a human approved it without re-running CI in their head; (d) "require signed commits" — prevents commits from an unverified/spoofed identity landing on the protected branch.
4. Two PRs can each be individually green against an older snapshot of `main`, but incompatible or broken once both are merged in sequence — CI ran against a base that's now stale for the second PR. "Require branches to be up to date" (or a merge queue that re-tests combinations before merging) closes that gap by re-validating against the *current* `main` right before merge, not just at PR-open time.
5. Client-side hooks (`pre-commit`, `commit-msg`, `pre-push`) run on the developer's own machine and can be skipped (`git commit --no-verify`, or simply never installed). Server-side hooks (`pre-receive`, `update`) run on the Git server/hosting platform and can't be bypassed by anything the client does — this is why the real enforcement boundary is CI + branch protection (the platform-side equivalent), while client-side hooks are for fast local feedback only.
6. Raw hooks live in `.git/hooks/`, which is inside `.git/` and therefore **never committed** — they exist only on whichever machine created them. `pre-commit`/`husky` store the hook *definitions* in a committed config file (`.pre-commit-config.yaml`, `.husky/*`) plus a one-time install step (`pre-commit install`, or `husky`'s `prepare` npm script), so every clone/CI runner gets the same hooks automatically instead of relying on each developer manually recreating them.
7. `.pre-commit-config.yaml`'s `repo:` entries point at external Git repositories that supply the actual hook code. An unpinned or floating `rev:` (like `main`) means that external repo's maintainer (or anyone who compromises it) can push new code that runs, unreviewed, on every developer's machine at commit time — pinning an exact tag/SHA is what prevents that code from changing without an explicit, reviewable bump.
8. `feat(api)!: change deploy() signature` with a footer `BREAKING CHANGE: deploy() now requires a currency argument` (the `!` and/or the footer both signal a breaking change). A `fix:` commit triggers a **patch** version bump; a `feat:` commit triggers a **minor** bump; any commit carrying a `BREAKING CHANGE` footer (or `!`) triggers a **major** bump — regardless of its `type` — because SemVer's own rules say incompatible changes require a major version increase.
9. Git never authenticates `user.name`/`user.email` against any real identity — they're just local config values the committer can set to anything. GPG signing adds a cryptographic signature over the commit object itself, produced with a private key only the real signer holds; anyone can verify it against the corresponding public key without needing access to that private key, which a plaintext `Author` string can never provide.
10. "Verified" proves the commit's signature is cryptographically valid **and** the signing key is registered to a specific account with a matching verified email — i.e., a specific keyholder produced this commit. It does **not** prove the code in the commit is safe, correct, or reviewed. A cryptographically valid signature shows as "Unverified" when the signing key simply isn't registered to any account, or its associated email doesn't match a verified email on that account — the math is still correct, the platform just can't attribute it to a known identity.
11. It reuses SSH keys engineers typically already have for authentication/pushing, avoiding GPG's separate toolchain, key-expiry management, and keyserver/web-of-trust ceremony — one fewer credential type and setup flow to maintain, while still producing the same "Verified" badge behavior on GitHub/GitLab.
12. Strong answer: "Automate everything mechanical so review time is spent on logic, not formatting. Fast, local feedback first — `pre-commit`/`husky` hooks run `shellcheck`, formatters, and `commitlint` in seconds, before a commit even exists, so developers fix issues immediately instead of waiting on a CI round-trip. Then a real, unbypassable gate in CI plus branch protection — required status checks, required reviews, no force-push — because client-side hooks can always be skipped with `--no-verify` and shouldn't be the only line of defense. Conventional Commits gives commit messages a machine-readable structure, which pays for itself by automating semantic versioning and changelog generation instead of requiring a human to write release notes. None of this slows the team down in practice — it removes categories of review comments ('please quote your variables', 'please fix your commit message') and manual release-prep work, so humans spend review time on design and correctness instead of formatting nits."
