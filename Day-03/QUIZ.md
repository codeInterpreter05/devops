# Day 3 — Quiz: Processes & System

Try to answer without looking at your notes. Answers are at the bottom.

1. What is a zombie process, and why does `kill -9 <pid>` do nothing to remove one?
2. What's the difference between a zombie process and an orphan process? What resolves each one?
3. Name the five main process states (`R`, `S`, `D`, `T`, `Z`) and say which one cannot be terminated even by `SIGKILL`, and why.
4. Why can `SIGKILL` never actually reach a process stuck in `D` state, and what are your real options if that's happening on a production box?
5. What's the practical difference between `SIGTERM` and `SIGKILL`? Why do systemd, Docker, and Kubernetes all send `SIGTERM` before `SIGKILL` when stopping something?
6. What does `nice`/`renice` actually control, and why can a regular (non-root) user only raise their own process's niceness, never lower it?
7. `nohup` and `disown` both seem to let a background job survive you logging out — what's the actual mechanical difference between them, and when would you reach for one over the other?
8. Why does killing a single parent PID often fail to stop an entire job's worker processes? What's the correct way to kill an entire process tree or pipeline at once?
9. What's the difference between `ps aux` and `ps -ef`? Are they showing fundamentally different information?
10. What can `lsof` tell you about a process that `ps` cannot? Give one concrete production scenario where `lsof` is the tool that solves the problem.
11. **Interview question:** How do you find and kill a process consuming 100% CPU without killing its children?
12. `SIGHUP` was originally about a terminal disconnecting — what does it commonly mean when sent to a long-running daemon like `nginx` or `sshd` today, and why did that repurposing happen?

## Answers

1. A zombie (`Z` state, shown as `<defunct>` in `ps`) is a process that has already finished executing — it has no code running, no memory footprint beyond a small process-table entry holding its PID and exit status. That entry is kept around specifically so its parent can retrieve the exit status via `wait()`/`waitpid()`. `kill -9` sends a signal to a running process to force it to stop running — but a zombie isn't running anything; there is nothing left for a signal to act on. The zombie disappears only when the parent reaps it (calls `wait()`) or when the parent itself exits (the zombie gets re-parented to PID 1, which reaps it immediately).
2. A zombie is a **dead child whose parent is still alive** but hasn't collected its exit status yet. An orphan is the opposite: a **still-running child whose parent has died**. Orphans are automatically re-parented to PID 1 (`init`/`systemd`), which periodically reaps its children — so orphans self-resolve once they eventually exit. Zombies do not self-resolve; they require either the original parent to call `wait()`, or the parent to die (after which init reaps the now-orphaned zombie instantly).
3. `R` = running or runnable (on a CPU or waiting in the run queue); `S` = interruptible sleep (waiting on an event, can be woken by a signal); `D` = uninterruptible sleep (blocked on low-level I/O, typically disk/NFS); `T` = stopped (via `SIGSTOP`/`SIGTSTP` or being traced); `Z` = zombie (exited, awaiting reaping). `D` is the one that cannot be terminated by `SIGKILL` — the kernel won't deliver any signal to a process while it's blocked inside a non-interruptible syscall, because the syscall's contract guarantees it will run to completion (or timeout) before the process can respond to anything, signals included.
4. Because delivering a signal requires the kernel to interrupt the process's execution, and a process in `D` state is, by definition, inside a kernel-level I/O operation the kernel has decided is not safe to interrupt (typically to avoid leaving a device driver or a remote filesystem protocol in an inconsistent state). Real options: fix whatever the I/O is stuck on (e.g., a hung NFS server, a failing disk — sometimes remounting or restarting the underlying service resolves it), wait for a timeout to expire on its own, or, if nothing else works, reboot the box (which the process can't prevent, since the kernel itself is being torn down).
5. `SIGTERM` (15) is the default, catchable signal — it asks a process to shut down gracefully, giving well-written programs the chance to close connections, flush buffers, finish an in-flight transaction, and exit on their own terms. `SIGKILL` (9) is enforced directly by the kernel with no way for the target process to intervene — it's removed from the scheduler immediately (once not blocked in `D` state), with zero cleanup. Orchestrators send `SIGTERM` first and wait a grace period (e.g., Kubernetes' `terminationGracePeriodSeconds`, default 30s) specifically to give the application a chance to shut down cleanly, only escalating to `SIGKILL` if it's still alive after the grace period — treating force-kill as the last resort, not the default.
6. `nice`/`renice` set a process's **niceness** (-20 to +19), a hint to the CPU scheduler (CFS) about how willing that process is to yield CPU time to others under contention — it's a bias on scheduling fairness, not a hard priority or guarantee, and has zero effect when the CPU isn't contended. Only root can lower niceness (raise priority) because if any unprivileged user could do it freely, everyone would set every process to -20 and the entire mechanism would stop meaning anything — it's a deliberate restriction to prevent users from starving each other of CPU.
7. `nohup` changes the **process's own signal disposition** — it makes the process ignore `SIGHUP` from the moment it's launched, and must be applied at launch time (`nohup cmd &`). `disown` is a **shell built-in** that removes an already-running background job from the current shell's job table, so the shell no longer treats it as something to signal when the shell itself exits; it doesn't touch the process's signal handling at all. Use `nohup ... &` when you know upfront a job needs to survive logout; use `disown` as a rescue for something you already backgrounded with plain `&` and now need to detach. For anything needing real monitoring/restart behavior in production, neither is a substitute for a proper process supervisor (systemd unit, `tmux`/`screen`, or a job queue).
8. Killing a single PID sends a signal only to that one process — its children are untouched and simply get orphaned (re-parented to PID 1) rather than terminated, so they keep running, invisible unless you specifically look with `ps --forest` or `pstree`. The correct approach is to signal the entire **process group** at once using a negative PGID: find the group with `ps -o pgid= -p <pid>`, then run `kill -TERM -<pgid>` (note the leading minus sign) — this delivers the signal to every process in that group simultaneously, which is exactly how shells implement Ctrl-C for an entire foreground pipeline.
9. They're two historical dialects (BSD-style `aux`, no dashes; POSIX/SysV-style `-ef`) of the same underlying `ps` tool, both implemented by the same `procps` package on Linux, reading the same `/proc` data — they differ mainly in default columns shown and flag syntax, not in what information is fundamentally available. `ps -ef` shows `PPID` and `STIME`/`C` by default; `ps aux` shows `%CPU`/`%MEM`/`VSZ`/`RSS` and `STAT` by default — but you can get equivalent data out of either with custom `-o` column selection.
10. `ps` tells you about the process itself (CPU, memory, state, command line) but nothing about what files, sockets, or devices it has open. `lsof` (list open files) enumerates every file descriptor a process holds — including network sockets and pipes, since Linux treats those as files too. Classic scenario: `df` shows a disk as full but `du` can't find what's consuming the space — this happens when a process has deleted (unlinked) a file it still holds open, so the space isn't released until the file descriptor closes. `lsof | grep deleted` finds exactly these phantom-space files; the fix is restarting/stopping the offending process (or, for logs specifically, configuring `logrotate` with `copytruncate`).
11. Identify the offending PID with `top` (default sort is by CPU) or `ps -eo pid,ppid,pcpu,cmd --sort=-pcpu | head`, note its exact PID (not just its name, in case multiple processes share a name). Then signal **only that specific PID** — `kill -TERM <pid>` (escalate to `kill -9 <pid>` only if it ignores `SIGTERM`) — rather than `kill -TERM -<pgid>` (which targets the whole process group and would hit children too) or `killall`/`pkill` by name (which could match sibling or child processes sharing a similar command name). Because a plain `kill <pid>` only ever signals that one process, any children it has are left completely alone — they don't receive the signal and don't get killed, satisfying the requirement directly. If the goal is actually to deprioritize rather than kill it, `renice` on that PID is a non-destructive alternative worth mentioning as a first step if the process seems recoverable.
12. Historically, `SIGHUP` ("hang up") was sent when a physical or dial-up terminal line disconnected, telling the foreground process attached to it that its controlling terminal was gone. Long-running daemons (`nginx`, `sshd`, and many others) have no terminal to lose in the first place, so they repurposed the same signal as a convention meaning "re-read your configuration file without a full restart" — a lightweight reload mechanism that avoids dropping existing connections. It stuck because `SIGHUP` was an otherwise-idle, universally available signal number that made sense semantically ("something about my environment changed, adjust") and didn't require inventing a new signal number outside the POSIX-standard set.
