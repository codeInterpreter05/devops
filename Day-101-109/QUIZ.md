1. Why does Loki deliberately avoid indexing log line content, only indexing a small set of labels? What would go wrong if you added `trace_id` as a Loki label on a high-traffic service?
2. What is an exemplar, and how does it let you jump directly from a spike on a Prometheus latency graph to the one specific trace that caused it?
3. Define SLI, SLO, and SLA precisely, and explain why an SLO is deliberately set looser than the best reliability you could technically achieve.
4. A service has an SLO of 99.9% availability over a rolling 30-day window and handles 10,000,000 requests during that window. How many "allowed" errors does that give you (the error budget, in absolute request count)? If 4,000 errors have occurred so far this window, what percentage of the error budget has been consumed?
5. Why is multi-window, multi-burn-rate alerting preferred over a single "SLI dropped below SLO" threshold alert? What specific failure mode does each window (long vs. short) individually guard against?
6. A service's SLO is 99.9% (error budget = 0.001). Its current 5-minute error rate is 0.05 (5%). Calculate the burn rate, and estimate roughly how long it would take to exhaust a 30-day error budget if this rate continued.
7. Explain the architectural difference between Prometheus's pull/scrape model and Loki/Tempo's push model. Why does Kubernetes observability tooling generally use `ServiceMonitor`/`PodMonitor` CRDs instead of hand-written static scrape configs?
8. What role does the OpenTelemetry Collector play between an instrumented application and Prometheus/Loki/Tempo, and what specific benefit does that decoupling give you if you later switch trace backends?
9. Explain the difference between RTO and RPO, and map "pilot light" and "warm standby" DR strategies to where they sit on the cost-vs-recovery-time spectrum.
10. On the AWS SAA-C03 exam, you're stuck on a dense multi-paragraph scenario question at the 90-second mark with no clear elimination progress. What's the correct move, and why does the exam's "no guessing penalty" rule make that move safe?
11. **Interview question:** Design the complete observability stack for a 50-microservice platform. What do you monitor, alert on, and trace?

## Answers

1. Loki's whole cost/performance advantage over something like Elasticsearch comes from **not** building a full-text index over log content — it only indexes the small label set (namespace, pod, app, etc.) and stores compressed log chunks searched at query time. If you promoted `trace_id` (or any per-request identifier) to a label, you'd create a new distinct label-set/stream for every single request on a high-traffic service — this is a cardinality explosion that overwhelms Loki's index and storage layout, the exact problem Loki's label-only-indexing design exists to avoid. Trace IDs belong in the log line body, extracted at query time with `| regexp` or `| json`, or surfaced via a Grafana derived field.
2. An exemplar is a single sample attached to a Prometheus metric (typically a histogram bucket) that carries extra metadata — most usefully, a trace ID for one specific request that fell into that bucket. Grafana renders exemplars as clickable dots overlaid on a metric graph; clicking one opens that exact trace in Tempo. This turns "latency spiked at 14:32" into "here is the literal slow request," instead of you having to guess which trace in a search result actually matches the spike.
3. **SLI** = a measurement (e.g., proportion of requests under 300ms). **SLO** = a target for that measurement over a window (e.g., 95% under 300ms over 30 days) — an internal engineering commitment. **SLA** = an SLO with a contractual/financial consequence attached, usually to an external customer. An SLO is set deliberately looser than technically achievable because it needs to leave room for *planned* risk — deploys, experiments, migrations — chasing the technical maximum burns engineering effort the business doesn't need and leaves zero room to ship anything risky without "breaking" your own target.
4. Error budget = `1 - SLO` = `1 - 0.999 = 0.001`. Over 10,000,000 requests: `10,000,000 × 0.001 = 10,000` allowed errors. With 4,000 errors observed: `4,000 / 10,000 = 40%` of the error budget consumed.
5. A single "SLI below SLO" threshold alert fails in both directions: over a long window, it reacts too slowly to a genuinely severe short outage (real damage is already done by the time a 30-day rolling average moves noticeably); over a short window alone, it fires on brief blips that self-resolve in minutes, causing alert fatigue. Multi-window alerting requires **both** a long window (proves the elevated error rate is real and sustained, not noise) **and** a short window (proves it's still happening right now, not already resolved) to breach before firing — this catches severe short outages fast (a high burn rate over even 5-10 minutes is real signal) while filtering out transient noise that a long window alone would eventually average away anyway.
6. `burn_rate = error_rate / error_budget = 0.05 / 0.001 = 50`. At a sustained burn rate of 1, the budget is exactly used up at the end of the 30-day window. At burn rate 50, the budget would be exhausted in `30 days / 50 ≈ 0.6 days`, roughly **14.4 hours**. This is far above the standard 14.4x critical/page threshold — this is a page-immediately situation, not a ticket.
7. Prometheus **pulls**: it scrapes `/metrics` HTTP endpoints on a schedule, meaning Prometheus itself decides when and from where to collect. Loki and Tempo are **push**-based: an agent (Promtail/Grafana Agent for logs, the OpenTelemetry Collector or an SDK exporter for traces) actively sends data to them. Kubernetes tooling uses `ServiceMonitor`/`PodMonitor` CRDs instead of static scrape configs because pods are ephemeral and rescheduled constantly — a static config listing specific pod IPs/ports would go stale immediately; the Prometheus Operator watches these CRDs and continuously regenerates scrape targets from live Kubernetes service/pod discovery, so scraping stays correct as pods come and go.
8. The OpenTelemetry Collector sits between instrumented applications and the observability backends, receiving all telemetry over the vendor-neutral OTLP protocol and fanning it out via configurable exporters (to Prometheus, Loki, Tempo, or anything else). The concrete benefit: application code only ever needs to know how to speak OTLP to the Collector — it has zero knowledge of which specific backend is in use. Swapping Tempo for a different trace backend later means changing the Collector's exporter configuration only; no application redeploys, code changes, or SDK reconfiguration across every service are needed.
9. **RTO (Recovery Time Objective)** = how long you can tolerate being down before service must be restored. **RPO (Recovery Point Objective)** = how much data loss (measured in time) is acceptable — i.e., how far back your last usable recovery point can be. **Pilot light** keeps only the most critical core infrastructure (e.g., a database replica) running at minimal capacity in the DR region, scaling the rest up on failover — cheaper, but slower RTO (still need to boot most of the stack). **Warm standby** keeps a scaled-down but fully functional full copy of the environment running at all times in the DR region — costs more continuously, but RTO is much faster since everything is already running and just needs scaling up, not booting from cold.

11. **Worked answer** — structure this the way a strong system-design answer is always structured: clarify, architecture, what to monitor/alert/trace specifically, then tradeoffs.
    - **Clarify first**: are these 50 services owned by one team or many? Polyglot or one language/runtime? Synchronous (HTTP/gRPC) or heavily async (event/queue-driven)? Existing tooling or greenfield? I'll assume: many teams, polyglot, mostly synchronous HTTP/gRPC with some async messaging, greenfield.
    - **Instrumentation layer**: every service instrumented with an OpenTelemetry SDK (auto-instrumentation where available) emitting metrics, logs, and traces with consistent semantic conventions (`service.name`, `service.namespace`, `deployment.environment`) and W3C trace-context propagation across every service-to-service call — without consistent propagation, traces fragment at service boundaries and you lose the entire point of tracing a request across 50 services.
    - **Metrics (Prometheus + Grafana)**: the **RED method** for every service (Request rate, Error rate, Duration) plus the **USE method** for infrastructure (Utilization, Saturation, Errors — CPU/memory/disk/network per node and pod). Aggregate via `kube-prometheus-stack`, federate or use remote-write to a long-term store (e.g., Thanos/Mimir/Cortex) if retention beyond Prometheus's local window is needed at this scale — 50 services generate enough series that a single Prometheus instance's local storage and query load need horizontal scaling consideration.
    - **Logs (Loki)**: structured JSON logging from every service, shipped via Promtail/Grafana Agent, labeled by `namespace`/`app`/`pod` only (never per-request IDs as labels — see Q1). Correlate via `trace_id` embedded in the log body.
    - **Traces (Tempo)**: full distributed tracing across every service boundary via the OTel Collector, sampled intelligently at this scale (head-based sampling at the edge, or tail-based sampling in the Collector to keep 100% of error/slow traces while dropping a fraction of "boring" successful traces to control storage cost) — at 50 services' worth of request volume, 100% unsampled tracing is usually not cost-effective, and this is a tradeoff worth stating explicitly.
    - **What to alert on**: SLO-based multi-window burn-rate alerts (per Q5/Q6) on each service's user-facing availability and latency SLOs — not raw infrastructure thresholds like "CPU > 80%," which are causes, not symptoms, and generate noisy pages disconnected from actual user impact. Route critical burn-rate alerts to PagerDuty with clear ownership (each alert maps to exactly one team's on-call, via labels), and require every alert to carry a `runbook_url`.
    - **What to trace**: every cross-service call path, with special attention to the highest-fan-out services (a request touching 10+ downstream services is exactly where tracing earns its keep — metrics alone can't show you *which* of 10 downstream calls is the actual bottleneck). Prioritize instrumenting the critical user journeys (checkout, login, search) for tracing depth first if full 50-service coverage can't happen simultaneously.
    - **Tradeoffs to state explicitly**: 100% trace sampling doesn't scale cost-wise at this size — tail-based sampling that preserves all errors/slow traces is the standard compromise. A single centralized platform team should own the shared OTel Collector config, Grafana dashboards-as-code, and SLO alerting framework, while individual teams own their own service's specific SLO targets and runbooks — full centralization becomes a bottleneck, full per-team autonomy causes inconsistent instrumentation that breaks cross-service correlation (the same "shared labels as the glue" point from the architecture file). Finally, call out cost: at 50 services, cardinality and retention decisions (label design, sampling rate, log retention window) are genuine cost levers, not just performance knobs — worth monitoring the observability stack's own resource usage, not just the platform it's observing.
