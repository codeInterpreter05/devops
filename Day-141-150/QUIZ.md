# Day 141-150 — Quiz: Final Interview Blitz (Full-Roadmap Review)

This is not a normal day's quiz. Every question below deliberately reaches back into an earlier phase — if you can answer all of these cleanly, out loud, without notes, you are genuinely ready to sit real interviews. Try each one as if a real interviewer just asked it, before checking the answers.

1. **(Phase 0 — Linux/Networking)** A teammate reports that requests from one specific pod to a downstream service are intermittently timing out, but requests from every other pod to the same downstream service are fine. Walk through your debugging process from the network layer up, naming actual commands/tools you'd reach for at each step.

2. **(Phase 1 — Kubernetes/Terraform)** Walk through, end to end, everything that happens when you run `kubectl apply -f deployment.yaml` to update an existing Deployment's image tag — from the API server to a running new pod. Separately: explain what Terraform state locking prevents, and what happens if two engineers run `terraform apply` against the same state at the same time without it.

3. **(Phase 2 — CI/CD/Security)** A teammate proposes rebuilding the application from source separately for staging and production so they can "catch environment-specific build issues early." What's wrong with this from a supply-chain and consistency standpoint, and what would you propose instead? Separately, explain why the order SAST → SCA → container scan → signing matters, not just that all four exist.

4. **(Phase 3 — Observability/SRE)** A service has a 99.9% availability SLO over a rolling 30-day window and handles 20,000,000 requests during that window. How many "allowed" errors does that give you in absolute terms? If the current 5-minute error rate is running at 2%, calculate the burn rate and estimate how long it would take to exhaust the full 30-day budget at that rate if it continued.

5. **(Phase 4 — Advanced/Specialization)** Explain what a service mesh's mutual TLS (mTLS) actually buys you in a zero-trust model that Kubernetes NetworkPolicies alone do not. Why can't NetworkPolicies alone provide strong workload identity?

6. **(Behavioral)** Tell a STAR-format story about a time you disagreed with a technical decision someone else made (a lab exercise counts). What made your Action section concrete rather than vague, and what did your Result section actually change afterward?

7. **(Cross-phase system design synthesis)** Design an internal developer platform (IDP) for an org that currently has each team hand-rolling its own CI/CD, provisioning, and dashboards. Your answer should explicitly draw on GitOps promotion flow (Phase 2), the observability correlation story (Phase 3), and platform-engineering self-service concepts (Phase 4) — not just describe a portal.

8. **(Salary negotiation)** A recruiter asks, "What are your salary expectations?" on a first call, before any scope or level has been discussed. What's the risk of answering directly with a number right then, and what's a better response?

9. **(Portfolio/GitHub)** A hiring manager will look at your GitHub profile for roughly 90 seconds before deciding whether to look closer. What specifically should be true of your pinned repos and their READMEs to survive that 90 seconds, and what's the single most common thing that fails it?

10. **Final self-assessment (verbatim interview question reframed):** *"Every question from every phase — you should be able to answer them all confidently by now."* Rather than answering this one directly, use it as a checklist: for each phase (0 through 4), name one topic you'd still hesitate on if asked cold right now. If you can't name any hesitation at all, that's itself worth double-checking honestly before you treat yourself as fully ready.

## Answers

1. Start narrow, don't jump straight to "the network is broken": (a) confirm it's actually that one pod and not a coincidence — repeat the failing request several times from that pod specifically (`kubectl exec -it <pod> -- curl -v <downstream>`); (b) check the pod's own health/resource pressure first (`kubectl top pod`, `kubectl describe pod` for recent restarts/OOM/throttling) since a starved pod can manifest as "network" timeouts that are actually scheduling/CPU-throttling issues; (c) check DNS resolution specifically from that pod (`kubectl exec -it <pod> -- nslookup <downstream-service>`) since a stale or slow DNS response is a classic single-pod-only symptom; (d) check whether that pod landed on a specific node with a node-level networking issue (compare `kubectl get pod -o wide` across the healthy vs. unhealthy pods — same node as other trouble reports?); (e) use `tcpdump`/`ss` on the node or a network-debugging sidecar to see whether packets are leaving the pod at all versus being dropped somewhere in between; (f) check for a NetworkPolicy or service mesh sidecar config that might apply differently to that pod (different labels, different namespace) if everything else looks identical. The key interview signal: narrowing systematically from "pod-specific" outward, not guessing at the network in general.

2. `kubectl apply` sends the updated manifest to the **API server**, which validates it (admission control, including any webhooks/policy-as-code checks) and writes the new desired state to **etcd**. The **Deployment controller** notices the spec changed and creates a new **ReplicaSet** for the new pod template (image tag), then drives a rolling update: it scales the new ReplicaSet up and the old one down incrementally, respecting `maxSurge`/`maxUnavailable`. For each new pod, the **scheduler** assigns it to a node based on resource requests/affinity/taints, then the **kubelet** on that node pulls the new image and starts the container, running any configured readiness probe before the **Endpoints/EndpointSlice** controller adds it to the Service's routable pool — only after it passes readiness does traffic actually reach it. Terraform state locking (via the backend, e.g., S3 + DynamoDB lock table, or Terraform Cloud) prevents two concurrent `terraform apply` runs from reading and writing the same state file simultaneously; without it, two applies can race, each computing a plan against a state that's about to be stale, resulting in corrupted state, resource drift, or one apply silently clobbering the other's changes — the fix is that the second `apply` blocks (or errors) until the first releases the lock.

3. Rebuilding per environment reintroduces the exact risk it's trying to catch: a different build (different dependency resolution timing, different base image patch level) can behave differently from "the same" source — the classic "works in staging, breaks in prod" bug, except now the artifacts were never actually identical, which also breaks any SBOM/provenance claim about what's running in prod. The fix: build the artifact once, immutably (tag/digest it), scan and sign it once, and promote that *exact* artifact through every environment; environment differences should come from injected configuration, not from rebuilding code. On ordering: SAST and SCA are cheap/fast and run pre-build/pre-package on code and the dependency manifest respectively, so they should fail fast before you spend time building and scanning a container image; container/image scanning runs post-build because it needs the actual built layers; signing runs last, after every prior gate passed, so a signature is a guarantee "this artifact passed everything," and the CD operator can refuse to deploy anything unsigned. Running these out of order (e.g., signing before scanning) breaks the guarantee the signature is supposed to represent.

4. Error budget = `1 - 0.999 = 0.001`. Over 20,000,000 requests: `20,000,000 × 0.001 = 20,000` allowed errors over the 30-day window. Burn rate = `error_rate / error_budget = 0.02 / 0.001 = 20`. At a sustained burn rate of 20, the 30-day budget would be exhausted in `30 / 20 = 1.5 days` — well above the standard 14.4x critical/page threshold, so this is a page-immediately situation, not a ticket-level warning.

5. Kubernetes NetworkPolicies operate on **IP addresses and ports**, matched via pod labels/namespace selectors at the network layer (L3/L4) — they control *which pods can talk to which*, but they don't verify *who is actually on the other end of the connection*. An IP can be spoofed, or a compromised/misconfigured pod that happens to match an allowed label selector can still connect — NetworkPolicies trust that "traffic from this IP/label matches this policy" is sufficient, with no cryptographic verification of identity. A service mesh's mTLS (e.g., Istio, Linkerd) issues each workload a cryptographic identity (typically a SPIFFE-style certificate) and encrypts and mutually authenticates every connection at L7 — both sides prove who they are via certificates, independent of IP/label spoofing, and traffic is encrypted in transit by default (defense against a compromised node sniffing traffic). Zero trust means *never* implicitly trusting network position alone — mTLS is what actually delivers strong workload identity and encryption; NetworkPolicies are a complementary, coarser-grained control on top, not a substitute for it.

6. No single correct answer — grade your own response against the rubric in CHEATSHEET.md. A strong answer names the *specific* disagreement (not "I disagreed with an approach" — the actual technical point, e.g., "a teammate wanted to skip a security gate to hit a deadline"), states the concrete action you took to raise it (not "I explained my concerns better" — what you specifically said or did, who you looped in, what alternative you proposed), and closes with a real result: did the decision change, did you concede and why, or did you learn something that changed how you'd raise it next time. The weakest version of this answer stays vague on the Action section ("we talked it through") — that's the part to tighten most.

7. A strong answer explicitly ties the three pillars together rather than describing a generic "developer portal": (a) **service catalog** — every service has an ownership/metadata record (team, on-call, repo, SLOs) that other tooling reads from, not a wiki page nobody updates; (b) **golden-path scaffolding** — creating a new service via the IDP generates a repo, a CI pipeline inheriting shared security gates, a GitOps config entry (so it's deployed via the same pull-based reconciliation model as everything else — Phase 2's GitOps promotion flow, not a bespoke deploy script), and a baseline Grafana dashboard wired to the shared observability stack (Phase 3's correlated metrics/logs/traces, provisioned automatically rather than built by hand per service); (c) **self-service boundary** — explicitly state what's self-service (scaffolding a standard service) versus what still requires a request (a genuinely non-standard infrastructure need), because an IDP that tries to make *everything* self-service on day one usually ships nothing; (d) **a real success metric** — e.g., time from "I need a new service" to "first successful deploy with working dashboards," not just "the portal launched." A weak answer stops at "we'd use Backstage" without connecting it to how GitOps and observability plug into the golden path.

8. Answering with a specific number immediately anchors the entire negotiation before you know the role's actual scope/level, and companies frequently negotiate toward (or below) whatever number is stated first — guessing low out of nervousness costs real money, and guessing high with no scope context can also misfire. A better response defers gracefully: "I'd like to learn more about the role and level first, but I'm confident we can find a number that works if it's a mutual fit" — this keeps the conversation moving without giving up your anchor, and shifts the first-number burden back onto the company, whose stated range then becomes useful market information for you.

9. Every pinned repo should have a README whose *first paragraph* states what the project does and why it exists — not installation steps first. The repo should show real, working evidence of the specific skill it's meant to demonstrate (a "CI/CD project" pinned with no actual pipeline config visible is a gap, not a highlight), have no dead links, leftover TODOs, or broken CI badges, and reflect your *current* skill level rather than being stale from early in the roadmap. The single most common failure: a README that only explains *how to run it* (install steps, commands) and never explains *what it is or why it matters* — a reviewer skimming for 90 seconds bounces off that immediately because they have to reverse-engineer the point of the project from the code itself.

10. There's no "correct" answer to grade here — the value is in the honesty of your own list. A genuinely ready candidate can name specific topics across all five phases (even strong candidates should have at least one per phase, because total mastery of 150 days of material with zero soft spots is rare and usually means you haven't looked hard enough) and, critically, has a concrete plan for each one — a specific re-study action, not just an acknowledgment. If you complete this exercise and truly cannot name a single hesitation anywhere, go back and actually run a mock interview session targeting your weakest-feeling phase before trusting that assessment — "I can't think of a gap" is a different, weaker signal than "I tested myself and found none."
