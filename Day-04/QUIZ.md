## Answers

1. `ifconfig`/`netstat` come from the unmaintained `net-tools` package and are frequently absent from modern minimal/container images. `ip`/`ss` (from `iproute2`) are actively maintained, faster (`ss` reads directly from the kernel via netlink instead of parsing `/proc/net/tcp`), and understand modern constructs like network namespaces that `ifconfig`/`netstat` predate.
2. `/16` is the CIDR prefix length â€” 16 bits of network, 16 bits of host, i.e. netmask `255.255.0.0`, giving 65,536 total addresses in the block (65,534 usable after reserving the network and broadcast addresses). It does **not** mean "16 addresses."
3. `127.0.0.1` is the loopback address â€” traffic bound to it never leaves the machine at the network layer at all, so no firewall rule can make it reachable externally. Only a service bound to `0.0.0.0` (all interfaces) or a specific routable interface IP is reachable from other machines, and even then only subject to firewall rules.
4. `/etc/nsswitch.conf`'s `hosts:` line controls the order (e.g. `files dns`), meaning `/etc/hosts` is checked before a DNS query is made. `/etc/resolv.conf` then supplies which DNS servers to query and any search-domain suffixes, if the lookup falls through to `dns`.
5. `dig` always queries DNS servers directly and ignores `/etc/hosts` entirely. `getent hosts` goes through the full system resolver (`nsswitch.conf` order), so if `/etc/hosts` has a matching entry, `getent hosts` returns that instead of ever touching the network â€” hence the two can disagree by design.
6. Many systemd-based distros manage DNS via `systemd-resolved`, which generates/overwrites `/etc/resolv.conf` (often as a symlink to a stub file) based on network configuration events (DHCP renewal, VPN connect, reboot). Manual edits aren't persisted because the file isn't the actual source of truth â€” `resolvectl`/`systemd-resolved`'s own config is.
7. Three-way handshake: client sends `SYN` (proposed initial sequence number) â†’ server replies `SYN-ACK` (acknowledges client's sequence, proposes its own) â†’ client sends `ACK` (acknowledges server's sequence); only after this is the connection `ESTABLISHED` and data can flow. Closing needs four steps (`FIN`/`ACK` from each side independently) because TCP connections are full-duplex â€” each side must separately declare "I'm done sending," since one side may still have data to send after the other has finished (a "half-closed" connection).
8. Not necessarily down. `ping` uses ICMP, a completely different protocol from TCP; many hosts and cloud providers deliberately block or rate-limit ICMP for security reasons while leaving TCP services like HTTPS fully reachable. A successful `curl` is stronger evidence of health than a failed `ping`, not a contradiction.
9. `traceroute` sends probe packets with increasing TTL (starting at 1). Each router along the path decrements TTL by 1; when TTL reaches 0, that router drops the packet and replies with an ICMP "Time Exceeded" message identifying itself â€” that reply becomes one line of `traceroute` output. `* * *` means no reply was received for that hop within the timeout, typically because that router silently drops or rate-limits ICMP responses (not necessarily that the path itself is broken, since it may still forward the actual traffic).
10. TCP guarantees ordered, reliable delivery via a handshake and retransmission, at the cost of extra round trips and head-of-line blocking. UDP has no handshake and no delivery guarantee, but has minimal overhead and latency. DNS queries are small and cheaply retried by the application, so UDP's low overhead wins; HTTP and SSH need guaranteed, ordered byte streams, so they use TCP. Streaming/VoIP prefer UDP because a late-arriving packet (which TCP would stall the whole stream to retransmit and reorder) is worse than simply dropping it.
11. `>` lines are the literal bytes curl sent (the HTTP request line and headers). `<` lines are the literal bytes curl received back (the response status line, headers). `*` lines are curl's own diagnostic narration about what it's doing (DNS lookup, TCP connect, TLS handshake details) â€” not actual wire traffic.
12. Strong answer: "First, curl resolves `example.com` to an IP address â€” checking `/etc/hosts` first, then querying the DNS resolvers listed in `/etc/resolv.conf` if there's no local override, respecting `search` domains and the `nsswitch.conf` order along the way. Once it has an IP, curl opens a TCP connection to port 443 via the three-way handshake: SYN, SYN-ACK, ACK. Because it's HTTPS, curl then performs a TLS handshake â€” negotiating a protocol version and cipher suite, validating the server's certificate chain, and establishing session keys (one round trip for TLS 1.3, more for TLS 1.2, or zero with session resumption). Only after that does curl send the actual HTTP request â€” the request line and headers â€” over the encrypted channel, and the server responds with a status line, headers, and body. Finally the connection is either kept alive for reuse or torn down via TCP's four-way FIN/ACK close." Mention that `curl -v` shows every one of these stages explicitly, and that you've used `tcpdump` alongside it to see the same exchange at the packet level if you want to demonstrate hands-on depth.
