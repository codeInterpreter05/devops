# Day 53 — Quiz: Phase 1 Project — Full Stack IaC

Try to answer without looking at your notes. Answers are at the bottom.

1. Why should VPC/networking always be the first layer provisioned, before EKS, RDS, or the ALB?
2. Why do workloads, RDS, and ElastiCache belong in private subnets, and what provides their outbound internet access?
3. What specific Kubernetes-related tags do subnets need for the AWS Load Balancer Controller to auto-discover them correctly?
4. What does `manage_master_user_password = true` on an RDS instance actually do, and why is it preferable to a Terraform `random_password` resource passed in as a variable?
5. Who actually provisions the ALB for EKS ingress traffic — Terraform directly, or something else? Explain the mechanism.
6. What is the core architectural difference between a traditional push-based CI/CD pipeline and ArgoCD's GitOps model, and why does that difference matter for security?
7. What does `selfHeal: true` do in an ArgoCD sync policy, and what workflow does it break if the team isn't aware of it?
8. What problem does External Secrets Operator solve, and what is the actual mechanism by which application code accesses a secret it manages?
9. Why does the `refreshInterval` on an ExternalSecret matter specifically in combination with RDS-managed password rotation?
10. What's the difference between what KEDA scales and what Karpenter scales? Can HPA alone scale a Deployment to zero replicas?
11. How does Karpenter's approach to node provisioning differ fundamentally from the older Cluster Autoscaler's approach?
12. **Interview question:** Walk me through the complete architecture of a production EKS environment you've built.

---

## Answers

1. Because EKS, RDS, ElastiCache, and the ALB all need to be placed inside specific subnets with correct routing (public vs. private, NAT egress) already established — every other layer has a hard dependency on the VPC/subnet structure existing first, so provisioning order must follow that dependency chain.
2. They should have no direct route from the public internet to minimize attack surface — data stores and application workloads have no legitimate reason to accept unsolicited inbound internet traffic. Outbound access (pulling container images, calling external APIs) is provided via a NAT Gateway in a public subnet, which allows outbound-initiated traffic without exposing any inbound public IP on the private-subnet resources themselves.
3. `kubernetes.io/role/elb` (value `1`) on public subnets, `kubernetes.io/role/internal-elb` (value `1`) on private subnets, and `kubernetes.io/cluster/<cluster-name>` (value `shared` or `owned`) on both — these exact tag keys are what the Load Balancer Controller and the Kubernetes cloud provider read to auto-discover which subnets to use and whether a load balancer should be internet-facing or internal.
4. It tells RDS to generate and store the master password directly in AWS Secrets Manager, with automatic rotation, rather than requiring you to generate/manage the password yourself. It's preferable to a `random_password` Terraform resource because that pattern still results in the password landing in Terraform state (even if the backend encrypts state at rest, it's a real exposure surface), whereas the RDS-managed approach never puts the plaintext password through Terraform's own state/plan output at all.
5. The AWS Load Balancer Controller, running in-cluster, provisions the ALB — not a Terraform `aws_lb` resource directly. Terraform's role is setting up the prerequisites (IAM role via IRSA, installing the controller itself); the controller then watches Kubernetes `Ingress` objects annotated for ALB and creates/manages the actual ALB, target groups, and listener rules in response to those objects.
6. In a push-based model, an external CI/CD pipeline needs credentials to reach into the cluster and apply changes — meaning broad cluster credentials live outside the cluster, in the pipeline's secret store, a common lateral-movement target if that system is compromised. In ArgoCD's pull-based GitOps model, an in-cluster controller reads a Git repository and reconciles cluster state to match it — the controller only needs read access to Git, not the other way around, so no external system holds cluster-wide write credentials.
7. `selfHeal: true` makes ArgoCD automatically revert any manual change to a synced resource back to whatever's declared in Git, as soon as it detects drift. It breaks the "quick manual kubectl fix during an incident" workflow — a hotfix applied directly to the cluster gets silently reverted moments later unless the same fix is also committed to Git, which the team needs to know and plan for.
8. It solves the problem of application/Helm/Terraform code never containing plaintext secrets, while still letting the app consume secrets normally. Mechanically, ESO (running in-cluster, authenticated via IRSA) continuously pulls values from an external store (AWS Secrets Manager) and writes them into native Kubernetes `Secret` objects — the application consumes that Secret exactly like any other Kubernetes Secret (env var or mounted file), with zero AWS-specific code or awareness that ESO is involved.
9. RDS-managed password rotation changes the password inside Secrets Manager automatically on a schedule. Without ESO's `refreshInterval` periodically re-pulling the current value into the Kubernetes Secret, the cluster would keep using the old, now-invalidated password after a rotation — causing an outage. `refreshInterval` is what makes the whole rotation chain (RDS rotates → Secrets Manager updates → ESO refreshes K8s Secret → pods eventually pick up the new value) actually function end-to-end.
10. KEDA scales the number of **pods** (extending HPA with event-driven metrics and scale-to-zero capability). Karpenter scales the underlying **nodes**, provisioning exactly-fitting EC2 instances for whatever pods are currently unschedulable. No, vanilla HPA cannot scale a Deployment to zero replicas — its minimum is always 1; scale-to-zero specifically requires KEDA (or an equivalent controller) layered on top.
11. Cluster Autoscaler operates within pre-defined node groups/Auto Scaling Groups whose instance types are fixed ahead of time, and can only scale the count of instances within those groups. Karpenter observes unschedulable pods directly and provisions nodes on demand, choosing whatever instance type/size (and Spot vs. On-Demand) best and most cheaply fits the pending pods' actual requirements, without needing pre-defined node groups per workload shape — and it can also actively consolidate underutilized nodes into better-fitting instance types, not just adjust a count.
12. Strong answer: "I'd start with networking: a VPC across three AZs with public subnets for the ALB and private subnets for nodes and data stores, with NAT Gateways for outbound egress, tagged correctly for the Load Balancer Controller to auto-discover them. On top of that, an EKS cluster with managed node groups in the private subnets, plus IRSA-scoped IAM roles for the controllers that need AWS access — the Load Balancer Controller, External Secrets Operator, and Karpenter. RDS Multi-AZ and an ElastiCache replication group sit alongside, with RDS-managed password rotation into Secrets Manager so no plaintext credential ever touches Terraform state or Git. The application itself is packaged as a Helm chart and deployed via ArgoCD in a pull-based GitOps model — ArgoCD watches the Git repo and reconciles the cluster to match it, with selfHeal reverting any manual drift, so no CI pipeline needs broad cluster credentials. External Secrets Operator syncs the RDS/Redis credentials from Secrets Manager into native Kubernetes Secrets that the app consumes normally. For elasticity, KEDA scales the app's worker pods based on SQS queue depth — including scaling to zero when idle — and when those pods can't be scheduled, Karpenter provisions exactly-fitting EC2 nodes on demand and consolidates them away again once load drops. The ALB itself is provisioned by the in-cluster Load Balancer Controller reacting to an Ingress resource, not hand-authored in Terraform. End to end, that's a VPC-to-traffic path that's secure by default, fully declared as code, auto-scaling on real demand signals, and recoverable via Git history or Helm rollback if something goes wrong."
