# Day 38 — Quiz: HashiCorp Vault

Try to answer without looking at your notes. Answers are at the bottom.

1. What does it mean for Vault to be "sealed," and can a sealed Vault answer any requests at all?
2. Explain Shamir's Secret Sharing as Vault uses it: what are "key shares" and "threshold," and what security property does splitting the master key actually buy you?
3. Why is auto-unseal (AWS/GCP/Azure KMS) the standard production pattern instead of manual Shamir unsealing on every restart?
4. What's the difference between the KV secrets engine and the AWS/Database secrets engines in terms of what Vault actually does when you read from them?
5. What do auth methods answer, and what do policies answer? Why are they deliberately kept separate?
6. How does the Kubernetes auth method let a pod prove its identity to Vault without any static credential baked into the pod?
7. What is a lease, and what's the difference between `default_ttl` and `max_ttl`?
8. What does `vault write database/rotate-root/mydb` do, and how is it different from a normal dynamic secret request?
9. What role does the Vault Agent Sidecar Injector play in a Kubernetes deployment, and why does the application container need zero Vault-specific code?
10. If you detect a compromised pod that holds a dynamic database credential, what's the fastest way to cut off its database access, and does the application itself need to cooperate?
11. **Interview question:** How do dynamic secrets in Vault work for database credentials? What's the rotation cycle?

---

## Answers

1. Sealed means Vault doesn't currently have the master key (or the key it protects) in memory, so it cannot decrypt any stored data. A sealed Vault can still respond to basic health/status checks (`vault status`), but it cannot serve any secret to any request, authenticated or not, until unsealed.
2. The master key is split into N "key shares" using Shamir's Secret Sharing; any T of those N shares ("threshold") can reconstruct the master key, but fewer than T reveal nothing about it mathematically. This means no single operator holding one share can unseal Vault alone (protects against insider threat/single leaked share), while the whole operation doesn't depend on any one specific person being available (protects availability, since any T-sized subset of keyholders suffices).
3. Every Vault restart requires unsealing again, and gathering a quorum of human operators to type in keys for every routine restart, rolling upgrade, or autoscaling event doesn't scale operationally. Auto-unseal delegates this to an already-highly-available cloud KMS, letting Vault unseal itself automatically on boot while still keeping Shamir-derived recovery keys as a break-glass fallback.
4. KV just stores whatever static data you put there (encrypted, versioned) — Vault does not generate or manage that data itself, and reading it repeatedly returns the same value until you change it. The AWS/Database engines are dynamic: each read causes Vault to actively call the real backend (AWS API, the database) at that moment to create a brand-new, unique credential, hand it back with a lease, and later revoke it.
5. Auth methods answer "who is this caller" (verifying identity via Kubernetes service account JWTs, AWS IAM STS identity, username/password, AppRole, etc.). Policies answer "what is this caller allowed to do" (path-level read/write/delete capabilities). They're kept separate so the same permission model (policy) can be reused across different ways of proving identity, and so identity verification logic doesn't need to be duplicated per permission scheme.
6. Kubernetes automatically mounts a projected service account token (a JWT) into every pod. The pod presents that JWT to Vault's Kubernetes auth login endpoint; Vault validates it against the cluster's TokenReview API to confirm it's a genuine, currently valid token for the expected service account and namespace, then issues a Vault token scoped to policies bound to that role. The pod's Kubernetes-native identity is the login credential — nothing Vault-specific needs to be pre-distributed into the pod.
7. A lease is Vault's record that a specific dynamic secret was issued, to whom, and when it expires. `default_ttl` is the normal lifespan a credential is valid for before automatic revocation if not renewed; `max_ttl` is the absolute hard ceiling — even repeated renewals cannot extend a lease's life past `max_ttl`.
8. It rotates the credential Vault itself uses to connect to the database with elevated/admin privileges (as opposed to the low-privilege dynamic credentials it issues to callers) to a new value that only Vault knows internally — even a human Vault administrator can no longer retrieve the plaintext afterward, closing the loop on "who could possibly know this password" at the top of the privilege chain.
9. The Injector is a Kubernetes mutating admission webhook that, based on pod annotations, automatically adds an init container (fetches the initial secret before the app starts) and a sidecar container (keeps authenticating, renewing, and re-templating the secret to a file for the pod's lifetime). The application just reads a file at a known path — it needs no Vault SDK, no auth logic, and no awareness that a dynamic, expiring secret is even involved; the sidecar handles all of that transparently.
10. `vault lease revoke <lease_id>` immediately triggers Vault to run the revocation statement (e.g., `DROP ROLE`) against the database, deleting that specific credential right away — the application does not need to cooperate or even be aware; its next database connection attempt using that credential will simply fail with an authentication error.
11. Strong answer: "Dynamic secrets for databases work by having Vault hold its own privileged connection to the database and a configured role with `creation_statements`. When an authenticated caller reads `database/creds/<role>`, Vault runs those creation statements against the real database at that moment, generating a brand-new, randomly-named database user with a random password, and returns it along with a lease that has a `default_ttl` and a `max_ttl`. There's no scheduled password-rotation job in the traditional sense — the 'rotation cycle' is that every credential is short-lived by construction: it either gets renewed (up to `max_ttl`) or automatically revoked via a `revocation_statement` (typically `DROP ROLE`) when its lease expires, and can also be revoked on-demand immediately if a compromise is suspected. Because every request mints an independent credential, you also get per-instance traceability — you can tell exactly which application instance a given DB session belongs to from its Vault-generated username."
