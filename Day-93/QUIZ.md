9. What's the structural analogy between an OpenSearch alerting monitor and a Prometheus alerting rule?

## Answers

1. ISM = Index State Management, OpenSearch's native feature for automating index lifecycle transitions (states, actions, transitions). Its Elasticsearch equivalent is ILM (Index Lifecycle Management) — OpenSearch forked from Elasticsearch 7.10 and reimplemented several X-Pack features, including this one, as open-source equivalents under a different name.
2. Hot: newest data, fastest storage, actively written/queried, kept small via rollover. Warm: no longer written to, queried less often, replica count often reduced and force-merged to fewer segments for better compression/search speed on read-mostly data. Cold: rarely queried, moved to cheaper/slower storage, made read-only to protect settled data and enable further optimization. Delete: removed once past retention requirements, keeping total cluster size bounded.
3. Rollover creates a new backing index (and repoints a write alias to it) once a size/age/doc-count threshold is crossed, so no single index grows unbounded. Applications write to the alias (e.g., `logs-write`) so the underlying index swap is completely transparent to them — they never need to know or track which specific dated/numbered index is currently receiving writes.
4. Roughly 10–50GB per shard is the commonly cited sweet spot. Too small: excessive shard count creates cluster-state overhead (every node tracks every shard's metadata, and cluster-state updates get slower to propagate) even though total data volume may be modest. Too large: a single shard's search/recovery time grows proportionally and can't be parallelized further since a shard is the unit of distribution — an oversized shard can't spread its work across more nodes.
5. Low watermark (default 85%): stop allocating new shards to that node. High watermark (default 90%): actively try to relocate existing shards away from that node. Flood-stage watermark (default 95%): make affected indices read-only to prevent the disk from filling completely.
6. Allocation awareness configures node attributes (e.g., `zone`) so OpenSearch places replica shards in a different zone/rack than their corresponding primary. It protects against a single zone/rack outage taking out both a primary and all of its replicas simultaneously, which would otherwise cause actual data unavailability rather than just reduced redundancy.
7. Nodes are tagged with an attribute (e.g., `box_type: hot` / `warm` / `cold`) matching their hardware tier, and `index.routing.allocation.require.<attr>` settings restrict where a given index's shards are allowed to live. ISM's state-transition actions update this required attribute as an index ages (hot → warm → cold), which physically relocates the shard's data onto the appropriately-tiered hardware as part of the transition — this is the real mechanism underlying "moving data between tiers," not just a logical label change.
8. CCR continuously replicates shard-level operations from a leader to a follower cluster in near-real-time, producing a warm, queryable standby. A snapshot is a point-in-time backup taken on a schedule to a repository (e.g., S3) — cold, not immediately queryable, but immune to *logical* corruption (a bad delete or mapping change) that CCR would faithfully replicate to the follower since CCR mirrors the leader's actual operations. Mature setups use CCR for fast failover and snapshots as protection against logical corruption and for longer-term immutable backup.
9. Both are: a query evaluated on a schedule, against a specific backend (OpenSearch query DSL vs. PromQL), producing a result that's checked against a trigger/threshold condition, which — if true — fires a notification action. The core "scheduled query → condition → notify" shape is the same; only the query language and underlying data model (documents vs. time series) differ.
10. A bucket-level monitor runs an aggregation query (e.g., a `terms` aggregation grouping by `service`) and evaluates its trigger condition **per bucket**, letting one monitor definition alert independently per group (e.g., per service exceeding its own threshold) instead of requiring a separate query-level monitor maintained per entity — reducing monitor sprawl and centralizing threshold changes to one place.
11. Force-merge is an expensive, one-time operation intended for read-mostly indices that are no longer being actively written to — merging segments down (e.g., to a single segment) improves compression and search speed for data that won't change further. Running it against a "hot" index still receiving writes is wasteful, since new writes immediately create fresh segments again, undoing the benefit almost immediately while still paying the operation's real CPU/IO cost.
12. Strong answer: "First I'd recognize that 85% is exactly OpenSearch's default low watermark — meaning the cluster may already be avoiding allocating new shards to affected nodes, which itself can cause symptoms unrelated to query complexity. I'd run `GET _cluster/allocation/explain` to see if shards are stuck unassigned due to watermark pressure, `GET _cat/shards?v` and `GET _cat/indices?v&s=store.size:desc` to check for an excessive shard count or oversized shards outside the ~10-50GB target, and `GET _cluster/health` to check for unassigned shards. I'd also check whether the ISM policy is actually moving older data out of the hot tier as designed — a stuck or misconfigured lifecycle policy is a common reason hot-tier disk fills up faster than intended. Based on what I find, remediation is either freeing space (accelerating stuck ILM transitions, deleting indices past retention that failed to auto-delete, force-merging read-mostly indices), adding capacity (more nodes/disk), or fixing an allocation imbalance (checking zone/tier allocation filters) if the disk pressure is concentrated on specific nodes rather than the whole cluster."
