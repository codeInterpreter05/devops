# Day 115 — Quiz: Kubernetes Cost Optimisation

Try to answer without looking at your notes. Answers are at the bottom.

1. Why can't a cloud provider's native billing dashboard (e.g., AWS Cost Explorer) tell you how much a single Kubernetes namespace costs?
2. Is cost typically allocated by resource **requests** or resource **limits** in Kubecost/OpenCost, and why does that choice matter?
3. What's the practical difference between OpenCost and Kubecost, and when would you actually need to pay for Kubecost?
4. What does "efficiency" mean in a Kubecost/OpenCost allocation report, and why is it the most actionable number on it?
5. Why is `updateMode: "Off"` the recommended starting point for VPA rather than `"Auto"`?
6. Why might right-sizing a pod's request to the exact average (p50) observed usage be a bad idea?
7. What is cluster bin packing, and how can having "fewer, larger nodes" versus "many small nodes" affect cost, independent of whether pods are right-sized?
8. How does Karpenter fundamentally differ from the traditional Cluster Autoscaler + fixed node groups model?
9. Why does listing multiple instance types/families in a Karpenter NodePool reduce Spot interruption risk?
10. What can silently prevent Karpenter (or Cluster Autoscaler) from consolidating/scaling down idle nodes, even when there's genuine idle capacity?
11. What does the "shared cost" bucket represent in a Kubecost/OpenCost report, and why does ignoring it distort chargeback numbers?
12. **Interview question:** How do you charge back infrastructure costs to individual product teams in a shared EKS cluster?

---

## Answers

1. Cloud billing tools meter at the resource level (an EC2 instance, an EBS volume) — they have no visibility into how that instance's capacity is subdivided among the many pods, from many namespaces, that Kubernetes has scheduled onto it. Attribution inside the cluster requires a tool that scrapes in-cluster metrics (Prometheus/kube-state-metrics) and combines them with node pricing, which is exactly what Kubecost/OpenCost do.
2. By resource **requests**. Requests are what actually reserves capacity on a node regardless of real usage, so allocating cost by requests correctly bills a team for capacity they reserved (and are blocking other workloads from using) even if they don't fully consume it — which is the signal that should drive right-sizing. Allocating by limits would overstate cost for workloads with generous limits but modest requests.
3. OpenCost is the free, CNCF, in-cluster cost allocation engine (the open-sourced core of Kubecost) — namespace/label-level allocation with no paid gate. Kubecost is the commercial product adding multi-cluster aggregation, long-term data retention beyond your Prometheus window, savings recommendations, and governance/budget-alert features for larger organizations. Pay for Kubecost when you need those specific gaps filled, not by default.
4. Efficiency is `actual usage ÷ requested resources`. It's the most actionable number because low efficiency directly identifies over-requested (wasteful) workloads — the single most common and most fixable driver of Kubernetes cost waste, as opposed to raw cost, which just tells you what's expensive, not whether it's wasteful.
5. `"Off"` mode only produces recommendations without touching running pods — it's non-disruptive. `"Auto"` mode actively evicts and recreates pods to apply new resource values, which can cause real disruption for singleton or stateful workloads without proper replicas and PodDisruptionBudgets in place.
6. Sizing to the exact average usage leaves no margin for normal variance or traffic spikes — the first time real usage exceeds that average (which happens routinely, not just in rare edge cases), the pod risks OOMKill (memory) or CPU throttling. A higher percentile (p95/p99) or VPA's `upperBound` recommendation is the safer basis for a shipped request value.
7. Bin packing is how tightly pods fit onto the available nodes without wasted "remainder" capacity per node. Many small nodes tend to strand more unusable leftover capacity per node (when pod sizes don't divide evenly) and force daemonset overhead (log shippers, CNI, sidecar injectors) to be paid on every additional node — fewer, larger nodes generally pack tighter and amortize that overhead across more workload capacity, subject to HA/spread-constraint tradeoffs.
8. Cluster Autoscaler scales nodes within pre-defined, fixed-shape node groups (ASGs) that a human configured in advance. Karpenter has no node-group abstraction — it watches unschedulable pods directly and provisions whatever instance type/size best fits their actual requirements from the cloud API in real time, and can also proactively consolidate existing nodes into fewer/cheaper ones.
9. Spot capacity pools are independent per instance-type-per-AZ combination. A NodePool that only allows one instance type draws from a single pool, so any Spot reclaim event in that pool interrupts the entire fleet running on it at once. Allowing many compatible instance types spreads the fleet across many independent pools, so an interruption event in any one pool only affects a fraction of total capacity.
10. Overly strict PodDisruptionBudgets (e.g., `maxUnavailable: 0` everywhere) can prevent the autoscaler from ever finding a node it's safe to drain and remove, even when that node is genuinely idle — the disruption budget, not the autoscaler's logic, becomes the blocker.
11. "Shared cost" is cluster-level overhead not attributable to any single tenant — kube-system, the monitoring stack, and unallocated idle node capacity. It's typically split evenly or proportionally across namespaces. Ignoring it and reporting only directly-attributed cost per namespace understates true cluster cost and makes any single team's number look artificially low, which breaks reconciliation against the actual cloud bill.
12. Strong answer: "Install Kubecost or OpenCost to get real per-namespace (or per-label) cost allocation based on resource requests, since namespaces usually already map to teams. Enforce a `Team`/`CostCenter` label convention (ideally via admission policy, not just documentation) so allocation queries are reliable. Explicitly decide and communicate how shared/idle cluster cost — kube-system, monitoring, unallocated capacity — gets split, rather than silently omitting it. Surface each team's efficiency (usage/requests), not just raw cost, so chargeback reports double as a right-sizing incentive instead of just a bill. Automate a recurring showback report (weekly/monthly) rather than a one-off manual pull, and reconcile the totals periodically against the actual cloud invoice to keep the numbers trustworthy." Mention a concrete tool/workflow if you've actually built one.
