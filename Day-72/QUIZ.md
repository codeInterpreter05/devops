## Answers

1. Whether the system behaves correctly under realistic or extreme **concurrent load** — concurrency bugs, connection-pool exhaustion, and resource contention only manifest under real concurrent traffic, which unit/integration/e2e tests (run in isolation or with minimal concurrency) don't simulate.
2. The `thresholds` configuration block (e.g., `http_req_duration: ['p(95)<500']`, `http_req_failed: ['rate<0.01']`) — k6 itself exits non-zero if the defined thresholds are violated during the run, which is what allows a CI pipeline step to actually fail the build/deploy automatically instead of relying on a human reading the output.
3. When the load scenario needs genuinely complex, stateful logic — multi-step user journeys, complex data generation, or reuse of existing Python libraries/client code — since Locust's tasks are real Python code with full language capability, whereas k6 is scoped to JavaScript with a lighter-weight execution model better suited to straightforward HTTP/API load patterns.
4. A measurable "steady state" baseline (e.g., request success rate, p99 latency) *before* injecting any failure. Skipping it is dangerous because you'd have no objective way to distinguish "the experiment caused a real problem" from ordinary metric noise — without a baseline, you can't actually conclude anything from the experiment's results.
5. It means the goal is testing under conditions realistic enough to give genuine confidence — since staging environments often lack production's real traffic patterns, scale, and dependency behavior, some failure modes simply won't reproduce anywhere else. It explicitly does NOT mean running large, uncontrolled experiments recklessly; it must be paired with deliberate blast-radius minimization (small scope, automated abort, gradual expansion) to be responsible.
6. It's blast-radius control expressed directly as configuration — limiting the experiment to affect only a percentage of matching pods (here, 25%) rather than all of them at once. It matters because starting small and proving safety at limited scope before considering a larger-scope experiment is the core practice that makes "test failure in production" responsible rather than reckless.
7. AWS FIS operates at the cloud infrastructure layer (EC2 instances, EBS volumes, AZ-level events, simulated AWS API errors); LitmusChaos operates inside the Kubernetes layer (pods, Services, container-level resource/network stress). A rigorous test of something like "does my EKS app survive an AZ failure" needs FIS to simulate the actual AZ-level event and LitmusChaos-style validation of how the application's pods/Services respond once nodes become unavailable — the two are complementary, not competing.
8. It ties a real CloudWatch alarm to the experiment so that if the alarm fires at any point during the run (e.g., error rate crosses a defined threshold), FIS automatically halts the experiment immediately. This directly implements the blast-radius-minimization principle's "automated abort mechanism" requirement, built into the tool rather than something you'd have to hand-roll with a human watching a dashboard.
9. A chaos experiment (LitmusChaos/FIS) tests whether the *system* technically recovers from an injected fault. A game day pairs that technical fault with the team's real, human incident-response process — paging, runbooks, communication, decision-making under pressure — testing whether the *people and process* are also resilient, which a chaos tool alone has no way to evaluate.
10. Because the value of a game day comes from rehearsing realistic, unannounced incident response — if responders know exactly what's coming and when, it tests only the technical fault-recovery path, not genuine decision-making, communication, and escalation behavior under real uncertainty, which is the harder and more valuable thing to validate.
11. Because it only proves resilience under that specific day's configuration — later code, infrastructure, or team changes can silently introduce resilience regressions that a one-time experiment will never catch. Continuous, automated re-running is what catches regressions as the system evolves, the same "don't just check once" theme as continuous vulnerability re-scanning.
12. Strong answer: "Chaos engineering is the discipline of deliberately injecting realistic failures — pod kills, network partitions, instance stops, dependency unavailability — into a system, ideally while it's serving real traffic, in order to build empirical confidence that it actually survives those failures rather than just assuming it does based on architecture diagrams or configuration. The core principles are: define a measurable steady-state baseline before injecting anything; form a specific hypothesis that the steady state holds under the fault; inject real-world-realistic failure categories, not arbitrary ones; and ideally run in production, since staging rarely reproduces real traffic patterns and scale closely enough for some failure modes to even show up. Running in production safely means minimizing blast radius at every step: start with a small percentage of pods/instances/traffic, use a tool with a built-in automated abort mechanism tied to a real alarm — AWS FIS's `stopConditions` or LitmusChaos's `PODS_AFFECTED_PERC` are concrete examples — run during a low-risk time window with the team aware a drill is happening, and only expand scope after building confidence at a smaller scale. Finally, automate experiments to run continuously rather than once, since a one-time test only proves resilience under that day's specific configuration, and pair technical chaos experiments with human-facing game days to validate that paging, runbooks, and incident response actually work too, not just the infrastructure."
