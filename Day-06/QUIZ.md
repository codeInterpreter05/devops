# Day 6 — Quiz: Bash Scripting II — Text Processing

Try to answer without looking at your notes. Answers are at the bottom.

1. What is the difference between BRE and ERE in `grep`/`sed`, and what does the `-E` flag change?
2. What are `grep`'s three possible exit codes, and why does that matter under `set -e`?
3. In `awk`, what's the difference between `NR` and `FNR`? Give a scenario where confusing them causes a bug.
4. Why does `awk -F',' '{print $2}'` on `"a,,b"` produce an empty field for `$2`, while the default whitespace `FS` would never do that on `"a  b"`?
5. Why must you `sort` before piping to `uniq -c` if you want a correct global count of duplicates?
6. What does `sed -i` actually do under the hood, and what two side effects does that implementation have?
7. Why does `sed -i 's/a/b/' file` work on Linux but fail with an error on macOS unless you write `sed -i '' 's/a/b/' file`?
8. What's wrong with `cmd 2>&1 > file` if your goal is to capture both stdout and stderr into `file`? What's the correct form?
9. Why can a pipeline like `awk '{...}' hugefile.log | sort | uniq -c` process a file far larger than available RAM without crashing?
10. What does `tr -d '\r' < winfile > unixfile` do, and why does `tr file.txt` (without `<`) not work the way a newcomer might expect?
11. Why do `awk -F','` and `cut -d','` both give wrong results on a real CSV file that contains a field like `"Smith, John"`?
12. **Interview question:** How would you extract all unique IP addresses from a 10GB log file efficiently?

---

## Answers

1. BRE (Basic Regular Expressions) is `grep`/`sed`'s default dialect, where `+ ? | ( ) { }` are literal characters unless escaped with a backslash. ERE (Extended Regular Expressions), enabled with `-E` (`grep -E`, `sed -E`), treats those same characters as special/metacharacters by default, matching what most other regex flavors (Python, JS, PCRE) do. Both are POSIX-standard; `-E` just changes which characters need escaping to mean what.
2. `0` = at least one match found, `1` = no match found, `2` = an actual error (bad option, unreadable file). Under `set -e`, a `grep` that legitimately finds nothing exits `1`, which `set -e` treats as a script failure and aborts — even though "no matches" is often a perfectly normal outcome, not an error. Fix with `grep ... || true` or explicit exit-code handling.
3. `NR` is the record number across *all* input processed so far (keeps climbing across multiple files); `FNR` is the record number *within the current file* (resets to 1 for each new file). Confusing them when processing `*.log` with `awk 'NR==1{print FILENAME}'` would only print the filename of the very first file overall, not the first line of every file — you'd want `FNR==1` for that.
4. With the default whitespace `FS`, `awk` treats *any run* of whitespace as one separator and trims leading/trailing whitespace, so repeated spaces never produce empty fields. Once you set an explicit single-character `FS` like `,`, that collapsing behavior stops — each individual comma is a separator boundary, so two commas in a row (`,,`) really do produce an empty field between them.
5. `uniq` (and `uniq -c`) only collapses **adjacent** duplicate lines — it has no global view of the file. If identical values aren't next to each other, they're counted as separate groups. Sorting first guarantees all identical values become adjacent, making `uniq -c`'s counts globally correct.
6. `sed -i` doesn't edit the file's bytes in place — it writes the transformed output to a new temporary file, then renames that temp file over the original. Side effects: (1) it requires write permission on the *containing directory* (rename is a directory operation), and (2) it changes the file's inode, which breaks any existing hard links to the old inode and can confuse tools that track a file by inode rather than path.
7. GNU `sed` (Linux default) treats the backup-suffix argument to `-i` as optional — `-i` alone means "no backup." BSD `sed` (macOS default) requires an explicit argument, even an empty string, so `-i` alone on macOS tries to consume the next argument (your script) as the suffix, causing a parse error. `-i ''` supplies the (empty) suffix BSD sed expects.
8. Redirections are applied left to right, and `2>&1` copies whatever fd 1 currently points to *at that moment* — it doesn't create an ongoing link. In `cmd 2>&1 > file`, fd 2 is first pointed at the current stdout target (the terminal), and only afterward is fd 1 redirected to `file` — so stderr keeps going to the terminal, not the file. The correct form is `cmd > file 2>&1`, where fd 1 is redirected to `file` first, so when `2>&1` runs, it correctly copies that same target.
9. Each tool in the pipeline (`awk`, `sort`, `uniq`) processes input as a stream and is connected to the next by a small, fixed-size kernel buffer — data flows through in small chunks, and (for `awk`/`uniq`) only the current line or a small window is held in memory at once. `sort` is the interesting exception: for genuinely huge input it switches to an external merge sort, writing sorted chunks to temporary disk files and merging them, rather than holding the entire input in RAM — which is why `sort` on a 10GB file works without needing 10GB of memory.
10. It strips carriage return characters (`\r`, the extra byte in Windows-style CRLF line endings) from `winfile`, writing the result to `unixfile`. `tr` never accepts a filename argument at all — it only ever takes the translation-set string(s), so passing a third positional argument like `file.txt` is an **extra operand** and both GNU and BSD `tr` reject it immediately with a usage error, instead of reading and processing the file. You must redirect the file in with `<`.
11. Both tools split every line strictly on every occurrence of the delimiter character, with no concept of quoting — they don't know that a comma inside a quoted field (`"Smith, John"`) is data, not a field separator. A real CSV value containing the delimiter character inside quotes will be split into extra fields, silently shifting every subsequent column. For real CSV with quoted/escaped fields, use a proper CSV-aware tool (`csvkit`, a scripting language's CSV library, or GNU `awk`'s `--csv`/`FPAT` extension) instead of naive `cut`/`awk -F`.
12. Best answer: `awk '{print $1}' huge.log | sort -u > unique_ips.txt` (assuming the IP is the first field) — extract just the IP column, then let `sort -u` deduplicate. Explain *why* this scales: `sort` doesn't require the whole file to fit in memory — for large inputs it performs an external merge sort, splitting the input into memory-sized sorted chunks written to temporary disk files, then merging those chunks (and removing adjacent duplicates for `-u`) in a final pass, so memory usage stays bounded regardless of file size. An alternative single-pass approach, `awk '!seen[$1]++ {print $1}' huge.log`, avoids `sort` entirely by keeping a hash of already-seen IPs in memory — this is typically fast and memory-efficient too, since memory scales with the number of *unique* IPs (usually far smaller than total log lines), not the file size; mentioning both approaches, and the memory/throughput tradeoff between them, is a strong answer.
