# Day 99 — AWS-native Observability III: Synthetics & Managed Prometheus/Grafana

**Phase:** 3 – Observability | **Week:** W17 | **Domain:** Observability | **Flag:** ⚡ Interview-critical

## Brief

The first two files covered *reactive* observability — metrics, logs, and traces generated by real traffic hitting your system. CloudWatch Synthetics flips that: it generates **synthetic** traffic proactively, so you find out a login flow is broken before a real user does. The second half of this file tackles a genuinely high-stakes architecture decision that shows up constantly in interviews and in real cost/ops trade-off discussions: **do you run Prometheus and Grafana yourself, or pay AWS to manage them for you?**

## CloudWatch Synthetics (canaries)

A **canary** is a scheduled script — written in Node.js (Puppeteer-based) or Python (Selenium-based) — that AWS runs for you on managed Lambda infrastructure, from one or more regions, on a schedule you define (e.g., every 5 minutes). Canaries do three things real users do, deliberately, so you catch problems before they do:

- **Heartbeat checks** — hit an endpoint, assert on status code and response time.
- **API canaries** — call a sequence of API endpoints, assert on response bodies/schemas.
- **UI flow canaries** — actually click through a browser flow (login, add-to-cart, checkout) using Puppeteer/Selenium scripting, catching *rendering* and *JavaScript* failures that a simple HTTP heartbeat check would miss entirely.

```javascript
// Minimal heartbeat canary (Node.js/Puppeteer runtime)
const synthetics = require('Synthetics');
const log = require('SyntheticsLogger');

const apiCanaryBlueprint = async function () {
  const page = await synthetics.getPage();
  const response = await page.goto('https://app.example.com/login', { waitUntil: 'domcontentloaded', timeout: 30000 });
  if (!response || response.status() !== 200) {
    throw new Error(`Failed to load page: status ${response ? response.status() : 'no response'}`);
  }
  await synthetics.takeScreenshot('loaded', 'loaded');
};

exports.handler = async () => {
  return await synthetics.executeStep('loginPageLoads', apiCanaryBlueprint);
};
```

Every run captures **screenshots and a HAR file** (full network waterfall), which is what makes Synthetics genuinely useful for debugging — when a canary fails at 3 a.m., you get a visual "here's exactly what the page looked like" artifact instead of just a boolean failure. Canary results feed CloudWatch metrics (`SuccessPercent`, `Duration`) which you wire into a **CloudWatch Alarm** the same way you would any other metric, so a failing canary can page someone.

**Where Synthetics fits in the bigger picture:** it's blackbox, outside-in monitoring — it tells you the system is broken *from the outside*, the same way a user would experience it, independent of whatever your internal metrics/dashboards say. This is a different, complementary signal to APM/RUM (which tell you *why* from the inside) — a mature setup runs both, because internal metrics can look perfectly healthy while a CDN misconfiguration or DNS issue makes the site completely unreachable to real users.

## Amazon Managed Grafana & Amazon Managed Prometheus

This is the pairing behind one of the most common real-world (and interview) questions in observability: **self-managed Prometheus + Grafana vs. the AWS-managed equivalents.**

**Amazon Managed Service for Prometheus (AMP)** is a fully managed, horizontally-scaling, multi-tenant Prometheus-compatible **metrics backend**. Critically, it is API-compatible: your existing Prometheus servers (or the ADOT Collector, or an EKS-hosted Prometheus doing local scraping) `remote_write` their scraped metrics into AMP over an IAM SigV4-signed HTTPS endpoint, and you query it with standard **PromQL** — the query language and your existing dashboards/alert rules do not need to change.

```yaml
# prometheus.yml remote_write block pointing at an AMP workspace
remote_write:
  - url: https://aps-workspaces.us-east-1.amazonaws.com/workspaces/ws-xxxxxxxx/api/v1/remote_write
    sigv4:
      region: us-east-1
    queue_config:
      max_samples_per_send: 1000
```

**Amazon Managed Grafana (AMG)** is a fully managed Grafana **workspace**: AWS runs the Grafana server, handles patching/upgrades, and integrates SSO via IAM Identity Center or SAML — so you're not managing a Grafana deployment, its database, or its auth integration yourself. It ships with first-class data source plugins for AMP, CloudWatch, X-Ray, OpenSearch, and (with a small setup step) most third-party sources Grafana normally supports.

### The self-managed vs. managed trade-off, concretely

| Dimension | Self-managed (Prometheus + Grafana on EKS) | Managed (AMP + AMG) |
|---|---|---|
| **Ops burden** | You own HA (multiple replicas, Thanos/Cortex for long-term storage and dedup), storage scaling, backups, upgrades, and Grafana auth wiring | AWS owns HA, scaling, storage, and upgrades; you just configure remote_write and dashboards |
| **Cost model** | Infra cost (EC2/EBS/EKS nodes) — can be cheaper at very high, steady-state scale since you're paying for raw compute/storage, not per-sample | Pay per metric sample ingested (AMP) and per active user + workspace (AMG) — predictable but scales with usage, not with your hardware choices |
| **Query language / ecosystem** | Native PromQL, full Prometheus Operator/ServiceMonitor ecosystem, no vendor lock beyond your own infra choices | Same PromQL (AMP is API-compatible) — but data now lives in an AWS-only backend; migrating off means re-pointing remote_write and re-doing long-term retention elsewhere |
| **Long-term retention** | Requires bolting on Thanos, Cortex, or Mimir yourself for retention beyond default Prometheus TSDB limits | AMP handles long-term storage natively as part of the managed service |
| **Team fit** | Makes sense if you already have platform engineers who live in this stack and want maximum control/cost efficiency at scale | Makes sense for smaller platform teams, or orgs standardizing on AWS-native tooling, who'd rather pay AWS than staff for Prometheus HA/scaling expertise |

**The honest answer AWS itself gives, and the one to give in an interview:** self-managed Prometheus/Grafana wins on raw cost-at-scale and ecosystem control (custom exporters, Prometheus Operator CRDs, arbitrary retention tuning) if you already have the operational maturity to run it reliably. AMP/AMG wins on time-to-value and reduces the "who's on call for our monitoring stack's own outages" problem — you're outsourcing the undifferentiated heavy lifting of HA and scaling. Neither is a strictly correct answer; the right answer depends on team size, existing AWS commitment, and whether "our monitoring system going down" is an acceptable risk to self-manage.

## Points to Remember

- Synthetics canaries generate proactive, outside-in synthetic traffic (heartbeat/API/UI-flow) and capture screenshots + HAR files — the complementary opposite of reactive, inside-out APM/metrics.
- Canary results are just CloudWatch metrics (`SuccessPercent`, `Duration`) — wire them into standard CloudWatch Alarms like any other metric.
- AMP is a Prometheus-*compatible* remote_write target and PromQL query engine — your existing scrape configs, dashboards, and alert rules largely carry over unchanged.
- AMG is a managed Grafana workspace with built-in SSO and first-class AWS data source integrations — you don't run the Grafana server or its backing database.
- The self-managed vs. managed decision is a genuine trade-off between operational burden and cost-at-scale vs. time-to-value and reduced on-call surface — there is no universally "correct" side.

## Common Mistakes

- Only running heartbeat canaries and assuming that covers "monitoring the frontend" — a heartbeat check can return 200 while the actual page is broken due to a JS error; UI-flow canaries catch what heartbeats miss.
- Treating Synthetics as a replacement for real-user monitoring (RUM) or APM instead of a complement — synthetic traffic from a handful of AWS regions doesn't reflect the full diversity of real user network conditions/devices.
- Assuming AMP is "just Prometheus" with no behavior differences — remote_write queue tuning, IAM SigV4 auth setup, and ingestion quotas are AMP-specific operational details that don't exist in a vanilla self-hosted setup.
- Picking AMP/AMG purely to "avoid ops work" without checking the per-sample cost projection at your actual metric cardinality — high-cardinality workloads (many labels, many series) can make managed Prometheus pricing surprisingly steep.
- Answering the self-managed-vs-managed interview question with a flat "always use managed" or "always self-host" — the strong answer names the actual trade-off dimensions (cost model, ops burden, team maturity) rather than picking a side dogmatically.
